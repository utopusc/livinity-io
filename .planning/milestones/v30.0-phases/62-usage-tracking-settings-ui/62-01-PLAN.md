<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External HTTP client → broker_usage row | apiKeyId comes from Phase 59 Bearer middleware (validated UUID); never user-supplied free text |
| schema.sql → live PG | Boot-time `client.query(schemaSql)` applies migration; idempotent so re-application is safe |
| database.ts query opts → SQL | Parameterized queries; no string interpolation of opts.apiKeyId |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-01 | T (Tampering) | apiKeyId param to queryUsageByUser/All | mitigate | parameterized $N placeholder; never concatenated into SQL string |
| T-62-02 | I (Information disclosure) | queryUsageByUser cross-user leak | mitigate | existing scope `WHERE user_id = $1` PRESERVED; api_key_id filter is AND-ed, not OR-ed — even if attacker passes another user's key UUID, query returns zero rows |
| T-62-03 | D (Denial of service) | ALTER TABLE lock on prod | accept | RESEARCH.md Pitfall 3 — Mini PC broker_usage row count is small (low millions max); ALTER is sub-second. If row count > 10M, plan revisits with NOT VALID strategy |
| T-62-04 | R (Repudiation) | hard-delete of api_keys would orphan usage attribution | mitigate | `ON DELETE SET NULL` preserves history; Phase 59 soft-deletes via `revoked_at` (defense-in-depth) |
</threat_model>

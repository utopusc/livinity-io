Note: `findApiKeyByHash` returns `ApiKeyRow` WITH `userId` field but does NOT return `key_hash` from PG (it's known by the caller — the input was the hash). The Bearer middleware needs `id` + `userId` from the row.

Schema block to append at end of schema.sql (line 339+, AFTER the existing broker_usage block):
```sql
-- =========================================================================
-- API Keys (Phase 59 FR-BROKER-B1-01..05) — Per-user `liv_sk_*` Bearer tokens.
-- Cleartext returned ONCE on create. SHA-256 hash stored. Revocation is soft
-- (revoked_at NOT NULL means revoked). Mirrors docker_agents shape (Phase 22).
-- gen_random_uuid() works without `CREATE EXTENSION pgcrypto` (matches
-- existing convention: 14 other tables use gen_random_uuid() with no extension
-- line). RESEARCH.md Open Question 1 verdict: drop the extension line.
-- =========================================================================
CREATE TABLE IF NOT EXISTS api_keys (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  key_hash      CHAR(64) NOT NULL UNIQUE,
  key_prefix    VARCHAR(16) NOT NULL,
  name          VARCHAR(64) NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used_at  TIMESTAMPTZ,
  revoked_at    TIMESTAMPTZ
);

CREATE INDEX IF NOT EXISTS idx_api_keys_user_id ON api_keys(user_id);
CREATE INDEX IF NOT EXISTS idx_api_keys_active  ON api_keys(key_hash) WHERE revoked_at IS NULL;
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `database.ts` ↔ PG (untrusted hash inputs from Wave 2 middleware) | Bearer middleware passes presented-key SHA-256 directly to `findApiKeyByHash($1)` — input is parameterized; no SQL injection surface |
| `createApiKey` plaintext exposure | Plaintext returned ONCE in function return value; never logged, never re-queried |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-05 | I (Information disclosure) | createApiKey return path | mitigate | Plaintext returned ONCE in `{row, plaintext}` tuple; never persisted to PG (only `key_hash` is stored), never written to logger; downstream Wave 3 route returns it ONCE in tRPC response |
| T-59-06 | T (Tampering) | revokeApiKey | mitigate | UPDATE is user-scoped (`user_id = $2`) — a malicious or compromised user cannot revoke another user's key by guessing IDs |
| T-59-07 | T (Tampering) | api_keys schema | mitigate | UNIQUE constraint on `key_hash` prevents duplicate-hash collisions; ON DELETE CASCADE on `user_id` ensures orphan keys auto-removed when user is deleted (consistent with `sessions`, `docker_agents`) |
| T-59-08 | S (Spoofing) | hashKey input | mitigate | SHA-256 of FULL plaintext (including `liv_sk_` prefix) — not just the body — so prefix collisions cannot collapse two distinct keys to the same hash |
| T-59-09 | I | logger output | mitigate | The logger is NEVER called with `plaintext` in `createApiKey`. Test 4 in `database.test.ts` covers param shape; an explicit grep guard could be added in Wave 4 if needed |
</threat_model>

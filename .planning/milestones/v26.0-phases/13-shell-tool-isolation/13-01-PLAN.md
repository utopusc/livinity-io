<objective>
Harden the shell tool isolation boundary: (1) lock the local `shell` tool's parameter shape to prevent future drift that could introduce a device_id escape hatch, (2) upgrade both local and device-shell tool descriptions so the AI never hallucinates a device_id onto the local shell, and (3) close the only remaining architectural loophole — external tool registration via `POST /api/tools/register` can currently overwrite the local `shell` tool because `ToolRegistry.register` silently replaces tools with the same name.

Purpose: Phase 12 already enforces ownership on device-routed tools, so the cross-user device_id rejection path for SHELL-01 is ALREADY load-bearing via the `device_<id>_shell` proxy naming + authorizeDeviceAccess gate. What remains is (a) contractually freezing the local shell's parameter surface so SHELL-02's local-only guarantee stays true across future edits, (b) documenting the constraint in tool descriptions so the AI correctly targets device tools by name instead of trying to pass a device_id into the local shell, and (c) preventing a rogue or misconfigured external caller from clobbering the local `shell` tool's execute function via `/api/tools/register`.

Output:
- daemon.ts shell tool: description explicitly names the local-only contract and directs the agent to `device_<id>_shell` proxies for remote execution
- api.ts: RESERVED_TOOL_NAMES set + 409 response on collision attempts in /api/tools/register
- device-bridge.ts DEVICE_TOOL_SCHEMAS.shell description: explicitly states ownership is enforced server-side
</objective>

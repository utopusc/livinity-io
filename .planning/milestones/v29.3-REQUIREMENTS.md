## v29.3 Requirements (Final disposition)

### Risk Fix (FR-RISK)

- [x] **FR-RISK-01** — User confirms `claude.ts:99-115` OAuth-fallback-with-raw-SDK code path is removed. After Phase 39, `ClaudeProvider.getClient()` either (a) returns API-key-based client only when API key is explicitly set in Redis, or (b) throws if subscription token would be the only auth source. Subscription tokens never reach raw `@anthropic-ai/sdk`. Verified by grep: zero `authToken: token` constructions referencing `claudeAiOauth`.
  → **Satisfied** (Phase 39, mechanism + 5/5 regression tests pinning the deletion).

### Per-User OAuth Isolation (FR-AUTH)

- [x] **FR-AUTH-01** — User in multi-user mode can run `claude login` independently from other users. Each user's `~/.claude/.credentials.json` lives in their own user HOME. Cross-user credential read attempts fail with permission denied.
  → **Satisfied at the application layer** (Phase 40, synthetic per-user dirs under `/opt/livos/data/users/<id>/.claude/`, mode 0o700). POSIX-enforced cross-user isolation explicitly NOT scoped per D-40-05 + D-40-16 — synthetic dirs share livinityd UID; future security audit may require real Linux user accounts.
- [x] **FR-AUTH-02** — User sees a "Connect my Claude account" button in Settings > AI Integrations (per-user). Login status (connected / not-connected / token-expired) is visible per user.
  → **Satisfied (mechanism)** (Phase 40, multi-user UI branch in `ai-config.tsx`, 3 tRPC routes). **Carry-forward to v29.4:** `claudePerUserStartLogin` subscription is NOT in `httpOnlyPaths` — UX hang risk under WS reconnect (audit gap C3).
- [x] **FR-AUTH-03** — When SdkAgentRunner spawns the `claude` CLI subprocess for a user request, the subprocess `HOME` env var is set to that user's home directory. Cross-user OAuth credential leak is impossible.
  → **Satisfied (mechanism)** (Phase 40 sacred-file surgical edit at line 266: `homeOverride || process.env.HOME || '/root'` + Phase 41-04 X-LivOS-User-Id header → homeOverride wiring at `nexus/.../api.ts`). **Carry-forward to v29.4:** integrity test `BASELINE_SHA = 623a65b9...` is now stale (current `4f868d31...` due to v43.x model-bump commits) — needs re-pin or revert (audit gap C2).

### Anthropic-format Broker (FR-BROKER-A)

- [x] **FR-BROKER-A-01** — Marketplace app can `POST /v1/messages` to `http://livinity-broker:<port>` with Anthropic Messages API format and receive a valid response. Endpoint reachable from container network.
  → **Satisfied (mechanism)** (Phase 41, route mounted at `server/index.ts:1215` via `mountBrokerRoutes`).
- [x] **FR-BROKER-A-02** — Same endpoint streams responses (SSE) with Anthropic Messages streaming schema. Streaming chunks observable end-to-end.
  → **Satisfied (mechanism)** (Phase 41, `sse-adapter.ts` emits Anthropic-spec wire format `event:` + `data:` chunks; 4/4 SSE format tests).
- [x] **FR-BROKER-A-03** — Broker translates incoming Anthropic Messages format request into `SdkAgentRunner.run()` invocation (Agent SDK `prompt` + `options`). Translation handles multi-turn message history, system prompts, and tool definitions.
  → **Satisfied (mechanism)** (Phase 41, `translate-request.ts`, 8/8 translator tests). Tools array intentionally **IGNORED** with warn log per D-41-14 — marketplace apps cannot inject tool definitions through the broker (intentional v29.3 scope decision).
- [x] **FR-BROKER-A-04** — Broker resolves the calling container's owner user_id (via existing app gateway middleware pattern from `livos/packages/livinityd/source/server/index.ts`) and spawns SdkAgentRunner with that user's HOME. Cross-user request cannot use another user's subscription.
  → **Satisfied (mechanism)** (Phase 41 Strategy B — HTTP proxy reuses single nexus SdkAgentRunner instance, homeOverride per-call via X-LivOS-User-Id header).

### OpenAI-compat Broker (FR-BROKER-O)

- [x] **FR-BROKER-O-01** — Marketplace app can `POST /v1/chat/completions` with OpenAI Chat Completions format and receive a valid response. Endpoint reachable from container network.
  → **Satisfied (mechanism)** (Phase 42, in-process TS translation layer).
- [x] **FR-BROKER-O-02** — Broker translates OpenAI ↔ Anthropic format bidirectionally: message history (role mapping), tool calls (function call format), streaming (chunk schema).
  → **Satisfied (in-scope subset)** (Phase 42, 16 translator tests + 10 SSE adapter tests). Tool/function-call format intentionally IGNORED per D-42-12.
- [x] **FR-BROKER-O-03** — Model name in OpenAI request (e.g., `gpt-4`, `claude-sonnet-4-6`) is mapped to a single Anthropic model (`claude-sonnet-4-6` default; configurable per LivOS deployment). Unknown models fall through to default with logged warning.
  → **Satisfied (mechanism)** (Phase 42, `resolveModelAlias()` + 5 alias tests + integration test 4 verifies unknown → 200 + warn).
- [x] **FR-BROKER-O-04** — Output format (chat completion vs streaming chunks) matches OpenAI spec exactly — verified by feeding response into the official `openai` Python SDK as a smoke test.
  → **Satisfied (format-verified by automated tests)** + **manual UAT deferred** (Phase 42, 42-UAT.md Section D contains verbatim Python SDK snippet; un-executed because no deploy occurred during milestone). **Carry-forward to v29.4 (audit gap C4):** OpenAI SSE adapter does not emit a `usage` chunk before `[DONE]` — affects FR-DASH-01/02 dashboard completeness for OpenAI streaming traffic.

No user action required.

  → **DROPPED** at milestone close 2026-05-01 per user direction ("MiroFish'i siktir et"). Mechanism is complete in Phase 43, but the MiroFish manifest remains as a draft artifact only — sibling-repo PR + GHCR fork+build + builtin-app registration are NOT pursued. Future anchor app candidate is a v30+ conversation if at all.

### Per-User Usage Dashboard (FR-DASH)

- [x] **FR-DASH-01** — User sees per-user usage stats in Settings > AI Integrations: cumulative input + output tokens, request count by app, current month total. Data persisted (PostgreSQL table; broker writes a row per request from Anthropic Messages API `usage` object).
  → **Satisfied (mechanism)** (Phase 44, `broker_usage` PG table + capture middleware + tRPC `usage.getMine` + UI `usage-section.tsx` + 39 livinityd tests). **Carry-forward to v29.4 (audit gap C4):** OpenAI SSE traffic produces zero rows. **Carry-forward (audit gap C3):** `usage.getMine` not on `httpOnlyPaths` — UX hang risk under WS reconnect.
- [x] **FR-DASH-02** — Admin sees per-app usage stats: which user, which model, request count, token total. Multi-user filterable view.
  → **Satisfied (mechanism)** (Phase 44, `usage.getAll` adminProcedure + admin-cross-user-view.tsx + RBAC backend gate verified). Same OpenAI SSE + `httpOnlyPaths` carry-forward as FR-DASH-01.
- [ ] **FR-DASH-03** — User sees subscription rate limit visibility (Pro 200/day; Max 5x). When user is within 80% of daily limit, a non-blocking warning banner appears in AI Integrations panel. When 100% reached, broker returns HTTP 429 with retry-after; UI surfaces this.
  → **PARTIAL — explicit debt accepted at milestone close** (Phase 44, mechanism for banner + capture row + UI banner-section verified, BUT broker error path collapses to HTTP 500 in `livinity-broker/router.ts:159` — never forwards 429 — and `agent-runner-factory.ts:75-76` drops the `Retry-After` header. Live 429 path is structurally unreachable; only synthetic INSERTs validate the UI surface). **Carry-forward to v29.4 (audit gap C1):** broker error-path repair as a small surgical phase (~10 lines, ~3 unit tests).

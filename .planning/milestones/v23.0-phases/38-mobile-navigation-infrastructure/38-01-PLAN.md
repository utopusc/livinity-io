<objective>
Create the core mobile navigation components: MobileAppContext for state, use-mobile-back for hardware back button, MobileNavBar for the top bar, and MobileAppRenderer for the full-screen overlay. Export WindowAppContent so mobile can reuse the same lazy-loaded app components as desktop windows.

Purpose: These components form the rendering pipeline that makes any system app openable full-screen on mobile. They are completely separate from the desktop WindowManager -- no desktop code is modified beyond adding the export keyword to WindowAppContent.

Output: 5 files (1 modified, 4 created) that together provide the full mobile app rendering pipeline. Not yet wired into the app -- Plan 02 handles integration.
</objective>

    **2. Create MobileAppContext** at `src/modules/mobile/mobile-app-context.tsx`:
    This is the core state management for mobile app navigation, completely separate from the desktop WindowManagerProvider (per CONTEXT.md architecture decision).

    Define the state type:
    ```typescript
    type MobileAppState = {
      appId: string
      route: string
      title: string
      icon: string
    } | null
    ```

    Define the context type:
    ```typescript
    type MobileAppContextT = {
      activeApp: MobileAppState
      openApp: (appId: string, route: string, title: string, icon: string) => void
      closeApp: () => void
    }
    ```

    Create and export:
    - `MobileAppProvider` component that wraps children with the context. Uses `useState<MobileAppState>(null)` for the active app. The `openApp` function sets the state. The `closeApp` function sets state to null.
    - `useMobileApp()` hook that calls `useContext(MobileAppContext)` and throws if used outside provider.

    The provider has NO mobile guard -- it renders on all viewports. The gating happens in the consumer components (MobileAppRenderer checks useIsMobile(), desktop-content.tsx checks useIsMobile()). This keeps the provider simple and testable.

    **3. Create use-mobile-back hook** at `src/modules/mobile/use-mobile-back.ts`:
    This hook integrates the MobileAppContext with the browser History API so that the Android hardware back button and iOS swipe-back gesture close the active app.

    Implementation:
    - Import `useMobileApp` from `./mobile-app-context`
    - Call the hook inside MobileAppRenderer (not in the context provider)
    - When `activeApp` transitions from null to non-null (app opens): call `window.history.pushState({mobileApp: activeApp.appId}, '')` to create a history entry
    - When `activeApp` transitions from non-null to null (app closes): do nothing (popstate already handled, or user tapped back button in UI)
    - Add a `popstate` event listener: when a popstate event fires and `activeApp` is not null, call `closeApp()`. This handles the hardware/OS back button.
    - Use `useEffect` with proper cleanup (remove the event listener on unmount)
    - Use a `useRef` to track the previous activeApp value to detect transitions (avoid pushing duplicate history entries)
    - Export: `export function useMobileBack()` (returns void, purely side-effect)
  </action>

    Per CONTEXT.md: full-screen overlay that renders the active mobile app using WindowAppContent. Right-to-left slide-in animation using Framer Motion with AnimatePresence.

    Implementation:
    - Import: `AnimatePresence, motion` from `framer-motion` (already used in 52+ files)
    - Import: `WindowAppContent` from `@/modules/window/window-content` (newly exported in Task 1)
    - Import: `useMobileApp` from `./mobile-app-context`
    - Import: `useMobileBack` from `./use-mobile-back`
    - Import: `MobileNavBar` from `./mobile-nav-bar`
    - Import: `Suspense` from `react`
    - Import: `Loading` from `@/components/ui/loading`
    - Import: `useIsMobile` from `@/hooks/use-is-mobile`

    Component logic:
    - Call `const {activeApp, closeApp} = useMobileApp()`
    - Call `useMobileBack()` to activate history API integration
    - Call `const isMobile = useIsMobile()`
    - If `!isMobile`, return `null` (per MOB-05 -- desktop UI completely unchanged)
    - Render `<AnimatePresence>` wrapping a conditional `motion.div` when `activeApp` is not null
    - The `motion.div` uses:
      - `key={activeApp.appId}` for AnimatePresence tracking
      - `className="fixed inset-0 z-50 flex flex-col bg-white"` (z-50 to layer above everything, bg-white for the app background)
      - Animation per CONTEXT.md decision: right-to-left slide-in (iOS native style)
        - `initial={{ x: '100%' }}`
        - `animate={{ x: 0 }}`
        - `exit={{ x: '100%' }}`
        - `transition={{ type: 'tween', duration: 0.25, ease: [0.32, 0.72, 0, 1] }}` (iOS ease-out curve, 250ms per decision 200-300ms range)
    - Inside the motion.div:

      - `<div className="flex-1 min-h-0 overflow-hidden pb-safe">` (pb-safe for home indicator, min-h-0 for flex child overflow)
        - `<Suspense fallback={<Loading />}>`
          - `<WindowAppContent appId={activeApp.appId} initialRoute={activeApp.route} />`
        - `</Suspense>`
      - `</div>`

    - Export: `export function MobileAppRenderer()`

    IMPORTANT: The `useMobileBack()` call MUST be inside MobileAppRenderer (not conditionally -- it needs to run even when no app is open to listen for popstate events). Place it before the `if (!isMobile) return null` guard.

    Wait -- actually `useMobileBack` uses `useMobileApp()` internally and listens for popstate. On desktop this is harmless (activeApp is always null, so popstate listener does nothing). But to avoid unnecessary listeners on desktop, move the `if (!isMobile) return null` check AFTER `useMobileBack()` but that means hooks run on desktop too. This is fine -- React hooks must not be called conditionally anyway. The hook is lightweight (one event listener that does nothing when activeApp is null).

    Correct order in the component:
    ```
    const {activeApp, closeApp} = useMobileApp()
    const isMobile = useIsMobile()
    useMobileBack()
    if (!isMobile) return null
    ```

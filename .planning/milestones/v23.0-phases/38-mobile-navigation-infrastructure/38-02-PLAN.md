  <action>
    Modify `desktop-content.tsx` so that on mobile, tapping an app icon calls `mobileAppContext.openApp()` instead of `windowManager.openWindow()`. On desktop, behavior is completely unchanged.

    **1. Add imports:**
    ```typescript
    import {useIsMobile} from '@/hooks/use-is-mobile'
    import {useMobileApp} from '@/modules/mobile/mobile-app-context'
    ```

    **2. Inside `DesktopContent` function**, add near the top (after existing hooks):
    ```typescript
    const isMobile = useIsMobile()
    const {openApp} = useMobileApp()
    ```

    **3. Modify the `openStreamApp` callback** to support mobile. Currently it calls `windowManager.openWindow()`. On mobile, it should call `openApp()` instead:

    Current:
    ```typescript
    const openStreamApp = useCallback((appId: string, route: string, title: string, icon: string) => {
      if (!windowManager) return
      for (const win of windowManager.windows) {
        if (streamAppIds.has(win.appId)) windowManager.closeWindow(win.id)
      }
      setTimeout(() => windowManager.openWindow(appId, route, title, icon), 50)
    }, [windowManager, streamAppIds])
    ```

    New:
    ```typescript
    const openStreamApp = useCallback((appId: string, route: string, title: string, icon: string) => {
      if (isMobile) {
        openApp(appId, route, title, icon)
        return
      }
      if (!windowManager) return
      for (const win of windowManager.windows) {
        if (streamAppIds.has(win.appId)) windowManager.closeWindow(win.id)
      }
      setTimeout(() => windowManager.openWindow(appId, route, title, icon), 50)
    }, [isMobile, openApp, windowManager, streamAppIds])
    ```

    **4. Modify `AppIconConnected` click behavior for mobile.** The `AppIconConnected` component (imported from `./app-icon`) handles clicks for user-installed apps. These apps use `useLaunchApp()` which calls `windowManager.openWindow()`. However, we do NOT need to modify `AppIconConnected` or `useLaunchApp` in this phase because:
    - User-installed apps (Docker containers like n8n, Home Assistant) are not system apps
    - They open via subdomain routing (e.g., n8n.livinity.cloud), not via WindowAppContent
    - They already work on mobile via their own web UIs
    - Only the hardcoded system apps (Remote Desktop, Chrome, web app shortcuts) in `DesktopContent` need the mobile guard, and those already go through `openStreamApp`

    The system apps that are in the dock but NOT in the desktop grid (AI Chat, Server, Terminal, Agents, Schedules, Devices) will be added to the mobile grid in Phase 39 (MOB-01). For now, they can only be opened from the dock. Since the dock is still visible on mobile (dock hiding is Phase 39), this is intentional -- Phase 38 builds the infrastructure, Phase 39 builds the mobile-specific UI.

    **5. Verify no other changes needed.** The `gridItems` array already includes Remote Desktop, Chrome, Gmail, Facebook, WhatsApp, YouTube -- all of which use `openStreamApp`. By modifying `openStreamApp` to check `isMobile`, ALL of these automatically work on mobile with zero per-app changes. This directly satisfies MOB-04 (reuse existing components, zero per-app rewrite).

    **CRITICAL per MOB-05**: On desktop, `isMobile` is false, so `openStreamApp` hits the existing path. The `openApp` function is called on mobile only. Desktop users see zero change.

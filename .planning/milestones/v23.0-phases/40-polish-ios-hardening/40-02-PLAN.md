Add a new `useEffect` inside the `useAgentSocket` hook (after the existing connect useEffect, around line 410) that listens for `visibilitychange` events and forces WebSocket reconnection when the page becomes visible.

Per user decision: 500ms delay after visibility change to let network resume, then check WS readyState and force reconnect if not OPEN. Reset backoff counter so reconnection is immediate.

**Implementation:**
```tsx
// --- Visibility-based reconnection (iOS background/resume) ---
useEffect(() => {
  const handleVisibilityChange = () => {
    if (document.hidden) return
    // Page became visible -- check if WebSocket needs reconnection
    setTimeout(() => {
      const ws = wsRef.current
      if (!ws || ws.readyState !== WebSocket.OPEN) {
        // Reset backoff for immediate reconnection
        backoffRef.current = 1000
        // Clear any pending reconnect timer to avoid double-connect
        if (reconnectTimerRef.current) {
          clearTimeout(reconnectTimerRef.current)
          reconnectTimerRef.current = null
        }
        connect()
      }
    }, 500)
  }
  document.addEventListener('visibilitychange', handleVisibilityChange)
  return () => document.removeEventListener('visibilitychange', handleVisibilityChange)
}, [connect])
```

Place this AFTER the existing connection useEffect (line ~410). The 500ms delay per user decision allows iOS networking stack to resume before attempting connection.

Key behaviors:
- Only acts when page becomes visible (`!document.hidden`)
- Checks if existing WS is already OPEN (skip if healthy)
- Resets `backoffRef.current = 1000` so first reconnect attempt is fast
- Clears pending reconnect timer to prevent double-connect race
- Calls the existing `connect()` function (reuses all existing WS setup logic)
- Does NOT touch `intentionalCloseRef` (that is for component unmount only)

This is purely additive -- the existing connect/reconnect logic is unchanged. Desktop browsers also fire visibilitychange but the WS is usually still OPEN, so the setTimeout check skips reconnection.

**useKeyboardHeight hook** (use-keyboard-height.ts):
Creates a hook that returns the current keyboard height in pixels using the `visualViewport` API. On iOS standalone PWA, the keyboard does NOT resize the layout viewport (100dvh stays the same), so we must detect keyboard presence via `window.visualViewport`.

```tsx
import {useState, useEffect} from 'react'

/** Returns the estimated keyboard height in px (0 when keyboard is closed).
 *  Uses the Visual Viewport API to detect iOS keyboard in standalone PWA mode. */
export function useKeyboardHeight(): number {
  const [keyboardHeight, setKeyboardHeight] = useState(0)

  useEffect(() => {
    const vv = window.visualViewport
    if (!vv) return // No Visual Viewport API (old browsers) -- keyboard height stays 0

    const handleResize = () => {
      // When keyboard opens, visualViewport.height shrinks below window.innerHeight
      const diff = window.innerHeight - vv.height
      // Only count as keyboard if diff > 100px (avoid small rounding differences)
      setKeyboardHeight(diff > 100 ? diff : 0)
    }

    vv.addEventListener('resize', handleResize)
    vv.addEventListener('scroll', handleResize)
    return () => {
      vv.removeEventListener('resize', handleResize)
      vv.removeEventListener('scroll', handleResize)
    }
  }, [])

  return keyboardHeight
}
```

Key details:
- 100px threshold prevents false positives from toolbar/address bar changes
- Returns 0 on desktop (visualViewport.height === window.innerHeight when no keyboard)
- Listens to both `resize` and `scroll` events on visualViewport (iOS fires scroll when keyboard adjusts position)

<objective>
Land the v29.4 Phase 45 C1 carry-forward: the marketplace AI broker (Phase 41/42) currently collapses ALL upstream errors to HTTP 500 (`router.ts:157-162` and `openai-router.ts:234-242`), and `agent-runner-factory.ts:75-77` discards the upstream `Retry-After` header by stringifying status into the Error message. As a result:
1. Anthropic upstream rate-limits (HTTP 429) reach the marketplace app as 500 — UI banners can't render the "rate-limited" state.
2. Anthropic infrastructure errors (HTTP 502/504) ALSO reach the marketplace app as 500 — same as rate-limits, but they need different UX.
3. The `Retry-After` countdown is destroyed.

This plan introduces a small `UpstreamHttpError` class that captures `{status, retryAfter}` at the upstream-fetch boundary in `agent-runner-factory.ts:75-77`. Both router catch blocks (Anthropic-side at `router.ts:157`, OpenAI-side at `openai-router.ts:234`) become `instanceof UpstreamHttpError`-aware:
- If status === 429: forward 429 verbatim with `Retry-After` preserved byte-identical.
- If status is any other upstream status (400/401/403/500/502/503/504/529): forward that exact status (NOT remapped to 429, NOT collapsed to 500).
- If err is NOT an `UpstreamHttpError` (genuinely-internal broker error before/after the upstream fetch): preserve existing 500 behavior.

Tests extend the existing `integration.test.ts` (bare `tsx` + `node:assert/strict` harness — NOT Vitest) with a new `mockUpstreamError()` helper sibling to the existing `mockUpstreamSse()`, plus Test 6 (single 429+Retry-After case) + Test 7 (parameterized loop over the 9-status-code list).

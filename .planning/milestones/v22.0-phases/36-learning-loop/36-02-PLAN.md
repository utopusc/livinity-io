**routes.ts changes:**

1. Add a `rateConversation` tRPC mutation after the getAnalytics route. Input: `{ conversationId: z.string(), rating: z.number().min(1).max(5), completed: z.boolean().optional() }`. Implementation:
   - Get Redis from context (`ctx.livinityd.ai.redis`)
   - Store feedback: `HSET nexus:feedback:{conversationId} rating {rating} completed {completed ? '1' : '0'} timestamp {Date.now()}`
   - Also store which tools were used in the conversation by reading from the tool_calls stream entries matching session_id (but we don't have session_id from conversationId easily). SIMPLER approach: the feedback stores just the rating. The aggregation into capability success_rate happens separately.
   - To aggregate: read all feedback keys `nexus:feedback:*` (small set), compute average rating, update capability metadata via nexus API call `PATCH /api/capabilities/{id}` with `{ metadata: { success_rate: averageRating * 20 } }` (convert 1-5 scale to 0-100%).
   - SIMPLER APPROACH per CONTEXT.md: Store `nexus:feedback:{conversationId}` with `{rating, completed, tools: [...]}` where tools comes from the client (the UI knows which tool calls were in the conversation). Then a separate aggregation step can process this.
   - SIMPLEST APPROACH: Store the feedback in Redis. Return success. The getAnalytics route will read from the Redis stream `nexus:tool_calls` directly for real stats (using XRANGE). Feedback aggregation into success_rate can be done lazily when getAnalytics is called.

   Final implementation:
   ```typescript
   rateConversation: privateProcedure
     .input(z.object({
       conversationId: z.string(),
       rating: z.number().min(1).max(5),
       completed: z.boolean().optional(),
     }))
     .mutation(async ({ ctx, input }) => {
       const redis = ctx.livinityd.ai.redis
       if (!redis) return { success: false }
       await redis.hset(`nexus:feedback:${input.conversationId}`, {
         rating: String(input.rating),
         completed: input.completed ? '1' : '0',
         timestamp: String(Date.now()),
       })
       return { success: true }
     })
   ```

2. Enhance `getAnalytics` to also fetch tool call stats from the Redis stream. After fetching capabilities from nexus, also call `XRANGE nexus:tool_calls - + COUNT 5000` on the Redis connection to get real tool usage data. Aggregate per-tool: totalCalls, successCount, avgDuration. Merge with the existing capability data so the analytics view shows both capability metadata AND actual usage statistics.

   The enhanced return type:
   ```typescript
   {
     toolStats: Array<{ name, type, toolCount, successRate, lastUsed }>,  // existing
     totalCapabilities: number,
     activeCapabilities: number,
     usageStats: Array<{ tool: string; totalCalls: number; successRate: number }>,  // NEW from stream
     coOccurrences: Array<{ toolA: string; toolB: string; count: number }>,  // NEW
   }
   ```

   For coOccurrences: group stream entries by session field, build pair counts (same algorithm as LearningEngine.getCoOccurrences but done in the tRPC route since livinityd has its own Redis connection). Keep it simple — read last 2000 entries, group by session, count pairs, return top 10.

3. Get Redis from `ctx.livinityd.ai.redis`. If redis is not available (null), fall back to the existing behavior (capability-only stats).

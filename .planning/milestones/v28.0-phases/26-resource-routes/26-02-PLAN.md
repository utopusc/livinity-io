Important: `useNetworks` hides `inspectedNetwork` as internal state. To wire the store-driven deep-link, the network section will: (a) read `useSelectedNetwork()` from the store, (b) call `inspectNetwork(selected)` in a useEffect when selected changes (drives the underlying inspect query), (c) when the user clicks the inspect icon on a row, call `setSelectedNetwork(id)` in the store. Same effect: clicking the icon writes to the store, the useEffect fires inspectNetwork on the hook, the hook fires the inspect query. Round-trip flow keeps `useNetworks` unchanged.

## Wave coordination with Plan 26-01

Plan 26-01 and Plan 26-02 both run in **Wave 1** because:
- They modify entirely separate `sections/*.tsx` files (containers + images vs. volumes + networks).
- They both consume `resource-store.ts` (created by 26-01) — but ONLY the `selectedVolume` / `selectedNetwork` slots. Those slots are declared up-front by 26-01 Task 1's `<behavior>` block (Tests B-D explicitly require all four slots present in default state). So 26-02 reads the store; 26-01 creates and tests it. No file-write conflict, no ordering hazard.
- `resources/action-button.tsx`, `resources/format-relative-date.ts`, `resources/filter-rows.ts`, `resources/format-bytes.ts` — all created by 26-01 and READ by 26-02. Wave-1 parallel safe because 26-02 does NOT write any files in 26-01's `files_modified` list.

If the executor running 26-02 starts before 26-01 lands the store, it WILL fail at module-import resolution. The orchestrator launches both plans in the same wave; checker waits for both. If 26-01 lands first (likely — fewer cross-imports), 26-02 has clear sailing. If 26-02 races ahead, the executor must rebase on 26-01's commits before its GREEN step. This is documented as a `<wave_safety>` clause per `<execution_safety>` below.

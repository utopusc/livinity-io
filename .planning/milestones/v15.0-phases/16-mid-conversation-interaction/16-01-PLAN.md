<objective>
Enable mid-conversation interaction: users can type and send messages while the agent is actively streaming, and can interrupt the agent with a stop button.

Purpose: SDK-05 requires users to interact with the agent mid-task. The entire backend infrastructure exists (AsyncGenerator input channel, sendFollowUp, interrupt) but the UI currently disables input during streaming. This plan removes the streaming lock and wires dual-path sending.

Output: ChatInput stays enabled during streaming with both send and stop buttons visible. Follow-up messages route through sendFollowUp to the existing input channel. Stop button interrupts the agent.
</objective>

   ```typescript
   const handleSend = useCallback(() => {
     const text = input.trim()
     if (!text || agent.isStreaming) return  // BLOCKS during streaming
     setInput('')
     agent.sendMessage(text)

   ```
   With a dual-path version:
   ```typescript
   const handleSend = useCallback(() => {
     const text = input.trim()
     if (!text) return
     setInput('')
     // Reset textarea height after clearing
     const textarea = document.querySelector('textarea')
     if (textarea) textarea.style.height = 'auto'
     if (agent.isStreaming) {
       agent.sendFollowUp(text)
     } else {
       agent.sendMessage(text)
     }
   }, [input, agent])
   ```

3. **No changes to ChatInput props**: The existing `onSend={handleSend}` and `onStop={handleStop}` wiring is sufficient since ChatInput now calls onSend during streaming (from Task 1).

Do NOT modify useAgentSocket -- sendFollowUp and interrupt are already correctly implemented with optimistic user message rendering.

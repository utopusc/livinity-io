<objective>
Fix install.sh so that a single `curl -sSL https://livinity.io/install.sh | bash -s -- --api-key KEY` command on a fresh Ubuntu server results in a fully operational LivOS with auth-server and tor Docker containers running.

Purpose: Currently, install.sh has three bugs that prevent a clean one-command install: (1) setup_docker_images warns but continues on pull failure, meaning compose up will silently fail later; (2) the tor/data directory and correct permissions are never created, so docker compose mount fails; (3) the Kimi CLI section can crash and abort the entire install.

Output: Modified livos/install.sh with hardened image pulls, new setup_docker_prerequisites function, and resilient Kimi handling.
</objective>

From docker-compose.yml (what compose expects):
- tor_proxy service mounts ${LIVINITY_TORRC}:/etc/tor/torrc:ro
- tor_proxy service mounts ${LIVINITY_DATA_DIR}/tor/data:/data
- tor_proxy runs as user "1000:1000"
- auth service needs AUTH_PORT, LIVINITY_AUTH_SECRET, JWT_SECRET, MANAGER_IP, DASHBOARD_IP, LIVINITYD_RPC_HOST
- Network subnet: $NETWORK_IP/16

From app-environment.ts (what livinityd provides at runtime):
- LIVINITY_DATA_DIR = livinityd.dataDirectory (which is /opt/livos/data when started with --data-directory /opt/livos/data)
- LIVINITY_TORRC = ${currentDirname}/tor-proxy-torrc (or tor-server-torrc if torEnabled)
  - currentDirname = the legacy-compat directory under livinityd source
  - These torrc files ALREADY EXIST in the repo at packages/livinityd/source/modules/apps/legacy-compat/
- All env vars (NETWORK_IP, AUTH_IP, AUTH_PORT, etc.) are hardcoded in app-environment.ts, NOT read from .env
- livinityd calls `docker compose up` on startup via appEnvironment()

Key insight: app-environment.ts provides ALL the env vars docker-compose.yml needs at runtime.
The torrc files exist in source. The ONLY things install.sh must ensure:
1. Docker images livos/auth-server:1.0.5 and livos/tor:0.4.7.8 are tagged and available
2. /opt/livos/data/tor/data directory exists with 1000:1000 ownership (compose mounts it)
3. /opt/livos/data/app-data directory exists (compose mounts it via auth service)
4. The install doesn't crash from non-critical failures (Kimi CLI)

  <action>
    Modify livos/install.sh with three changes:

    **Change 1: Harden setup_docker_images() (line 311-347) — per INST-01, INST-02**

    Change the error handling in the for loop. Currently when `docker pull` fails, it does `warn` + `continue`. Change to `fail` so the install stops if Docker images cannot be pulled (without them, compose up will fail anyway, better to fail fast with a clear message):

    Replace:
    ```bash
    if ! docker pull "$src"; then
        warn "Failed to pull $src - will retry on first use"
        continue
    fi
    ```

    With:
    ```bash
    if ! docker pull "$src"; then
        fail "Failed to pull $src — Docker images are required for LivOS. Check your internet connection and retry."
    fi
    ```

    **Change 2: Add setup_docker_prerequisites() function — per INST-03, INST-04**

    Add a new function AFTER the setup_docker_images function (after line 347) that creates the directory structure docker-compose.yml expects:

    ```bash
    setup_docker_prerequisites() {
        info "Preparing Docker container prerequisites..."

        local data_dir="$LIVOS_DIR/data"

        # Create tor data directory (mounted by tor_proxy as /data)
        mkdir -p "$data_dir/tor/data"

        # Create app-data directory (mounted by auth as /app-data)
        mkdir -p "$data_dir/app-data"

        # Tor container runs as 1000:1000, needs ownership on mounted volumes
        chown -R 1000:1000 "$data_dir/tor"

        ok "Docker prerequisites ready (tor/data, app-data directories)"
    }
    ```

    **Change 3: Call setup_docker_prerequisites in main flow**

    In the main install flow, add the call AFTER `build_project` (which creates $LIVOS_DIR/data via `mkdir -p`) and the .env symlink but BEFORE `create_systemd_service` and `start_services`. Insert at approximately line 1138, right before the Kimi CLI section:

    ```bash
    # === Docker Prerequisites ===
    setup_docker_prerequisites
    ```

    The exact insertion point is between the "Symlink .env for Liv" block (line 1132-1137) and the "Kimi CLI" section (line 1139). This ensures data dir exists (created by build_project) before we set up prerequisites.

    **Change 4: Wrap Kimi CLI section in error handling — per INST-05**

    The Kimi CLI section (lines 1139-1153) runs under set -euo pipefail. If kimi --version fails or install-kimi.sh fails, the entire install aborts. Wrap the entire block:

    Replace lines 1139-1153 with:
    ```bash
    # === Kimi CLI (AI provider — optional, non-blocking) ===
    (
        if [[ -f "$NEXUS_DIR/scripts/install-kimi.sh" ]]; then
            if ! command -v kimi &>/dev/null; then
                step "Installing Kimi CLI"
                bash "$NEXUS_DIR/scripts/install-kimi.sh" 2>&1 | tail -5 || true
                # Ensure kimi is globally accessible
                if [[ -f /root/.local/bin/kimi ]]; then
                    ln -sf /root/.local/bin/kimi /usr/local/bin/kimi
                    ln -sf /root/.local/bin/kimi-code /usr/local/bin/kimi-code 2>/dev/null || true
                    ok "Kimi CLI installed: $(kimi --version 2>/dev/null || echo 'installed')"
                fi
            else
                ok "Kimi CLI already installed: $(kimi --version 2>/dev/null)"
            fi
        fi
    ) || warn "Kimi CLI setup skipped (non-critical)"
    ```

    The outer subshell + `|| warn` ensures ANY failure in the Kimi block is caught and the install continues. The inner `|| true` on individual commands provides defense-in-depth.

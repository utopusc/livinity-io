<objective>
Wire SDK WebSocket sessions into Redis conversation persistence so messages survive page refresh.

Purpose: Currently, SDK session messages live only in the browser's useReducer state and vanish on refresh. This plan bridges the WebSocket relay to the existing AiModule Redis conversation storage, so every user/assistant turn is persisted.

Output: AgentSessionManager reports turn completions, ws-agent.ts saves them to Redis via AiModule, and a new tRPC route returns conversation messages for the frontend.
</objective>

   async getOrCreateConversation(id: string, title: string, userId?: string): Promise<Conversation> {
     const existing = await this.getConversation(id, userId);
     if (existing) return existing;
     const conv: Conversation = {
       id,
       title,
       messages: [],
       createdAt: Date.now(),
       updatedAt: Date.now(),

     await this.saveConversation(conv, userId);

   }

**In ai/routes.ts:**

3. Add a `getConversationMessages` route that returns messages in the format the UI ChatMessage type needs. This is different from `getConversation` because it needs to transform the backend ChatMessage format to include fields the frontend expects:

   ```typescript
   getConversationMessages: privateProcedure
     .input(z.object({ id: z.string() }))
     .query(async ({ ctx, input }) => {
       const userId = ctx.currentUser?.id
       const conversation = await ctx.livinityd.ai.getConversation(input.id, userId)
       if (!conversation) return { messages: [] }
       // Transform to UI format: the backend ChatMessage has role, content, toolCalls (legacy format)
       // The frontend ChatMessage needs: id, role, content, toolCalls (with id, name, input, status, output), timestamp
       const messages = conversation.messages.map((msg) => ({
         id: msg.id,
         role: msg.role as 'user' | 'assistant',
         content: msg.content,
         toolCalls: msg.toolCalls?.map((tc, i) => ({
           id: `${msg.id}_tool_${i}`,
           name: tc.tool,
           input: tc.params,
           status: (tc.result.success ? 'complete' : 'error') as 'complete' | 'error',
           output: tc.result.output,
           ...(tc.result.success ? {} : { errorMessage: tc.result.output }),
         })),
         isStreaming: false,
         timestamp: msg.timestamp,
       }))
       return { messages, title: conversation.title }
     }),
   ```

4. Add `getConversationMessages` to `httpOnlyPaths` in the common.ts tRPC config (check file at `livos/packages/livinityd/source/modules/server/trpc/common.ts`) since it needs HTTP transport. Actually, check first if listConversations and getConversation already use HTTP -- they are standard queries so they should work over both WS and HTTP. Only add to httpOnlyPaths if there are issues.

   updateConversationTitle: privateProcedure
     .input(z.object({ id: z.string(), title: z.string().min(1).max(200) }))

  <action>
Fix the tool call lifecycle handling in the useAgentSocket hook. Currently the hook has a critical bug: `content_block_stop` sets tool status to `'complete'` but at that point the tool input JSON is finalized, not the tool execution. The actual tool execution happens AFTER `content_block_stop`.

**Changes to ChatToolCall interface:**
- Add `elapsedSeconds?: number` field to track execution time from `tool_progress` events
- Add `errorMessage?: string` field for error details

**Changes to messagesReducer:**
- In `content_block_stop` case: change `status: 'complete'` to `status: 'running'` -- the tool input is now parsed, but the tool is about to execute (still running). Parse the accumulated JSON and update input, but keep status as `'running'`.

**Changes to handleSdkMessage:**
Add handling for these new SDK message types:

1. `tool_progress` (data.type === 'tool_progress'):
   - Extract `data.tool_use_id` and `data.elapsed_time_seconds`
   - Dispatch `UPDATE_TOOL_CALL` with `{ elapsedSeconds: data.elapsed_time_seconds }`
   - This keeps the spinner showing while updating elapsed time display

2. `tool_use_summary` (data.type === 'tool_use_summary'):
   - Extract `data.summary` and `data.preceding_tool_use_ids`
   - For each tool_use_id in `preceding_tool_use_ids`, dispatch `UPDATE_TOOL_CALL` with `{ status: 'complete', output: data.summary }`
   - This is the signal that tool execution completed successfully

3. `user` message with tool results (data.type === 'user'):
   - Check if `data.message?.content` is an array containing blocks with `type === 'tool_result'`
   - For each `tool_result` block: extract `tool_use_id`, `content` (may be string or array of text blocks), and `is_error`
   - Build output string from content (if array of objects with `text`, join the text; if string, use directly)
   - Dispatch `UPDATE_TOOL_CALL` with:
     - `status: is_error ? 'error' : 'complete'`
     - `output: outputString` (the tool result content)
     - If is_error: `errorMessage: outputString`
   - This provides the actual tool output (file contents, shell output, etc.)

4. In the `assistant` handler (data.type === 'assistant', the non-streaming fallback):
   - After processing tool_use blocks, also check for any tool_result blocks in `data.message?.content`
   - For tool_result blocks, dispatch `UPDATE_TOOL_CALL` with status and output (same logic as user handler)

**Important:** Do NOT handle `tool_progress` or `tool_use_summary` inside the `stream_event` case. These are top-level SDK message types (data.type === 'tool_progress'), not stream events (data.type === 'stream_event' with data.event === ...).

Keep all existing functionality intact -- only add new cases and fix the content_block_stop status.

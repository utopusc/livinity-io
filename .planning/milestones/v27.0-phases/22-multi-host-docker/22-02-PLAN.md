UI for Phase 22: surface multi-environment management to the user. Server Control gets an environment-picker dropdown that re-scopes every Docker view; Settings gets an Environments section for adding/editing/removing remote-host configurations and for generating one-time agent registration tokens.

Purpose: Satisfies MH-03 (environment selector in header). The Settings UI is a natural pairing — without it, users have no way to actually create the environments they'd select. Agent token generation has its UI flow here even though the backend ships in 22-03; 22-03 wires the mutation handler.
Output: User-facing multi-host experience. Picking 'local' shows the local host (today's behavior); picking a remote env scopes all Docker views to that host without page reload.
</objective>

    ```typescript
    import {create} from 'zustand'
    import {persist} from 'zustand/middleware'

    export const LOCAL_ENV_ID = '00000000-0000-0000-0000-000000000000'

    interface EnvironmentStore {
      selectedEnvironmentId: string  // never null; defaults to LOCAL_ENV_ID
      setEnvironment: (id: string) => void
    }

    export const useEnvironmentStore = create<EnvironmentStore>()(
      persist(
        (set) => ({
          selectedEnvironmentId: LOCAL_ENV_ID,
          setEnvironment: (id) => set({selectedEnvironmentId: id}),
        }),
        {
          name: 'livos:selectedEnvironmentId',
          // Only persist the id; the setter is rebuilt on hydrate
          partialize: (state) => ({selectedEnvironmentId: state.selectedEnvironmentId}),
        },
      ),
    )

    // Convenience hook for components that only need the id
    export const useSelectedEnvironmentId = () =>
      useEnvironmentStore((s) => s.selectedEnvironmentId)
    ```

**Section 4: Express middleware mount in server/index.ts**
- Quote lines 1211-1215 (mountBrokerRoutes call).
- Document that `mountBrokerRoutes(app, livinityd)` does `app.use('/u', router)` internally.
- **Decision for Plan 44-02 mount strategy:** insert ONE line BEFORE `mountBrokerRoutes(this.app, this.livinityd)` on line 1215:
  ```typescript
  mountUsageCaptureMiddleware(this.app, this.livinityd)
  mountBrokerRoutes(this.app, this.livinityd)
  ```
  The capture middleware uses `app.use('/u/:userId/v1', captureMiddleware)` — Express middleware ordering means it runs BEFORE the broker handler on the same path prefix. Recommend (per D-44-06): the capture middleware does NOT terminate the request — it patches `res.json`, `res.write`, `res.end` to tap the response stream, then calls `next()` so the broker handler runs normally.

**Section 5: dockerode reverse lookup feasibility (app_id source)**
- Confirm dockerode is already a livinityd dep (used in `docker/ai-resource-watch.ts`, `docker/container-files.ts`, `docker/docker-clients.ts`, `ai/routes.ts`).
- Document the pattern for IP -> container reverse lookup:
  ```typescript
  const docker = new Dockerode({socketPath: '/var/run/docker.sock'})
  const containers = await docker.listContainers({all: false})
  // Each container has NetworkSettings.Networks.<network>.IPAddress
  // Match req.socket.remoteAddress (after stripping ::ffff: prefix) against IPAddress
  ```
- Cost: every captured request triggers a `listContainers` call (~10-50ms on Mini PC). Mitigation: cache `Map<ip, containerName>` with 60s TTL, refresh on miss.
- **Decision for Plan 44-02:** ship app_id reverse lookup with a 60s in-memory IP→container TTL cache. If `listContainers` fails or no match → fall back to `app_id = null` (column allows it). Honest framing per D-44-07: app_id is observability, not security.

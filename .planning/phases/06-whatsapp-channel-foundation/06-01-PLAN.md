    * Pino-compatible logger adapter for Baileys.

   export const baileysLogger = {
     level: 'warn',
     child: () => baileysLogger,

2. Create `nexus/packages/core/src/channels/whatsapp-auth.ts`:
   Redis-backed auth state for Baileys. This replaces the file-based `useMultiFileAuthState` which Baileys docs explicitly warn against using in production (per WA-02).

   The class must:
   - Import `initAuthCreds` and `BufferJSON` from `baileys` (these are named exports from the baileys package)
   - Import `type { AuthenticationCreds, AuthenticationState, SignalDataTypeMap }` from `baileys`
   - Import `Redis` from `ioredis` (named import, NOT default — per project convention)
   - Use Redis key prefix `nexus:wa:auth` for all keys
   - Store creds at `nexus:wa:auth:creds` as JSON string using `BufferJSON.replacer` for serialization
   - Store individual signal keys at `nexus:wa:auth:keys:{type}:{id}` as JSON strings using `BufferJSON.replacer`
   - Use Redis pipeline for batch `keys.get` and `keys.set` operations (atomic, fast)
   - For `keys.set`, when value is null/undefined, DELETE the key (key removal)
   - For `keys.get`, use `BufferJSON.reviver` when parsing stored JSON
   - Provide a `clearAll()` method that deletes all keys matching `nexus:wa:auth:*` (for logout/disconnect)
   - Export the class as `WhatsAppAuthStore`

   The `loadState()` method returns `{ state: AuthenticationState, saveCreds: () => Promise<void> }` — the exact shape Baileys expects when creating a socket.

   Key implementation detail: `BufferJSON.replacer` and `BufferJSON.reviver` handle Buffer serialization in Signal protocol keys. Without these, auth state will corrupt and require QR re-scan.
  </action>

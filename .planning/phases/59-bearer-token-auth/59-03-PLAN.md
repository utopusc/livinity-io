<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External HTTP client → Bearer middleware | Untrusted Authorization header is parsed and used as a SHA-256 input; never logged in plaintext |
| Bearer middleware → cache | Cache stores `{userId, id}` per hash for 60s; an attacker with leaked PG access cannot poison the cache (cache is in-process only, no external write surface) |
| Cache flusher → PG | Background flusher issues UPDATEs only on hashes already in `pendingLastUsed`; cannot escalate to other tables |
| cli.ts dispose | Best-effort cleanup; cache flusher errors are swallowed so process exit is never blocked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-10 | I (Information disclosure) | bearer-auth.ts logging | mitigate | NEVER log `plaintext`; debug-level may log `keyPrefix` only — enforced by code review + future grep guard. Test 8 sets up the timingSafeEqual call assertion. |
| T-59-11 | S (Spoofing) | bearer-auth.ts comparison | mitigate | `crypto.timingSafeEqual` defense-in-depth on `presentedHash` vs `row.key_hash` (RESEARCH.md Pattern 2). Buffer length check before comparison to prevent length-leak. |
| T-59-12 | D (Denial of service) | findApiKeyByHash brute-force | mitigate | Negative cache TTL 5s caps PG QPS from key-guessing attacks (RESEARCH.md Specific Idea); 39-char keys with ~190 bits entropy make brute force infeasible regardless |
| T-59-13 | T (Tampering) | mount order | mitigate | mount-order.test.ts string-asserts position(usage) < position(bearer) < position(broker) — catches reorder regression at CI |
| T-59-14 | R (Repudiation) | last_used_at flush | mitigate | Dispose hook (cli.ts) flushes pending writes on shutdown so `last_used_at` is never lost (RESEARCH.md Pitfall 2). Best-effort: PG outage during flush is logged but never blocks exit |
| T-59-15 | I | PG outage 500 leak | mitigate | `findApiKeyByHash` errors caught and converted to 401 invalid (RESEARCH.md Pattern 3) — server never returns a 500 with stack trace to unauthenticated callers |
</threat_model>

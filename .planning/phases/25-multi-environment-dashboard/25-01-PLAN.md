<objective>
Replace the Phase 24 Dashboard placeholder with a multi-environment health card grid (DOC-04) and add the `tags` column to the `environments` table that DOC-06 (filter chips, Plan 25-02) will consume.

Purpose: Surface every Docker environment's health at a glance so operators can spot unreachable hosts, restart-loops, and resource churn without round-tripping through the env selector. Establish the per-env query composition pattern (one hook per metric, env-id override) that downstream phases will reuse for cross-env aggregation.

Output: PostgreSQL schema migration (one-line ALTER TABLE), Environment type/CRUD field additions, tRPC schema additions (createEnvironment/updateEnvironment accept `tags`), Dashboard section component, EnvCard + EnvCardGrid + useEnvCardData hook, format-events pure utility module with unit tests, useEnvironment(envId) cache-derived getter.
</objective>

<threat_model>
Security enforcement is enabled by default in this project — see CLAUDE.md / mandatory-initial-read. The Phase 25 Dashboard is a read-mostly UI surface; the only WRITE we add is the optional `tags: string[]` field on existing admin-only mutations. Trust boundaries below.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → tRPC | Authenticated admin user submits createEnvironment/updateEnvironment with new optional `tags` field. |
| tRPC → PG | Tagged INSERT/UPDATE statements parameterized via $N — no string concatenation. |
| PG → UI | listEnvironments returns tags array; UI renders inside a Tailwind chip. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Tampering | tags array on createEnvironment input | mitigate | Zod schema bounds: `z.array(z.string().min(1).max(50)).max(20).optional()` — blocks both null bytes via z.string() type coercion AND DOS via 10k-element arrays. |
| T-25-02 | Information Disclosure | UI tag chip render | mitigate | React renders strings as text nodes (auto-escapes) — no `dangerouslySetInnerHTML` for any tag rendering. |
| T-25-03 | Tampering | SQL injection via tags array | mitigate | Tags pass through pg parameterized queries ($N positional) — never string-concatenated. Same defense the rest of environments.ts uses. |
| T-25-04 | Denial of Service | Polling fanout: every env card hits 6 queries every 30s minimum (1 per metric). Many envs × many users = N×M load. | accept | Local-only constraint for Plan 25-01 (typical user has 1-3 envs); Plan 22-01 D-06 already documents the multi-host fanout limit. UI pollers staggered naturally by React-Query's per-mount initial fetch. Aggregator query (`docker.dashboardSnapshot`) deferred — re-evaluate at 5+ envs. |
| T-25-05 | Elevation of Privilege | tags column read by non-admin users | accept | Both createEnvironment and updateEnvironment are `adminProcedure`-gated (existing per Plan 22-01); listEnvironments is also adminProcedure. Tags inherit the same gate. Member/guest roles cannot read or write env data at all. |
| T-25-06 | Repudiation | Tag changes not audited | accept | The `environments` table has no audit log (no other field changes are audited). Tags are operational metadata, not security state. Out of scope per scope-boundary rule. |
| T-25-07 | Spoofing | Card click sets selectedEnvironmentId without re-auth | accept | The selection is purely client-side; every backend call still passes through adminProcedure auth + session JWT. The store change has no security effect. |

```sql
-- =========================================================================
-- Custom Domains (Phase 08 - custom domain routing)
-- =========================================================================
CREATE TABLE IF NOT EXISTS custom_domains (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  domain            TEXT NOT NULL UNIQUE,
  verification_token TEXT NOT NULL,
  status            TEXT NOT NULL DEFAULT 'pending_dns',
  dns_a_verified    BOOLEAN NOT NULL DEFAULT false,
  dns_txt_verified  BOOLEAN NOT NULL DEFAULT false,
  error_message     TEXT,
  last_dns_check    TIMESTAMPTZ,
  verified_at       TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_custom_domains_user_id ON custom_domains(user_id);
CREATE INDEX IF NOT EXISTS idx_custom_domains_domain ON custom_domains(domain);
CREATE INDEX IF NOT EXISTS idx_custom_domains_status ON custom_domains(status);
```

This mirrors the Phase 07 Drizzle schema columns exactly. The relay's platform PostgreSQL is the same database as the platform/web app (both on Server5), so this table may already exist from Phase 07's migration. The `CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS` are idempotent.

2. Create `platform/relay/src/custom-domains.ts` with these exports:

- `CustomDomainInfo` interface: `{ userId: string; username: string; domain: string; status: string }`
- `lookupCustomDomain(pool: pg.Pool, redis: Redis, hostname: string): Promise<CustomDomainInfo | null>` -- Checks Redis cache first (`relay:custom-domain:{hostname}`), falls back to DB query joining custom_domains with users table to get username. Only returns domains where status is `dns_verified` or `active`. Caches result in Redis with 60-second TTL. Caches negative results (null) with 30-second TTL to prevent repeated DB queries for unknown domains.
- `isCustomDomainAuthorized(pool: pg.Pool, redis: Redis, domain: string): Promise<boolean>` -- Lighter version for ask endpoint: checks Redis cache key `relay:custom-domain-auth:{domain}`, falls back to `SELECT 1 FROM custom_domains WHERE domain = $1 AND status IN ('dns_verified', 'active') LIMIT 1`. Caches boolean in Redis with 60s TTL.
- `warmDomainCache(pool: pg.Pool, redis: Redis): Promise<void>` -- Called on relay startup. Queries all verified/active custom domains and pre-populates Redis cache.
- `invalidateDomainCache(redis: Redis, domain: string): Promise<void>` -- Deletes both `relay:custom-domain:{domain}` and `relay:custom-domain-auth:{domain}` keys.

For the ask endpoint check, also handle parent domain lookup: if `blog.mysite.com` is requested, check both `blog.mysite.com` AND `mysite.com` in the custom_domains table. This allows subdomains of verified custom domains to get certificates.

Redis key format:
- `relay:custom-domain:{hostname}` -> JSON string of CustomDomainInfo or `"null"` for negative cache
- `relay:custom-domain-auth:{domain}` -> `"1"` (authorized) or `"0"` (not authorized)

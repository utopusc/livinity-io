<objective>
Close the verification gap for Phase 08 Truth #4: "User can pause, resume, or stop the session from the LivOS UI, and the AI responds immediately to those controls."

The root cause is that stopComputerUse only deletes chatStatus visually but never aborts the SSE fetch stream from Nexus. The agent loop keeps running. The fix: store an AbortController per conversation, use its signal in the SSE fetch call, and have stopComputerUse call controller.abort().

Pause remains cosmetic (UI shows "Paused" badge, agent continues) -- this is the pragmatic approach since true pause would require bidirectional signaling between livinityd and the Nexus agent loop, which is architecturally complex for minimal user benefit. The stop control is what matters for safety.

Purpose: Make the stop button actually stop the AI from continuing to control the device.
Output: Modified ai/index.ts with AbortController plumbing, modified routes.ts with abort call in stopComputerUse.
</objective>

```typescript
stopComputerUse: privateProcedure.input(z.object({conversationId: z.string()})).mutation(({ctx, input}) => {
  const status = ctx.livinityd.ai.chatStatus.get(input.conversationId)
  if (!status) {
```

```typescript
  // Abort the active SSE stream to nexus — this kills the agent loop
  const controller = ctx.livinityd.ai.activeStreams.get(input.conversationId)

  return {ok: true}
}),
```

3. chat() catch block detects AbortError and sets a clean "stopped by user" message
4. chat() finally block cleans up the controller from activeStreams

The chatStatus.delete stays — it removes the visual status immediately so the frontend sees the session as ended.

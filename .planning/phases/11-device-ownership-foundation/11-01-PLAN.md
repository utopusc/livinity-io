<objective>
Enforce the OWN-01 invariant at the database layer: every row in the platform `devices` table is bound to exactly one existing user via a real foreign-key constraint. Today `user_id UUID NOT NULL` exists as a column but there is NO FK reference to `users(id)` — so a malformed INSERT could write an orphan UUID, and a user deletion would silently leave dangling devices. This plan adds the FK, backfills any legacy NULL rows (defensive — schema.ts already says NOT NULL but relay DB may have pre-NOT-NULL history), ensures the index exists, and hardens `createDeviceRecord` against being called without a userId.

Purpose: Close the schema-level loophole so Phase 12 authorization middleware can trust the DB to be the source of truth for device ownership. Without this, a stale or hand-inserted device row with a random `user_id` could silently pass ownership checks.

Output: Two matching DB schema changes (platform/web migration + platform/relay schema.sql mirror), Drizzle schema update, and a defensive guard in the device-auth helper.
</objective>

Create a new migration file `platform/web/src/db/migrations/0007_device_user_id_fk.sql` containing this exact SQL. Every statement must be idempotent (use DO $$ blocks or IF NOT EXISTS / IF EXISTS guards) because migrations may run multiple times on platform restart.

```sql
-- =========================================================================
-- Migration 0007: Add FK constraint devices.user_id -> users(id) (v26.0 Phase 11 OWN-01)
-- =========================================================================

-- Step 1: Defensive backfill. schema.ts already declares user_id NOT NULL, but the
-- relay DB may have rows from before that constraint was added. Any NULL user_id
-- gets assigned to the oldest admin (fallback: oldest user by created_at if no admin
-- concept exists in platform/relay users schema).
UPDATE devices
SET user_id = (SELECT id FROM users ORDER BY created_at ASC LIMIT 1)
WHERE user_id IS NULL;

-- Step 2: Re-enforce NOT NULL (idempotent — no-op if already NOT NULL).
ALTER TABLE devices ALTER COLUMN user_id SET NOT NULL;

-- Step 3: Add FK constraint. Use ON DELETE RESTRICT so deleting a user with active
-- devices FAILS loudly rather than silently cascading — v26.0 demands explicit
-- device lifecycle management, not implicit delete propagation.
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_constraint
    WHERE conname = 'devices_user_id_fkey' AND conrelid = 'devices'::regclass
  ) THEN
    ALTER TABLE devices
      ADD CONSTRAINT devices_user_id_fkey
      FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE RESTRICT;
  END IF;
END $$;

-- Step 4: Ensure index exists (migration 0004 already creates this, but we add it
-- here for safety in case an environment ran the schema without the index).
CREATE INDEX IF NOT EXISTS idx_devices_user_id ON devices(user_id);
```

Rationale for choices:
- ON DELETE RESTRICT (not CASCADE): v26.0 treats device records as security artifacts; silent cascade delete on user removal would erase audit history. Admin must explicitly revoke devices first.
- Backfill picks oldest user (created_at ASC) — in platform/relay users schema there is no `role` column, so we cannot literally query `WHERE role = 'admin'`. The oldest user is the deployment's first/owning account by convention.
- Named constraint `devices_user_id_fkey` (Postgres default naming convention) so later migrations/Drizzle introspection can reference it.
- No DROP/RECREATE — pure additive changes so rollback is safe and existing rows are preserved.

```typescript
export async function createDeviceRecord(userId: string, deviceInfo: {
  deviceName: string;

}): Promise<string> {
  // OWN-01 hard invariant: device records MUST be bound to a user at insert time.
  // The FK constraint (migration 0007) enforces this at the DB level; we also
  // guard in application code for a clearer error message and early rejection.
  if (!userId || typeof userId !== 'string' || userId.length === 0) {

     VALUES ($1, gen_random_uuid(), $2, $3)
     RETURNING device_id`,
    [userId, deviceInfo.deviceName, deviceInfo.platform]
  );
  return result.rows[0].device_id;
}
```

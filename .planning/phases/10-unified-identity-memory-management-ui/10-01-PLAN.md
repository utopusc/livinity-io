    **1. PostgreSQL schema (schema.sql):** Append to end of file:
    ```sql
    -- =========================================================================
    -- Channel Identity Map (unified cross-channel userId, v25.0 Phase 10)
    -- =========================================================================
    CREATE TABLE IF NOT EXISTS channel_identity_map (
      id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      user_id     TEXT NOT NULL,
      channel     TEXT NOT NULL,
      channel_user_id TEXT NOT NULL,
      linked_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      UNIQUE(channel, channel_user_id)
    );

    CREATE INDEX IF NOT EXISTS idx_cim_user ON channel_identity_map(user_id);
    CREATE INDEX IF NOT EXISTS idx_cim_channel_user ON channel_identity_map(channel, channel_user_id);
    ```
    The `user_id` is the canonical ID (e.g., the LivOS PostgreSQL user UUID or "admin" for single-user). The `channel_user_id` is the platform-specific ID (Telegram chat ID, WhatsApp JID, etc.).

    **2. Database module (database/index.ts):** Add two exported functions:
    - `resolveCanonicalUserId(channel: string, channelUserId: string): Promise<string>` — queries `channel_identity_map` for the canonical `user_id`. If no mapping exists, auto-creates one using `channelUserId` as both canonical and channel user ID (so first-time users get a mapping automatically). Returns the canonical userId. If PostgreSQL pool is null (no DB), falls back to returning `channelUserId` unchanged.
    - `linkChannelIdentity(userId: string, channel: string, channelUserId: string): Promise<void>` — upserts a row into `channel_identity_map` (ON CONFLICT DO UPDATE SET user_id = $1, for admin manual linking in the future).

    **3. Memory service endpoints (memory/src/index.ts):** Add two new endpoints:
    - `GET /conversation-turns/:userId` — query params: `limit` (default 50), `channel` (optional filter), `offset` (default 0). Returns `{ turns: [{id, userId, channel, chatId, role, content, metadata, createdAt}], total: number }`. Query conversation_turns WHERE user_id = ? ORDER BY created_at DESC, with optional channel filter. Also return a `SELECT COUNT(*)` for total. Parse metadata JSON if present.
    - `DELETE /conversation-turns/:id` — deletes a single conversation turn by id. Also delete from FTS5: `INSERT INTO conversation_turns_fts(conversation_turns_fts, rowid, content) VALUES('delete', ?, ?)` where you first SELECT the content. Returns `{ success: true }`.

    **4. Daemon identity resolution (daemon.ts):** In `saveChannelTurn()`, before calling `archiveToMemory()`, resolve the canonical userId:
    - Import `resolveCanonicalUserId` — BUT daemon.ts is in nexus/packages/core which does NOT have access to livinityd's database module. Instead, use Redis as an identity cache:
      - Add a private method `resolveCanonicalUserId(channel: string, chatId: string): Promise<string>` that checks Redis key `nexus:identity:${channel}:${chatId}`. If found, return the cached canonical userId. If not found, return chatId as-is (the PostgreSQL mapping will be created lazily by livinityd).
      - In `saveChannelTurn()`, call `const canonicalId = await this.resolveCanonicalUserId(channel, chatId)` and pass `canonicalId` instead of `chatId` as the first argument to both `archiveToMemory()` calls.
    - Also add a method `linkIdentity(channel: string, channelUserId: string, canonicalUserId: string): Promise<void>` that sets the Redis cache key.

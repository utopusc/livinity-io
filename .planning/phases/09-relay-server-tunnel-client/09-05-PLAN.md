Purpose: Bandwidth metering is essential for the free tier model. Without tracking, there's no way to enforce limits. Without quota enforcement, a single user could consume unlimited resources.
Output: Working bandwidth tracking pipeline from Redis hot counters to PostgreSQL cold storage, with quota enforcement at the request level.

  <action>
Create `bandwidth.ts` with these exports:

**Helper: `currentMonth(): string`**
- Returns "YYYY-MM" format for the current month (e.g., "2026-03")

**`trackBandwidth(redis: Redis, userId: string, direction: 'in' | 'out', bytes: number): Promise<void>`**
- Redis key format: `bandwidth:{userId}:{direction}:{currentMonth()}`
- Use `redis.incrby(key, bytes)` -- atomic, no race condition
- Set expiry on key: `redis.expire(key, 90 * 24 * 60 * 60)` (90 days) -- only sets if not already set (use `redis.expire` which is idempotent, or use `redis.pipeline()` with INCRBY + EXPIRE for atomicity)
- Pipeline both commands for efficiency: `redis.pipeline().incrby(key, bytes).expire(key, 90 * 24 * 60 * 60).exec()`

**`checkQuota(redis: Redis, userId: string): Promise<{ allowed: boolean; usedBytes: number; limitBytes: number }>`**
- Get both in and out counters for current month
- Use `redis.mget()` with both keys for efficiency
- Parse to integers (default 0 if null)
- total = inBytes + outBytes
- allowed = total < config.FREE_TIER_BYTES
- Return { allowed, usedBytes: total, limitBytes: config.FREE_TIER_BYTES }

**`startBandwidthFlush(redis: Redis, pool: pg.Pool): NodeJS.Timeout`**
- Start a setInterval at config.BANDWIDTH_FLUSH_INTERVAL_MS (60s)
- Each tick calls `flushBandwidthToPostgres(redis, pool)`
- Return the interval handle for cleanup

**`stopBandwidthFlush(interval: NodeJS.Timeout): void`**
- clearInterval(interval)

**Private: `flushBandwidthToPostgres(redis: Redis, pool: pg.Pool): Promise<void>`**
- Scan for keys matching `bandwidth:*` using `redis.keys('bandwidth:*')` -- at 50-100 users this is fine
- For each key:
  1. Parse key: `bandwidth:{userId}:{direction}:{month}`
  2. Get value with `redis.get(key)`, parse to int
  3. If value is 0, skip
  4. Determine column: direction === 'in' ? 'bytes_in' : 'bytes_out'
  5. Upsert to PostgreSQL:
     ```sql
     INSERT INTO bandwidth_usage (user_id, period_month, {column})
     VALUES ($1, $2, $3)
     ON CONFLICT (user_id, period_month)
     DO UPDATE SET {column} = bandwidth_usage.{column} + $3,
                   updated_at = NOW()
     ```
  6. After successful DB write: `redis.decrby(key, bytes)` -- use DECRBY (not DEL) to handle concurrent writes between GET and DECRBY
- Wrap entire flush in try/catch -- log errors but don't crash. Bandwidth flushing is best-effort.
- Log: `Flushed bandwidth: {count} keys processed`

**Important notes:**
- Use `import { Redis } from 'ioredis'` (named export)
- Use `import pg from 'pg'` for Pool type
- The column name in the SQL is dynamic but from a fixed set ('bytes_in' | 'bytes_out') -- this is safe from SQL injection since it's derived from 'in' | 'out', not user input
- Do NOT use parameterized queries for the column name (PostgreSQL doesn't support parameterized identifiers). The direction-to-column mapping is hardcoded.

**server.ts** -- Add quota check before proxying:
- Import `checkQuota` from bandwidth.ts
- The request handler needs access to Redis (it should already have it from createRequestHandler parameters, or add it)
- After finding the tunnel but BEFORE calling proxyHttpRequest:
  1. Get userId from tunnel connection: `tunnel.userId`
  2. Call `await checkQuota(redis, tunnel.userId)`
  3. If NOT allowed: respond with 429 status, set headers `Retry-After: 86400`, `Content-Type: text/plain`, body: "Bandwidth quota exceeded. Your monthly limit resets on {first day of next month}."
  4. Also send a TunnelQuotaExceeded message through the tunnel: `{ type: 'quota_exceeded', usedBytes, limitBytes, resetsAt: firstDayOfNextMonth.toISOString() }`
  5. If allowed: proceed with proxyHttpRequest as normal

**request-proxy.ts** -- Track bandwidth on both send and receive:
- Import `trackBandwidth` from bandwidth.ts
- The proxy function needs access to Redis and userId. Add these as parameters or get them from the tunnel connection object.
- After building the TunnelRequest message (before sending):
  1. Calculate outbound bytes: `Buffer.byteLength(JSON.stringify(message))`
  2. Call `trackBandwidth(redis, tunnel.userId, 'out', outboundBytes)` -- fire-and-forget (don't await, just call and ignore the promise or add .catch for logging)
- In handleTunnelResponse (when response comes back):
  1. Calculate inbound bytes: `Buffer.byteLength(JSON.stringify(msg))` (the raw message size)
  2. Call `trackBandwidth(redis, tunnel.userId, 'in', inboundBytes)` -- fire-and-forget

**Note:** Track the JSON envelope size, not just the body. The envelope includes headers which can be significant. This is simpler and more accurate than trying to reconstruct the original HTTP bytes.

**index.ts** -- Start and stop bandwidth flush:
- Import `startBandwidthFlush` and `stopBandwidthFlush`
- After creating Redis and Pool connections, call: `const bandwidthInterval = startBandwidthFlush(redis, pool)`
- In the SIGTERM/SIGINT handler, call: `stopBandwidthFlush(bandwidthInterval)` before closing pool and Redis
- Log: "Bandwidth flush started (every ${config.BANDWIDTH_FLUSH_INTERVAL_MS / 1000}s)"

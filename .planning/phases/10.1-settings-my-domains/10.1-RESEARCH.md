# Phase 10.1: Settings My Domains - Research

**Researched:** 2026-03-26
**Domain:** React Settings UI + tRPC integration (LivOS frontend)
**Confidence:** HIGH

## Summary

Phase 10.1 is a focused UI replacement task. The old "Domain & HTTPS" section in Settings (a multi-step wizard for manual domain setup, DNS verification, and Caddy activation) must be removed and replaced with a "My Domains" section that displays domains synced from livinity.io via the tunnel relay. Users manage domains on livinity.io dashboard; they appear here automatically via the existing `domain_sync` / `domain_list_sync` tunnel protocol.

The critical finding is that **all backend routes already exist**. The `domain.platform.listCustomDomains`, `domain.platform.updateAppMapping`, and `domain.platform.removeCustomDomain` tRPC procedures were built in Phase 09/10 and are fully functional. Furthermore, a nearly identical UI component (`DomainsTab` in `server-control/domains-tab.tsx`) already exists in the Servers app. This phase is essentially: (1) swap the Settings menu item, (2) create a `MyDomainsSection` component adapted from the existing `DomainsTab` pattern, and (3) clean up old domain wizard references from settings-content.tsx.

**Primary recommendation:** Reuse the existing tRPC routes and adapt the `DomainsTab` UI pattern for the Settings context. No new backend code needed. The Caddy integration mentioned in CONTEXT.md should be limited to awareness -- in tunnel mode (the primary use case), LivOS Caddy stays as `:80 { reverse_proxy localhost:8080 }` and the relay handles TLS for custom domains.

### Anti-Patterns to Avoid
- **Creating new tRPC routes when existing ones work:** The `domain.platform.listCustomDomains`, `domain.platform.updateAppMapping`, and `domain.platform.removeCustomDomain` routes already exist and are registered in `httpOnlyPaths`. Creating duplicate `domain.myDomains.*` routes adds unnecessary code. Recommendation: use existing routes, or if CONTEXT.md insists on new names, create thin aliases.
- **Modifying Caddy in tunnel mode:** STATE.md decision says "LivOS in tunnel mode does NOT need Caddy changes -- relay terminates TLS." The CONTEXT.md mentions Caddy integration, but for tunnel-mode users (the primary use case), calling `applyCaddyConfig()` would be incorrect -- Caddy should stay as `:80 { reverse_proxy localhost:8080 }`.
- **Deleting domain-setup.tsx:** CONTEXT.md says do NOT delete `domain/routes.ts` or `domain/caddy.ts`. The old `domain-setup.tsx` UI file can remain (it also serves `/settings/domain-setup` route), but its references from `settings-content.tsx` should be removed.

## Common Pitfalls

### Pitfall 1: SettingsSection Type Union Not Updated
**What goes wrong:** TypeScript error because `'my-domains'` is not in the `SettingsSection` type union
**Why it happens:** The type is defined as a union of string literals at line ~114 of settings-content.tsx
**How to avoid:** Replace `'domain'` with `'my-domains'` in the `SettingsSection` type union AND in `MENU_ITEMS` AND in the switch statement
**Warning signs:** TypeScript compilation error

### Pitfall 2: httpOnlyPaths Missing for New Routes
**What goes wrong:** tRPC mutations hang because they're routed to disconnected WebSocket
**Why it happens:** New mutation paths not in `httpOnlyPaths` in `common.ts`
**How to avoid:** If creating new route names (not reusing `domain.platform.*`), add them to `httpOnlyPaths`. The existing `domain.platform.*` routes are already listed.
**Warning signs:** Mutation calls timeout or never resolve

### Pitfall 3: Caddy Integration in Tunnel Mode
**What goes wrong:** User maps domain to app, Caddy gets reconfigured with domain block, breaks tunnel-mode `:80` config
**Why it happens:** CONTEXT.md mentions Caddy integration, but the architecture decision (STATE.md) says LivOS in tunnel mode does NOT need Caddy changes
**How to avoid:** Check if tunnel mode is active before calling `applyCaddyConfig()`. In tunnel mode, the mapping is stored in Redis/PG and the relay handles routing -- no local Caddy change needed. Only apply Caddy changes when in direct-domain mode (non-tunnel).
**Warning signs:** After mapping a domain, existing tunnel access breaks

### Pitfall 4: Removing Too Much from DomainSection
**What goes wrong:** Breaking the `/settings/domain-setup` standalone route or other places that import from `domain-setup.tsx`
**Why it happens:** Aggressively deleting the old domain wizard
**How to avoid:** Only remove: (1) the `DomainSection` function, (2) the `DomainSetupInner` lazy import, (3) the `case 'domain'` in the switch -- all within `settings-content.tsx`. Keep `domain-setup.tsx` and `domain-setup-dialog.tsx` files intact.
**Warning signs:** Build errors about missing imports

### Pitfall 5: `listContainers` Requires Admin Role
**What goes wrong:** Non-admin users see empty container list or permission error
**Why it happens:** `docker.listContainers` uses `adminProcedure` -- but `My Domains` is also admin-only so this is fine
**How to avoid:** Verify the section is already gated by `adminOnly: true` in the menu item (it is)
**Warning signs:** N/A -- this is actually not a problem since both are admin-only

## Code Examples

### Example 1: MyDomainsSection Component Structure
```typescript
// Adapted from domains-tab.tsx for Settings context
// Source: livos/packages/ui/src/routes/server-control/domains-tab.tsx

function MyDomainsSection() {
  const domainsQuery = trpcReact.domain.platform.listCustomDomains.useQuery(undefined, {
    refetchInterval: 10_000,
  })
  const containersQuery = trpcReact.docker.listContainers.useQuery(undefined, {
    refetchInterval: 10_000,
  })
  const updateMappingMutation = trpcReact.domain.platform.updateAppMapping.useMutation({
    onSuccess: () => domainsQuery.refetch(),
  })

  const domains = (domainsQuery.data ?? [])
  const runningContainers = (containersQuery.data ?? []).filter((c) => c.state === 'running')

  if (domainsQuery.isLoading) return <LoadingState />
  if (domains.length === 0) return <EmptyState />

  return (
    <div className='space-y-3'>
      {domains.map((domain) => (
        <DomainCard
          key={domain.domain}
          domain={domain}
          containers={runningContainers}
          onUpdateMapping={updateMappingMutation.mutate}
        />
      ))}
    </div>
  )
}
```

## Open Questions

1. **New routes vs. reuse existing routes?**
   - What we know: CONTEXT.md specifies `domain.myDomains.list/configure/remove` but `domain.platform.listCustomDomains/updateAppMapping/removeCustomDomain` already exist and do the exact same thing
   - What's unclear: Whether the user wants new route names for clarity or is okay reusing existing ones
   - Recommendation: Reuse existing `domain.platform.*` routes. They are already in `httpOnlyPaths`, tested, and functional. Creating aliases adds unnecessary code duplication. If the planner wants new names, create thin wrappers that delegate to the existing implementations.

2. **Caddy integration scope**
   - What we know: CONTEXT.md says "When user maps domain -> call existing caddy.ts applyCaddyConfig()." But STATE.md decision says "LivOS in tunnel mode does NOT need Caddy changes -- relay terminates TLS."
   - What's unclear: Whether Caddy integration should only apply in direct-domain mode (non-tunnel)
   - Recommendation: Check tunnel mode before applying Caddy config. In tunnel mode, just store the mapping (Redis+PG) and let the relay handle routing. Only call `applyCaddyConfig()` in direct-domain mode. This matches the architecture decision.

3. **Should domain-setup.tsx and domain-setup-dialog.tsx be deleted?**
   - What we know: CONTEXT.md says "Do NOT delete domain/routes.ts or domain/caddy.ts." It does not explicitly say whether to delete the old UI files.
   - What's unclear: Whether there are other consumers of these files
   - Recommendation: Keep the files. The `/settings/domain-setup` route in `index.tsx` still references `DomainSetupPage`. Removing the files would break that route. Just remove the references from `settings-content.tsx`.

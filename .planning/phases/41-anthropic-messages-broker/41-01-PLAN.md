    Recommended strategy (document in AUDIT.md Section 7 sub-section "AI Chat carry-forward design"):

    1. **livinityd side (livos/packages/livinityd/source/modules/ai/index.ts:470)**: Add `X-LivOS-User-Id: ${userId}` header to the proxy fetch call when both (a) `userId` is defined (chat handler already accepts it as 4th arg, line 395) AND (b) `await isMultiUserMode(livinityd)` returns true. Single-user mode: do NOT send the header (preserves byte-identical behavior — existing single-user requests look unchanged on the wire).

    2. **nexus side (nexus/packages/core/src/api.ts:2399)**: After existing `extractUserIdFromRequest(req)` call (line 2407), check for `X-LivOS-User-Id` header as a higher-priority source ONLY when LIV_API_KEY auth has succeeded (which is enforced earlier in the request via `requireApiKey` middleware on the registerApi router at api.ts:11). The X-LivOS-User-Id header is trusted ONLY because LIV_API_KEY auth gates the entire route — livinityd is the only authenticated source that can send it.

    3. **homeOverride threading**: When userId is resolved (via either path), construct the per-user .claude/ path as `path.join(LIVINITYD_DATA_DIR, 'users', userId, '.claude')` and pass into agentConfig as `homeOverride`. The data dir constant is needed in nexus — document its source. Two options to choose from in Plan 41-04: (a) hardcode `/opt/livos/data` (matches MEMORY.md and PROJECT.md production layout), (b) read from env var `LIVOS_DATA_DIR` defaulting to `/opt/livos/data`. Plan 41-04 picks; recommend (b) for testability.

    4. **Single-user mode preservation**: When the X-LivOS-User-Id header is absent OR LIV_API_KEY is unset OR multi-user mode is off (livinityd doesn't send it), homeOverride is NOT set, and SdkAgentRunner falls back to `process.env.HOME || '/root'` — the pre-Phase-41 behavior. Plan 41-04's regression test asserts this byte-identically.

    Document in AUDIT.md Section 7 sub-section "Container source IP guard implementation":
    - Per D-41-08: explicit allowlist of `127.0.0.1`, `::1` (IPv6 loopback), and CIDR `172.17.0.0/16` (Docker default bridge). Reject everything else with 401.
    - Implementation note: `req.socket.remoteAddress` returns IP. For IPv4-mapped-IPv6 (`::ffff:127.0.0.1`), strip the prefix before matching. Pull the CIDR check from a small inline helper — no new dependency. Test cases: 127.0.0.1 → allow, ::1 → allow, 172.17.0.5 → allow, 172.17.255.254 → allow, 172.18.0.1 → reject (different bridge), 8.8.8.8 → reject.

    Document in AUDIT.md Section 7 sub-section "SSE streaming format reference":
    - Cite Anthropic Messages streaming spec: https://docs.anthropic.com/en/api/messages-streaming
    - Required event sequence per Anthropic spec: `message_start` → `content_block_start` → 1+ `content_block_delta` → `content_block_stop` → `message_delta` → `message_stop`. Plus `ping` events between for keep-alive.
    - SSE wire format per spec: `event: <type>\ndata: <json>\n\n` (note: the `event:` line is REQUIRED per Anthropic spec, not just `data:`).
    - Plan 41-03 implements this mapping; Plan 41-05 validates with a captured stream replay against the schema.

Create `nexus/packages/core/src/channels/whatsapp-rate-limiter.ts` with a `WhatsAppRateLimiter` class:

**Constructor:** Takes `redis: Redis` instance.

**Core mechanism — Redis sliding window (ZRANGEBYSCORE pattern):**
- Redis sorted set key: `nexus:whatsapp:ratelimit`
- Each send adds `Date.now()` as both score and member (use `ZADD`)
- Before each send, remove entries older than 60s (`ZREMRANGEBYSCORE key 0 now-60000`)
- Count remaining entries (`ZCARD key`) — if >= 10, message is queued
- Set key TTL to 120s (`EXPIRE`) to auto-cleanup

**Constants:**
- `MAX_PER_MINUTE = 10`
- `MIN_DELAY_MS = 1000` (1 second)
- `MAX_DELAY_MS = 3000` (3 seconds)

**Public method `enqueue(sendFn: () => Promise<boolean>): Promise<boolean>`:**
1. Check sliding window count
2. If under limit: add entry to sorted set, apply random delay (`MIN_DELAY_MS + Math.random() * (MAX_DELAY_MS - MIN_DELAY_MS)`), then call `sendFn()`
3. If at limit: push to an in-memory queue (array of `{ sendFn, resolve, reject }`)
4. Return a Promise that resolves when the message is actually sent

**Private method `processQueue(): void`:**
- Called after each successful send
- Uses `setTimeout` to check sliding window again after 6s (60s / MAX_PER_MINUTE)
- If under limit, dequeue next item, apply random delay, send it
- Recursive: keeps draining until queue is empty
- Guard against concurrent processing (use a `processing` boolean flag)

**Export:** `export class WhatsAppRateLimiter`

Import `Redis` type from ioredis. Import `logger` from `../logger.js`.

Log at info level when a message is queued (include queue depth), and when a queued message is finally sent.

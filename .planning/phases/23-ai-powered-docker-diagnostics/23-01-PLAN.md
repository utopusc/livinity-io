<objective>
Ship Phase 23 reactive AI diagnostics: one-click container log analysis (AID-01), natural-language compose generation (AID-03), and vulnerability-scan plain-English explainer (AID-04). All three flow through a single new Kimi-completion bridge: a fresh `POST /api/kimi/chat` one-shot endpoint on nexus-core that wraps `brain.chat()` with a system+user prompt and no tools, called from livinityd via fetch with `X-API-Key: $LIV_API_KEY`. Livinityd's `ai-diagnostics.ts` engine redacts secrets, trims logs to 200 lines, builds task-specific prompts, and caches results in Redis. UI surfaces match existing patterns: AI Diagnose button on the container detail Info tab (mirroring the Redeploy button shape), an "AI" tab next to YAML/Git in DeployStackForm (extends the existing Tabs primitive), and an "Explain CVEs" button inside ScanResultPanel (mirrors the Cached/Rescan button shape).

Purpose: Closes 3 of the 5 AID requirements. These three are batched together because they share the Kimi-completion bridge (`callKimi` + `/api/kimi/chat`) and the secret-redaction utility — separating them would force a redundant scaffolding step in each plan.

Output: One new nexus HTTP endpoint, one new livinityd module + unit tests, three new tRPC mutations, one new UI hook, and three UI integration points. After this plan ships, Phase 23 still owes AID-02 (proactive resource watch) and AID-05 (AI Chat sidebar autonomous diagnostics) — both delivered by Plan 23-02.
</objective>

<why_now>
Phase 23 is the AI-diagnostics moat — the unique value proposition of the v27.0 milestone (per ROADMAP.md `v27.0 Roadmap Decisions`). Phases 17/19 prerequisites are satisfied: Phase 17 expanded the AI tool surface and shipped the Kimi provider, Phase 19 shipped `getCachedScan`/`scanImage` with the `VulnScanResult` shape this plan consumes for AID-04. Reactive features ship first because the user-facing button → API → result loop is the tightest possible feedback cycle for validating the Kimi prompt quality before we wire it into autonomous (AID-05) and proactive (AID-02) flows in Plan 23-02.
</why_now>

    1. `redactSecrets(text)` — given input string containing lines like `POSTGRES_PASSWORD=hunter2`, `API_KEY=sk-abc`, `DB_TOKEN=xyz`, `MY_SECRET=foo`, replaces values after the `=` with `[REDACTED]` for keys matching `/(KEY|TOKEN|SECRET|PASSWORD|PASSWD|PWD|API_KEY|ACCESS_KEY|PRIVATE_KEY|CREDENTIAL)/i`. Non-matching keys (e.g. `LOG_LEVEL=debug`) are unchanged. Also redacts JSON-shaped values: `"password":"foo"` → `"password":"[REDACTED]"`. Returns the same string with redactions applied. Idempotent: applying twice produces identical output.

    2. `buildContainerDiagnosticPayload({logs, stats, inspectInfo})` — given trimmed inputs, returns a structured object: `{logsTrimmed: string, stats: {cpuPercent, memoryPercent, memoryUsageMb, memoryLimitMb, networkRxKb, networkTxKb, pids}, container: {state, restartCount, healthStatus, exitCode, image, imageDigest, startedAt}}`. `logsTrimmed` is the input logs after `redactSecrets()`. Memory bytes are converted to MB (round to 1 decimal). Network is converted to KB.

    3. `parseDiagnosticResponse(rawText)` — given a Kimi response that should follow the structured prompt template ("Likely cause: …\nSuggested action: …\nConfidence: …"), extracts the three sections. Robust to markdown headers (`## Likely cause`), uppercase variants, missing sections (returns `''` for missing). Returns `{likelyCause: string, suggestedAction: string, confidence: 'low'|'medium'|'high'|'unknown'}`.

    4. `parseComposeResponse(rawText)` — given a Kimi response that contains a YAML code block, extracts the contents of the first ` ```yaml … ``` ` (or ` ```yml … ``` `, or unfenced block prefixed by `version:` or `services:`). Returns `{yaml: string, warnings: string[]}` where `warnings` may include `"no fenced code block found"` if heuristic was used.

    Implementation:

    - `callKimi(systemPrompt: string, userPrompt: string, opts?: {tier?, maxTokens?}): Promise<{text, inputTokens, outputTokens}>` — fetches `${getNexusApiUrl()}/api/kimi/chat` with `X-API-Key` header (from `process.env.LIV_API_KEY`); 90s `AbortSignal.timeout`. Throws `[ai-unavailable]` on network error, `[ai-bad-response]` on non-JSON, `[ai-error]` on 5xx, `[ai-timeout]` on AbortError.

    - `diagnoseContainer(name: string, environmentId?: string|null): Promise<AiDiagnosticResult>` — fetches logs (200 lines), stats, inspect; builds payload; redacts; computes cache key `nexus:ai:diag:${containerId}:${sha256(payload.logsTrimmed.slice(-2000)).slice(0,16)}`; on hit returns cached `{...result, cached: true}`; on miss calls `callKimi()` with the diagnose system prompt below + `JSON.stringify(payload)` as user prompt, parses, caches with `EX 300`, returns.     - `generateComposeFromPrompt(prompt: string): Promise<AiComposeResult>` — calls `callKimi()` with the compose system prompt + `prompt` as user prompt; parses YAML out; runs `js-yaml.load()` defensively to detect malformed output (catch + add to `warnings`). NO Redis cache (each request is unique). `AiComposeResult = {yaml: string, warnings: string[], generatedAt: string}`.

    - `explainVulnerabilities(imageRef: string): Promise<AiCveExplanationResult>` — calls `getCachedScan(imageRef)`; if null throws `[no-scan-result]`; selects top 5 CVEs by `severity ∈ {CRITICAL, HIGH}` then by `cvss DESC`; computes cache key `nexus:ai:diag:cve:${sha256(top5_ids.join(',')).slice(0,16)}`; on miss calls `callKimi()` with the CVE system prompt + serialized `{imageRef, top5}` as user prompt; caches with `EX 300`. `AiCveExplanationResult = {explanation: string, upgradeSuggestion: string, model: string, generatedAt: string, cached: boolean}` — `parseCveResponse` extracts "Explanation:" and "Upgrade path:" sections similar to `parseDiagnosticResponse`.

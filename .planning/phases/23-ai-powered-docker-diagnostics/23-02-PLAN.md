<objective>
Ship Phase 23's two remaining requirements: AID-02 (proactive resource-pressure alerts via a new `ai-resource-watch` scheduler handler that runs every 5 minutes, finds memory- or CPU-stressed containers, asks Kimi for a projection, and persists alerts to a new `ai_alerts` PG table) and AID-05 (AI Chat sidebar autonomously diagnoses containers when asked, by registering a new `docker_diagnostics` tool in the existing nexus tool registry — the LLM router decides when to invoke it; we add zero regex intent-matching).

Purpose: Closes the two remaining AID requirements and completes Phase 23, making v27.0 ready for milestone audit. Reuses Plan 23-01's `callKimi()` bridge and `diagnoseContainer()` engine — no Kimi-bridge duplication. The proactive watch ships gated `enabled: false` because (a) it generates persistent Kimi spend even when nothing is wrong (one call per 5 minutes per stressed container) and (b) the user hasn't validated Kimi's projections in production yet. The `docker_diagnostics` tool ships enabled because it is pull-based — Kimi only invokes it when the user asks a relevant question.

Output: One new PG table, one new built-in scheduler handler, one alerts CRUD module, three tRPC routes, one new UI component (AlertsBell), one new AI tool registration, and unit tests covering the threshold + dedupe logic.
</objective>

<why_now>
Plan 23-01 ships the foundation (`callKimi`, `diagnoseContainer`). Plan 23-02 layers the autonomous and proactive surfaces on top. Splitting them is mandatory by the 2-3-task-per-plan rule — combining would mean 5+ tasks across 2 subsystems (scheduler + UI + nexus tool registration). Wave-2 placement reflects the import dependency on 23-01's exports — Plan 23-02 cannot start until `ai-diagnostics.ts` exists. Once both ship, milestone audit can proceed.
</why_now>

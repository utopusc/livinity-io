Implement the OAuth Device Authorization Grant flow for the agent's `setup` command, secure token storage, and token expiry checking with re-authentication.

Purpose: This is the authentication bridge between the agent binary and the livinity.io platform. The agent uses RFC 8628 device flow (POST /api/device/register, user approves at /device, agent polls /api/device/token) to obtain a JWT that grants relay access.

Output: Working `livinity-agent setup` command that authenticates the device, stores credentials, and a token expiry check wired into ConnectionManager's connect flow.

**agent/src/auth.ts:**

Create the OAuth device flow module with these exports:

`PLATFORM_URL` constant: `'https://livinity.io'` (hardcoded, not configurable for security -- agent must always auth against the real platform).

`deviceFlowSetup(deviceName: string)` async function — the full RFC 8628 device flow:
1. Determine platform: `process.platform` (cast to 'win32' | 'darwin' | 'linux')
2. POST to `${PLATFORM_URL}/api/device/register` with JSON body `{ deviceName, platform, agentVersion: AGENT_VERSION }`. Use native `fetch()` (Node 22+ has it built in). Set `Content-Type: application/json`.
3. If response is not ok, throw with response body error message.
4. Parse response JSON. Extract `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval`.
5. Display to user (clear formatting):
   ```

   Device Authentication
   =====================

   To authorize this device, visit:

     {verification_uri}

   And enter code:

     {user_code}

   Waiting for approval...
   ```
6. Start polling loop: every `interval` seconds (default 5), POST to `${PLATFORM_URL}/api/device/token` with JSON body `{ device_code }`.
   - If response is 400 with `error: 'authorization_pending'`: print a dot (`.`) to stdout (no newline, use `process.stdout.write('.')`) and continue polling.
   - If response is 400 with `error: 'expired_token'`: throw new Error('Device code expired. Please run setup again.')
   - If response is 400 with `error: 'invalid_grant'`: throw new Error('Invalid device code.')
   - If response is 200: parse body. Extract `access_token`, `relay_url`, `expires_in`.
7. On successful token receipt:
   - Decode the JWT payload (base64url decode the middle segment, JSON.parse) to extract `deviceId`. Do NOT use a JWT library -- just split on '.', base64url decode index 1, JSON.parse. The agent does not verify the token (the relay does that).
   - Call `writeCredentials({ deviceToken: access_token, deviceId: payload.deviceId, deviceName, relayUrl: relay_url, platform })` from state.ts.
   - Print:
     ```

     Device authorized successfully!

     Device ID:  {deviceId}
     Device:     {deviceName}
     Relay:      {relay_url}
     Token expires in 24 hours.

     Run `livinity-agent start` to connect.
     ```
8. Return the credentials object.

`isTokenExpired(token: string)` function:
- Decode JWT payload (same base64url decode as above).
- Read `exp` claim (epoch seconds).
- Return `true` if `Date.now() / 1000 >= exp - 300` (expired or within 5-minute buffer).
- Return `false` if token is still valid.
- On any decode error, return `true` (treat malformed tokens as expired).

`refreshOrReauth()` async function:
- Read credentials via `readCredentials()`.
- If no credentials, throw 'No credentials found. Run `livinity-agent setup` first.'
- Check `isTokenExpired(credentials.deviceToken)`.
- If expired: print "Device token has expired. Re-authentication required." then print "Run `livinity-agent setup` to re-authenticate." and throw 'Token expired'.
- If valid: return credentials unchanged.
NOTE: There is no refresh token endpoint in the current platform (Phase 47 only built register/token/approve). When the 24h token expires, the user must re-run `livinity-agent setup`. This is acceptable for v14.0.

**agent/src/cli.ts** — Replace the setup stub:

Update `setupCommand()` to:
1. Ask for device name: Use the system hostname as default. Get it via `import { hostname } from 'node:os'`. Print "Device name [{hostname()}]: " and read from stdin. If user presses Enter, use hostname. Use a simple readline approach:
   ```typescript
   import { createInterface } from 'node:readline';
   const rl = createInterface({ input: process.stdin, output: process.stdout });
   const answer = await new Promise<string>(resolve => rl.question(`Device name [${hostname()}]: `, resolve));
   rl.close();
   const deviceName = answer.trim() || hostname();
   ```
2. Check if credentials already exist via `readCredentials()`. If they do, print "This device is already authenticated as '{credentials.deviceName}'. Re-running setup will replace the existing credentials." and ask "Continue? [y/N]: ". If not 'y' or 'Y', exit.
3. Call `deviceFlowSetup(deviceName)` wrapped in try/catch. On error, print the error message in red-ish format: `console.error('Setup failed:', err.message)` and exit with code 1.

Also update `startCommand()` to add token expiry check before connecting:
1. After reading credentials (existing code), add: call `isTokenExpired(credentials.deviceToken)`. If expired, print "Device token has expired. Run `livinity-agent setup` to re-authenticate." and exit with code 1.

Update `ConnectionManager.connect()` to check token expiry before attempting WebSocket connection.

At the very start of `connect()`, before creating the WebSocket:
1. Import `isTokenExpired` from `./auth.js`.
2. Check `isTokenExpired(this.credentials.deviceToken)`.
3. If expired:
   - Log: "Device token has expired. Cannot reconnect. Run `livinity-agent setup` to re-authenticate."
   - Set `this.status = 'error'` (which prevents scheduleReconnect from retrying, since it checks `if (this.status === 'error') return`).
   - Call `writeState({ status: 'token_expired', connectedAt: null, relayUrl: this.credentials.relayUrl, deviceName: this.credentials.deviceName })`.
   - Return early (do not create WebSocket).

This ensures:
- If the token expires while the agent is running and reconnecting, it stops trying instead of hammering the relay with expired tokens.
- The relay would reject expired tokens anyway (JWT verify fails), but checking client-side avoids unnecessary network traffic and gives a clear error message.
- The status file shows 'token_expired' so `livinity-agent status` can report the issue.

Also update the `statusCommand()` in cli.ts to handle the 'token_expired' status:
- If `state.status === 'token_expired'`, print the status as "Token expired -- run `livinity-agent setup` to re-authenticate" in the status display.

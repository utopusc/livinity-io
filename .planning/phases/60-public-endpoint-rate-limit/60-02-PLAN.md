<objective>
Wave 1 — install caddy-ratelimit on Server5 by replacing `/usr/bin/caddy` with an xcaddy-built custom binary that adds the `mholt/caddy-ratelimit` module while preserving the existing `caddy-dns/cloudflare` module. This is the gate before any Caddyfile change in Wave 3 (`rate_limit` directive will fail config-validation otherwise — RESEARCH.md Pitfall 2).

Production-adjacent: `systemctl restart caddy` causes ~1-2s downtime (RESEARCH.md Code Example 4). `reload` is INSUFFICIENT — does not pick up new binary. Mitigations:
- Pre-flight: smoke-test current health (one known live subdomain) so we have a baseline.
- Backup current binary BEFORE swap.
- Diff `list-modules` before/after to prove no modules dropped (RESEARCH.md Open Q5).
- Smoke-test post-restart against the same baseline subdomain: NOT 502, NOT TLS error.
- Rollback runbook: restore `.bak` binary + restart (≤30s).

Per `60-DIAGNOSTIC-FIXTURE.md` Q5 verdict, this plan branches:
- Q5="yes on Server5" → run xcaddy on Server5 directly.
- Q5="no" → build on local dev box + scp the resulting `caddy-custom` binary.

Output: New Caddy binary live on Server5 with `rate_limit` module loaded; rollback target preserved; existing traffic unaffected.

Sacred file SHA reasserted at start AND end.
</objective>

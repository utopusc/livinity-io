<objective>
Wave 2 — extend the relay (the `@livinity/relay` Node process at Server5:4000) to recognize `Host: api.livinity.io` as a special-case routing target. This closes the load-bearing architectural correction RESEARCH.md surfaced: Caddy at Server5 reverse-proxies `api.livinity.io` → `localhost:4000` (the relay), and the relay's current `parseSubdomain` logic returns `{username:'api'}` → `registry.get('api')` → null → `serveOfflinePage` (broken HTML response). Without this code change, Wave 3 Caddyfile addition produces an HTTP 200 page that breaks all external SDK clients.

Implements FR-BROKER-B2-01 (`api.livinity.io` reachable from open internet via Server5).

Per RESEARCH.md Pattern 3: insert routing IMMEDIATELY after `parseSubdomain(req.headers.host)` (`server.ts:206`) and BEFORE the `if (!username && req.url === '/health')` line. Resolution path:
1. lowercase hostname == `api.livinity.io` → call `findAdminTunnel(registry, pool)`.
2. tunnel resolved + online → `proxyHttpRequest(adminTunnel, req, res, /* targetApp */ null)` (broker is at root path; livinityd's broker router will read `/u/<adminUserId>/v1/messages` based on URL path, OR — more accurately — the request URL hits `/v1/messages` and the Mini PC livinityd's app routing is responsible. **Per RESEARCH.md A5, livinityd's `/u/:userId/v1/messages` route validates `:userId` against the URL path. For Wave 2, the relay forwards the request URL VERBATIM. Wave 3 broker IP-guard removal handles the URL-path identity case via Phase 59 Bearer middleware which now sets req.userId. The URL path can therefore be `/v1/messages` directly without `/u/:adminUserId/` prefix; Phase 59 middleware mounted at `/u/:userId/v1` does NOT match an unprefixed `/v1/...`. Decision: relay rewrites the URL to inject `/u/<adminUserId>` prefix when forwarding api.livinity.io — Phase 59 middleware then runs and Bearer wins identity.**

   ⚠ This URL rewrite is the load-bearing detail. The plan task explicitly performs `req.url = '/u/' + adminTunnel.userId + req.url` BEFORE `proxyHttpRequest`.
3. tunnel resolved + offline → 503 JSON Anthropic-spec body. Do NOT call `serveOfflinePage` (HTML — would break SDK clients).
4. tunnel resolution failed (DB error / pool not present) → 503 JSON.

Per RESEARCH.md threat (Tunnel Hijack — Spoofing): admin resolution queries by `role = 'admin'`, NOT by username string match.

Per RESEARCH.md A4 risk: admin tunnel offline = 100% of `api.livinity.io` traffic 503s. This is acceptable for v30 (single-tenant; documented as multi-user-implication in 60-CONTEXT.md). Wave 3 Caddyfile addition does NOT mitigate (Caddy can't fail-over to a different upstream). Future v30+ revisits.

TDD this plan: vitest unit tests for `findAdminTunnel` BEFORE writing the implementation; tests cover the 4 cases above + the existing-route regression assertion.

```typescript
// platform/relay/src/admin-tunnel.ts

import type pg from 'pg'
import type http from 'node:http'
import type { TunnelRegistry, TunnelConnection } from './tunnel-registry.js'

/**
 * Resolve the admin user's currently-registered tunnel.
 * Phase 60 — used to route `api.livinity.io` traffic.
 *
 * Threat-model: queries by `role = 'admin'` (NOT username = 'admin') to defeat
 * spoofing — see RESEARCH.md "Tunnel hijack" threat.
 *
 * Returns the TunnelConnection if admin user exists AND has an open tunnel,
 * else null. Errors (DB failure / pool absent) → null (caller responds 503).
 */
export async function findAdminTunnel(
  registry: TunnelRegistry,
  pool: pg.Pool | null,
): Promise<TunnelConnection | null>

/**
 * Write a 503 JSON Anthropic-spec body for `api.livinity.io` when the admin
 * tunnel is offline. Use INSTEAD OF `serveOfflinePage` (which is HTML and
 * breaks SDK clients).
 */
export function sendBrokerTunnelOffline(res: http.ServerResponse): void
```

```typescript
// Insertion in platform/relay/src/server.ts (right after `parseSubdomain` at line 206):

const hostnameLower = req.headers.host?.split(':')[0]?.toLowerCase()
if (hostnameLower === 'api.livinity.io') {
  if (!pool) {
    sendBrokerTunnelOffline(res)
    return
  }
  const adminTunnel = await findAdminTunnel(registry, pool)
  if (!adminTunnel || adminTunnel.ws.readyState !== 1) {
    sendBrokerTunnelOffline(res)
    return
  }
  // Bandwidth quota — same path as existing per-user proxy at server.ts:281.
  const apiQuota = await checkQuota(redis, adminTunnel.userId)
  if (!apiQuota.allowed) {
    const now = new Date()
    const nextMonth = new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth() + 1, 1))
    const resetsAt = nextMonth.toISOString()
    res.writeHead(429, { 'Content-Type': 'application/json', 'Retry-After': '86400' })
    res.end(JSON.stringify({
      type: 'error',
      error: { type: 'rate_limit_error', message: 'Bandwidth quota exceeded' },
    }))
    return
  }
  // URL rewrite — inject /u/<adminUserId> prefix so Phase 59 Bearer middleware
  // (mounted at /u/:userId/v1 on Mini PC livinityd) catches the request.
  // Bearer middleware sets req.userId from the liv_sk_* token, then the legacy
  // resolveAndAuthorizeUserId validates the URL :userId equals admin (admin
  // tunnel owns this URL slot). Bearer wins identity at the broker handler.
  if (req.url && !req.url.startsWith('/u/')) {
    req.url = '/u/' + adminTunnel.userId + req.url
  }
  proxyHttpRequest(adminTunnel, req, res, null)
  return
}
```

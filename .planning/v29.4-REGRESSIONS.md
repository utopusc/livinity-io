## Per-Regression Root-Cause Analysis

### A1. Tool registry — built-in tools missing

**Evidence:**
- `nexus:cap:tool:*` = 0 keys in Redis on Mini PC
- Nexus chat (live, on Mini PC) cannot call `shell`, `docker_list`, etc.
- `livinity_list` returns 115 capabilities (skills/MCPs/etc.) but NO built-in tools

**Hypothesis (HIGH confidence):**
- Phase 22's `nexus/packages/core/src/capability-registry/syncTools()` flow is supposed to seed built-in tools to `nexus:cap:tool:*` on every livinityd boot. Either:
  - It's not running at boot (require/import side effect missing)
  - It's running but Redis is rejecting writes (auth, ACL, prefix collision)
  - The v29.2 factory reset wiped the registry and syncTools is non-idempotent (it only writes if `nexus:cap:meta:lastSyncAt` is absent — which would be missing post-reset, BUT then it should run; if it ran but failed silently, that's the actual bug)
  - syncTools depends on a runtime-discovered tool list (e.g., `toolRegistry.listAll()`) and that list is empty in production due to dynamic-import loading order

**Phase 47 implementation gap (KNOWN):**
- `47-02-SUMMARY.md` documented `D-WAVE5-SYNCALL-STUB`: "production Re-sync writes ZERO keys to nexus:cap:_pending:* because there's no real seed flow for built-in tools"
- This means the user clicking "Re-sync registry" from the Diagnostics card I shipped does NOTHING for built-in tools — it just runs an atomic-swap on an empty pending prefix.

**Recommended fix:**
- Path 2 (defensive eager seed): write a `seed-builtin-tools.ts` module that runs on livinityd startup unconditionally, idempotently writes the 9 BUILT_IN_TOOL_IDS manifests to `nexus:cap:tool:*` in Redis, and sets `nexus:cap:meta:lastSeedAt`. Survives factory resets, partial syncs, and registry corruption.
- Tests: integration test on isolated Redis DB 15 (mirror Phase 47-02 W-15 isolation guard), seed → verify all 9 keys exist → flush → re-seed → verify same keys exist (idempotent).

**Suggested file location:** `livos/packages/livinityd/source/modules/diagnostics/seed-builtin-tools.ts` (sibling of capabilities.ts; reuses `BUILT_IN_TOOL_IDS` already defined there).

### A2. Streaming regression — token-by-token replaced with batch-at-end

**Evidence:**
- User says "streaming tamamiyla gitmis artik tamamen butun islemi bitirdikten sonra gonderiyor"
- `nexus/packages/core/src/api.ts` was NOT changed in v29.4 (Phase 45 audit-only re-pin + Phase 47 Branch N — sacred file byte-identical at SHA `4f868d31...`)

**Hypotheses:**
1. **UI build not actually deployed** — 1m 2s deploy is suspiciously short. update.sh log will reveal if `pnpm --filter ui build` ran. If skipped, browser is serving the v43.x bundle (which may have its own streaming bug).
2. **PWA service worker stale cache** — even if vite built, the SW might serve old assets. Hard reload (`Ctrl+Shift+R`) will reveal.
3. **Phase 41-04 `homeOverride` wiring side-effect** — the `X-LivOS-User-Id` header pipeline runs synchronously at top of every `/api/agent/stream` request. If livinityd's per-user-claude.ts does an SSH-equivalent expensive call during HOME resolution, it could delay first-token-time enough to look like "no streaming".
4. **Sacred file v43.10 identity-line injection** — if the systemPrompt construction runs synchronously and inflates the prompt enough to push first-token-time past the user's perceived threshold, it WOULD look like streaming is broken.

**Diagnostic before fix (cheap → expensive):**
1. SSH: `cat /opt/livos/data/cache/install.sh.cached/update.sh.log 2>/dev/null | grep -E 'vite|build' | tail -20` — does vite build run?
2. SSH: `ls -la /opt/livos/livos/packages/ui/dist/index.html` — when was UI bundle built?
3. Browser: hard-reload Nexus chat → does streaming come back?
4. SSH: `journalctl -u livos -n 200 | grep -i homeoverride` — is per-user HOME resolution slow?
5. SSH + curl: send a streaming AI Chat request, time first byte vs full response.

If (1) reveals UI build skipped: fix `update.sh`. If (2) shows old timestamp + (3) hard-reload fixes it: PWA cache invalidation needed. If (4)+(5) show slow first-byte: livinityd-side latency, NOT streaming bug.

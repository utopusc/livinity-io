## 2. Cross-Phase Integration

### 2a. Phase 11 FK + userId Propagation → Phase 12 authorizeDeviceAccess

Integration chain: **relay assigns `userId: tokenPayload.userId` in `TunnelDeviceConnected`** (relay/src/index.ts:429) → **LivOS `onDeviceConnected` stores it in Redis** (device-bridge.ts:205: `JSON.stringify({userId, deviceId, ...})`) → **`authorizeDeviceAccess` reads `livos:devices:{deviceId}` and compares `entry.userId === userId`** (authorize.ts:37).

The chain is complete. A `device_connected` event without `userId` causes a hard reject in `onDeviceConnected` with a logged error and early return — it never enters the Redis cache. `authorizeDeviceAccess` will therefore return `device_not_found` for stale or tampered device IDs.

**Verdict: WIRED.**

### 2b. Phase 12 authorizeDeviceAccess → Phase 13 Shell Isolation

Phase 13 adds three complementary defenses but explicitly relies on Phase 12 as the runtime enforcer:
- Local shell parameter list is frozen (no `device_id` escape hatch).
- Device proxy shell description says "Do not pass a device_id parameter — the device is implicit in the tool name."
- `RESERVED_TOOL_NAMES` blocks any attempt to register a `shell` override via `/api/tools/register`.

But the actual runtime rejection of a cross-user `device_<id>_shell` call flows through `executeOnDevice` → `authorizeDeviceAccess`, which is exactly the Phase 12 path. Confirmed: `authorizeDeviceAccess` import at device-bridge.ts:12 and call at line 322 are unchanged by Phase 13.

**Verdict: WIRED. Phase 13 is an additive hardening layer; Phase 12 is the enforcer.**

### 2c. Phase 12 recordAuthFailure stub → Phase 15 PG audit writer

The migration path: Phase 12 created `audit-stub.ts` with `recordAuthFailure` writing to Redis list `nexus:device:audit:failures`. Phase 15 created `audit-pg.ts` with `recordDeviceEvent` inserting to PostgreSQL, and wired all three callsites to use the PG writer. Confirmed:
- `device-bridge.ts` imports `recordDeviceEvent from './audit-pg.js'` (line 13) and has zero `recordAuthFailure` references.
- `routes.ts` imports `recordDeviceEvent` from barrel (`./index.js`, line 5) and has zero `recordAuthFailure` references.
- `audit-pg.ts` itself imports `recordAuthFailure` from `./audit-stub.js` as a PG-failure fallback — this is intentional and documented.

**Verdict: MIGRATION COMPLETE. No dangling stub callsites remain.**

### 2d. Phase 14 WS 4403 close code → Phase 16 admin force-disconnect

Phase 14-02 established 4403 for `session_revoked`. Phase 16 reuses 4403 for `admin_disconnect`, distinguishing them only by the close reason string. The relay closes session-revoked bridges at `session-revocation.ts:ws.close(4403, 'session_revoked')` and admin-disconnected bridges at `index.ts:318: target.ws.close(4403, 'admin_disconnect')`. Client agents can inspect the reason string to distinguish the cause. The code partitioning is:

| Code | Reason string | Triggered by |
|------|---------------|--------------|
| 1008 | session_invalid / session_expired / etc. | Handshake validation failure |
| 4401 | token_expired | Watchdog (token lifetime exceeded) |
| 4403 | session_revoked | Logout pub/sub (user-initiated) |
| 4403 | admin_disconnect | Admin force-disconnect |

**Verdict: WIRED. Close codes are consistent and correctly partitioned.**

## 3. End-to-End Attack Scenario Verification

**Scenario:** User B (member role) tries to execute a shell command on User A's device.

**Attack vector 1: Direct device proxy tool call via agent loop**

1. User B's agent invokes `device_<A_device_id>_shell` with command `rm -rf /`.
2. Nexus tool execute posts to `callbackUrl` which embeds `expectedUserId=<B.id>` (captured at User B's device registration time — but User B has no devices, or B's ID differs from A's device owner).
3. `livinityd /internal/device-tool-execute` reads `req.query.expectedUserId = B.id`.
4. `executeOnDevice(tool, params, B.id)` calls `authorizeDeviceAccess(redis, B.id, A_device_id)`.
5. Redis cache entry `livos:devices:{A_device_id}` has `userId = A.id`. `A.id !== B.id` → `{authorized: false, reason: 'device_not_owned'}`.
6. `recordDeviceEvent` fires with `success=false, error='device_not_owned'`.
7. HTTP response: `403 {success: false, code: 'device_not_owned'}`.
8. **No `sendTunnelMessage(device_tool_call)` is ever called.** Confirmed: `type: 'device_tool_call'` appears exactly once in the codebase (device-bridge.ts:350), 28 lines below the auth gate at line 322.

**Attack vector 2: Attempt to register a 'shell' tool override via /api/tools/register**

1. User B's malicious agent posts `POST /api/tools/register {name: 'shell', ...}` to Nexus.
2. `RESERVED_TOOL_NAMES.has('shell')` is true.
3. Handler returns `HTTP 409 {error: 'tool_name_reserved'}` before `toolRegistry.register` is called.
4. Local shell tool behavior unchanged. No local shell tool modification possible.

**Attack vector 3: Device bridge connection with mismatched userId in JWT**

1. User B crafts a device JWT (or intercepts a stale one) with `userId = A.id` but for a device that actually belongs to A.
2. Relay `onDeviceConnect` handshake queries `SELECT user_id FROM devices WHERE device_id = $1`.
3. `devices.user_id = A.id`, JWT `userId = A.id` — these match. But B cannot sign a valid JWT because the relay verifies the JWT signature against the shared `DEVICE_JWT_SECRET`.
4. If B somehow obtained A's device JWT: relay queries `SELECT id, user_id, expires_at FROM sessions WHERE id = $1` against B's session. B's sessions.id is different from the sessionId embedded in A's JWT. The session lookup returns 0 rows → `1008 session_invalid`.
5. If B creates a new session: their `user_id` in `sessions` would be B's ID, which mismatches `devices.user_id = A.id` → `1008 device_ownership_mismatch`.

**Attack vector 4: User B calls devices.list to enumerate other users' devices**

1. `devices.list` calls `bridge.getDevicesForUser(ctx.currentUser.id)` where `ctx.currentUser.id = B.id`.
2. `getDevicesForUser` filters Redis entries where `d.userId === B.id` — returns only B's devices.
3. `GET /api/devices` queries `WHERE user_id = $1` with B's session userId.
4. Neither path returns A's devices.

**Verdict: All four attack vectors are blocked at multiple independent layers. No bypass path identified.**

## 4. Anti-Patterns and Technical Debt

### Blockers
None.

### Warnings

**TD-01: `platform/web` REDIS_URL deployment gap**

`platform/web/src/lib/session-revocation.ts` uses `process.env.REDIS_URL ?? 'redis://127.0.0.1:6379'`. On production Server4 (`45.137.194.103`), `platform/web` is a Next.js process managed by PM2. If `REDIS_URL` is not set in platform/web's PM2 ecosystem config or environment, the publisher silently connects to `localhost:6379` — which will fail in a containerized or multi-host deployment, silently swallowing the failure (soft-fail pattern). SESS-03 (logout bridge teardown) would degrade to the 60s watchdog fallback rather than instant teardown.

14-02-SUMMARY explicitly documents this ("Rollout risk" section) and notes the platform/web PM2 process needs `REDIS_URL=redis://:LivRedis2024%21@127.0.0.1:6379`. This is a deployment configuration gap, not a code bug, but it needs operator action before v26.0 is live.

**Action required at deploy time:** Add `REDIS_URL` to `platform/web`'s PM2 process environment (or `.env.local`) on Server4.

### Informational

**TD-02:** Stale comment at `server/index.ts:984` says "already called recordAuthFailure" — should reference `recordDeviceEvent`. No functional impact.

**TD-03:** REQUIREMENTS.md AUTHZ-03 says "Nexus REST /api/devices/\* endpoints." No such endpoints exist. The defense-in-depth is at `/internal/device-tool-execute`. The behavior fully satisfies the intent; the requirement text is imprecise. Both 12-02-SUMMARY and 12-VERIFICATION.md document this reconciliation explicitly.

**TD-04:** 365 pre-existing TypeScript errors in `livos/packages/livinityd` (skills tier enum mismatches, `ai/routes.ts` `ctx.livinityd` possibly-undefined, `tunnel-client.ts` `sendMessage` generics). All traced back to pre-v26.0 baseline in 11-02-SUMMARY. Zero new errors introduced by v26.0 work.

## 8. Overall Verdict

**v26.0 milestone goal is ACHIEVED.**

The invariant that makes the milestone goal true — "a user's terminal/shell session cannot reach another user's connected devices" — is enforced at 5 independent layers:

1. **Protocol layer (Phase 11):** Every device in PostgreSQL has a non-null `user_id` FK. No orphan devices possible.
2. **Cache layer (Phase 11):** Redis entries include `userId`; missing `userId` causes hard-reject at `onDeviceConnected`.
3. **Authorization layer (Phase 12):** `authorizeDeviceAccess` is the single source of truth, called before any tunnel message is sent. One `device_tool_call` occurrence in the codebase, 28 lines after the gate.
4. **Schema/description layer (Phase 13):** Local shell has no `device_id` parameter; `RESERVED_TOOL_NAMES` prevents shell tool clobbering.
5. **Session layer (Phase 14):** Device bridge requires a valid user session at handshake; expired sessions terminate the bridge; logout triggers instant teardown.

Audit coverage (Phase 15) and admin incident response (Phase 16) complete the defense-in-depth picture.

One deployment action is required before v26.0 is fully operational: **set `REDIS_URL` on the `platform/web` PM2 process on Server4** so that SESS-03 (logout bridge teardown) works instantly rather than falling back to the 60s watchdog.

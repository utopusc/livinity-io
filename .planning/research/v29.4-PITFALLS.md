# v29.4 Pitfalls Research — Server Management Tooling + Bug Sweep

**Domain:** Fail2ban admin panel + SSH gateway (cloudflared) + Nexus tool registry restore + 4 surgical v29.3 carry-forwards
**Researched:** 2026-05-01
**Confidence:** HIGH (each pitfall verified against the actual LivOS code paths it touches)

## Executive Summary

v29.4 is a **bug-sweep + surgical-additions milestone** layered on top of an already-running multi-user home-server stack. Its pitfall surface is unusual: instead of "what could go wrong building X from scratch," the question is "what already-running invariants could v29.4 silently break?"

Five existing invariants matter most:

1. **Sacred file integrity contract (`sdk-agent-runner.ts`)** — Phase 40 D-40-01 ritual must be re-applied for ANY new edit. v29.3 already drifted from baseline `623a65b9...` to `4f868d31...` without re-pinning; v29.4 piles A2 (model-identity systemPrompt edit) on top — that's TWO layers of audit debt unless C2 + A2 sequence the SHA re-pin correctly.
2. **D-NO-SERVER4** — v29.4 must touch ONLY the Mini PC. Server4 is hard-walled.
3. **`httpOnlyPaths` invariant** — any new tRPC mutation/query that uses long-running ops MUST be in `httpOnlyPaths` in `common.ts` or it hangs on WS reconnect.
4. **livinityd-runs-as-root attack surface** — every new `execFile` from livinityd is a root-privilege primitive. `fail2ban-client unbanip` is benign; `fail2ban-client set sshd banip` with un-validated input is not.
5. **`update.sh` pnpm-store dist-copy ambiguity** — A2 root cause may be DEPLOYMENT bug (stale dist), not source bug. Diagnostic-before-fix is mandatory.

The catastrophic pitfall classes — by phase — are:

- **Fail2ban admin:** self-ban (unban-via-UI re-bans the admin's own session), banning the admin's current IP without confirmation, hardcoding `sshd` jail, distinguishing "missing binary" from "service stopped" from "no jails configured."
- **SSH gateway (cloudflared B3b):** tunnel-name collision with the existing `pc.{user}.livinity.io` cloudflared tunnel, no copy-paste-able `ProxyCommand` surfaced to user, plaintext-at-edge threat-model leak undocumented.
- **Tool registry restore (A1):** flush+resync race window, blowing away user-set `enabled: false` overrides, false-positive diagnostics that conflate "missing because Redis lost it" with "missing because precondition not met."
- **Model identity (A2):** fixing the wrong root cause (source vs deployment), sacred-file-edit ritual skipped, SHA re-pin not coordinated with C2.
- **C1-C4 carry-forwards:** 429 forwarding allowlist too loose, `Retry-After` normalization, sacred-file content drift hidden inside an "audit-only" re-pin commit, OpenAI SSE chunk ordering.
- **Cross-cutting:** mobile UX (cellular vs WiFi IP), test-environment vs production reality, "44+ commits ahead of origin" v29.3 lesson, backout strategy for every new module.

## Critical Pitfalls (BLOCKER class — undetected leads to data loss / security bug / sacred-file corruption / lockout)

### B-01 — Self-Ban: Unban-via-UI Re-Bans Admin's Active SSH/HTTPS Session
**Phase:** Fail2ban admin (Phase 47)
**Severity:** BLOCKER
**Warning sign:** Admin clicks "Unban 1.2.3.4" in UI → unban succeeds → admin's NEXT request to livinityd fails with connection-refused → admin is back on the banned list. Repro: hit unban twice in 10s, watch fail2ban logs for `Ban 1.2.3.4` appearing AFTER the unban call. (Real bug pattern in webmin / cockpit fail2ban modules circa 2019-2022.)
**Why it happens:** fail2ban tails its log source (default journal `_SYSTEMD_UNIT=ssh.service`) and re-bans on subsequent failed connection attempts. The unban-via-UI request itself is HTTP — it doesn't touch sshd — but if the admin is ALSO trying to keep an SSH session alive with the wrong key, fail2ban re-classifies the next failed auth attempt within `findtime` window.
**Prevention:**
- The unban call site MUST also clear the failregex match-counter via `fail2ban-client set sshd action <action> unban <ip>` for sshd-iptables-multiport (the explicit ban-action targeted unban, not just the global unban).
- Surface in UI: "After unban, fail2ban may re-ban this IP if connection attempts continue with bad credentials. Verify your SSH key is correct first."
- Log the unban event to `/opt/livos/data/security-events/<ts>-unban.json` with `{actor: userId, target: ip, jail, sourcePort: req.socket.remotePort}` so audit trail captures the immediate-re-ban pattern if it recurs.
**Detection:** Integration test — call `unbanIp("sshd", "5.6.7.8")`, then within 1s assert `getJailStatus("sshd").bannedIps` does NOT contain `5.6.7.8`. Re-poll at +5s.

### B-02 — Banning Admin's Own Current IP Without "ARE YOU SURE THIS IS YOU?" Confirmation
**Phase:** Fail2ban admin (Phase 47)
**Severity:** BLOCKER
**Warning sign:** Admin types `1.2.3.4` into "Ban IP" field → clicks Confirm → loses access to the box → has to physically reboot the Mini PC to recover. Or worse: SSH-locks themselves out from the office while traveling.
**Why it happens:** Standard "are you sure?" modal isn't enough — the user can confirm without realizing the IP they typed equals their current connection's source IP. (Especially likely on shared NAT — admin sees `5.6.7.8` in fail2ban's banned list, types it into "Ban additional IP" thinking it's a different attacker.)
**Prevention:**
- Server-side check: the ban tRPC route must extract the request's source IP (the SSH-bound public IP — see B-08 below for the X-Forwarded-For pitfall) and compare against the ban target.
- If equal: return `{warning: "self_ban", currentSourceIp, banTargetIp}` — UI must show DIFFERENT modal: "WARNING: 1.2.3.4 is YOUR CURRENT CONNECTION IP. Banning will lock you out. Type `LOCK ME OUT` to confirm."
- Mirror the v29.2 factory-reset confirmation pattern: type-`LOCK ME OUT` strict equality gate, NOT just an OK button.
**Detection:** Negative-path test — POST to ban endpoint with `ip = req.connection.remoteAddress`, assert HTTP 409 + `{warning: "self_ban"}` payload. Verify UI surfaces the second modal.

### B-03 — Unban-by-CIDR Accidentally Unbans `0.0.0.0/0`
**Phase:** Fail2ban admin (Phase 47)
**Severity:** BLOCKER
**Warning sign:** Admin pastes a CIDR like `0.0.0.0/0` (or any /0) into the unban field — fail2ban accepts it — every banned IP is unbanned at once. Or pastes ` 1.2.3.4 ` with leading/trailing whitespace and parser splits incorrectly.
**Why it happens:** `fail2ban-client set sshd unbanip` accepts both single IPs and CIDR. The UI's input validation must reject /0 explicitly (and /8 for safety).
**Prevention:**
- Backend Zod schema: `z.string().ip()` (single IP only — reject CIDR). If CIDR is needed in future, add `z.string().regex(/^\d+\.\d+\.\d+\.\d+\/(?:[8-9]|[12][0-9]|3[0-2])$/)` (rejects /0 through /7).
- Frontend trim() before submit + match against the same regex.
- Server-side defense-in-depth: even if CIDR is allowed, count `bannedIps` before and after the unban — if delta > 10, log a warning and require admin re-confirmation.
**Detection:** Unit test — `parseUnbanInput("0.0.0.0/0")` throws ValidationError. `parseUnbanInput(" 1.2.3.4 ")` returns `"1.2.3.4"` (trimmed). `parseUnbanInput("1.2.3.4/0")` throws.

### B-04 — Sacred File SHA Coordination Failure (C2 + A2)
**Phase:** A2 model-identity fix + C2 SHA re-pin (Phase 46 + 49)
**Severity:** BLOCKER
**Warning sign:** Either (a) C2 re-pins to current `4f868d31...` claiming it's "audit-only" but accidentally also includes the A2 systemPrompt edit in the same commit (silent code change behind an audit-only commit message), OR (b) A2 lands first, captures NEW SHA, then C2 re-pins to the post-A2 SHA — losing the audit trail of WHY the SHA moved from baseline `623a65b9...`.
**Why it happens:** The sacred file (`nexus/packages/core/src/sdk-agent-runner.ts`) has the D-40-01 ritual that requires per-edit pre/post SHA capture + audit comment. v29.3 violated this with v43.x model bumps (drift from `623a65b9...` to `4f868d31...` un-audited). v29.4 must un-do the sin AND add a new edit on top.
**Prevention:**
- Sequence: **C2 lands FIRST as audit-only commit.** The commit message MUST say "no source change — re-pin BASELINE_SHA from `623a65b9...` to `4f868d31...` documenting v43.x model-bump drift (commits abc123, def456, ...)." `git diff --shortstat nexus/packages/core/src/sdk-agent-runner.ts` MUST be empty.
- Then **A2 lands as separate commit.** Pre-edit SHA capture → behavior-preserving systemPrompt edit → post-edit SHA capture → integrity test re-pinned with NEW BASELINE_SHA + audit comment quoting the diff.
- Verification gate: CI / pre-merge hook that fails if any commit touching `sdk-agent-runner.ts` does NOT also touch `sdk-agent-runner-integrity.test.ts:33` (BASELINE_SHA constant). Force-couples the two.
**Detection:** Post-commit grep — `git log --oneline nexus/packages/core/src/sdk-agent-runner.ts` should show TWO commits in v29.4 (C2 audit-only, A2 systemPrompt) with paired test re-pins. Integrity test runs green.

### B-05 — A2 Wrong-Root-Cause Fix (Source Edit When Bug Is in Pnpm-Store Dist)
**Phase:** A2 model-identity fix (Phase 46)
**Severity:** BLOCKER
**Warning sign:** Phase 46 ships a sacred-file edit; user runs `update.sh`; user reports "still says I'm Claude 3.5 Sonnet." Looking at source confirms edit is correct. Looking at running container's `node_modules/@nexus/core/dist/index.js` shows OLD systemPrompt — pnpm-store has multiple `@nexus+core*` dirs and `update.sh:populate_dist` copied to the wrong one (memory-blocker BACKLOG 999.5b).
**Why it happens:** `update.sh` runs `find /opt/livos/node_modules/.pnpm -maxdepth 1 -type d -name '@nexus+core*'` and copies into the FIRST match. If sharp version drift creates two resolution dirs, livinityd's actual import path resolves to the OTHER dir.
**Prevention:**
- **Phase 46 MUST start with a diagnostic step BEFORE any source change:**
  1. SSH to Mini PC.
  2. `curl -s http://localhost:8080/u/<userId>/v1/messages` (or whatever broker hits) with a deterministic identity prompt.
  3. Inspect response — if `model: "claude-3-5-sonnet"` literal in API response → broker config issue, not systemPrompt issue.
  4. `ls /opt/livos/node_modules/.pnpm/@nexus+core*` — if multiple dirs, surface as a deployment bug, NOT a source bug.
  5. `readlink -f /opt/livos/node_modules/@nexus/core` to find the ACTUAL resolved path.
  6. `grep -l 'identity_marker_string' /opt/livos/node_modules/.pnpm/@nexus+core*/node_modules/@nexus/core/dist/index.js` — if marker missing in resolved dir, it's a dist-copy bug.
- ONLY if all 6 diagnostics confirm source-level confabulation does Phase 46 proceed with the sacred-file systemPrompt edit.
- If diagnostics point at deployment bug, fix `update.sh:populate_dist` to copy into ALL matched dirs (mirror Phase 31 BUILD-02 multi-dir loop) — NOT a sacred-file edit.
**Detection:** Phase 46 plan MUST embed the 6-step diagnostic as Plan 01, with explicit "if condition X, route to Plan 02a (source edit); if condition Y, route to Plan 02b (update.sh fix)." Two distinct fix plans; pick exactly one.

### B-06 — Tool Registry Flush+Resync Race Window
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** BLOCKER
**Warning sign:** Admin clicks "Re-sync Registry" → for ~500ms, AI Chat returns "no tools available" → user sends a message during the window → agent runs without `shell` / `docker_*` / `files` → silently fails OR worse, user gets a "tool not found" cascade through `discover_capability` self-modification flow that creates a phantom skill.
**Why it happens:** `redis.del("capability:*")` followed by `syncFromAllSources()` is non-atomic. Any tool invocation arriving between the two calls sees an empty registry.
**Prevention:**
- Wrap in Redis MULTI: `redis.multi().del("capability:*").exec()` THEN sync — but resync is async (4 data sources, network IO). Better:
- Build new manifests in a TEMPORARY Redis prefix (`capability:_pending:*`), then atomic-rename with `RENAME` per key OR `EVAL` Lua script that swaps prefix atomically.
- During swap, return cached snapshot from in-memory `IntentRouter` cache (Phase 22 already has 30% context budget cache). If snapshot age > 60s, queue the request and re-fire after sync completes.
- Surface a UI "Re-syncing… (X/Y tools)" overlay on the admin button — disable other capability operations during the sync window.
**Detection:** Concurrency test — fire 50 parallel `discover_capability("shell")` calls during a `flushAndResync`. Assert ZERO return `{found: false}` (all either return the tool or queue-and-retry).

### B-07 — Tool Registry Resync Reverts User-Set `enabled: false`
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** BLOCKER
**Warning sign:** Admin had previously disabled `shell` via Phase 22 Settings > Capabilities > Disable (security hardening). Admin clicks "Re-sync Registry" to restore missing built-ins. Re-sync re-enables `shell`. AI Chat starts running shell commands again. User reports "I disabled shell three weeks ago, it's back."
**Why it happens:** `flushAndResync` deletes the entire `capability:*` keyspace including the user-override merge layer. Phase 22 stored user disables in PG (`user_capability_overrides` table). The resync writes from manifest sources (4 data sources) WITHOUT consulting the override table.
**Prevention:**
- Resync must read user overrides FIRST: `SELECT capability_id, enabled FROM user_capability_overrides WHERE user_id = $1 AND enabled = false`
- After populating Redis from manifest sources, re-apply overrides: `for each row: redis.hset("capability:" + id, "enabled", "false")`
- Audit log every resync: `{actor, timestamp, capabilityCountBefore, capabilityCountAfter, overridesPreserved: [...ids]}` written to `/opt/livos/data/security-events/<ts>-registry-resync.json`.
- Idempotency: running resync twice in a row produces identical Redis state (overrides survive both runs).
**Detection:** Integration test — disable `shell` via override, run flushAndResync, assert `getCapability("shell").enabled === false`.

### B-08 — Cloudflared Tunnel Collision with Existing pc.{user} Tunnel
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** BLOCKER
**Warning sign:** Admin enables SSH gateway → cloudflared restarts → existing v18.0 `pc.{user}.livinity.io` desktop streaming tunnel goes down → admin loses BOTH SSH gateway AND noVNC desktop stream simultaneously. Or: cloudflared config `--config` clobbers the existing tunnel-credentials path; admin's CF account auth state lost; both tunnels die.
**Why it happens:** v18.0 already deployed cloudflared with one tunnel for `pc.{user}.livinity.io`. cloudflared supports multiple ingress rules in a SINGLE config OR multiple `cloudflared@<unit>.service` systemd units. The naive `setup_ssh_gateway()` rewrites `/etc/cloudflared/config.yml` losing the pc.* ingress.
**Prevention:**
- Use SEPARATE systemd units: `cloudflared@desktop-stream.service` (existing) + `cloudflared@ssh-gateway.service` (new). Each owns its own `/etc/cloudflared/<unit>-config.yml` + tunnel credentials JSON.
- `install.sh:setup_ssh_gateway()` MUST `[ -f /etc/cloudflared/ssh-tunnel.json ]` idempotency guard at top.
- Pre-flight check before `cloudflared tunnel create`: `cloudflared tunnel list` and verify no existing `ssh-{hostname}` named tunnel — if found, reuse it; never create duplicate.
- Backup the existing pc.* config before any cloudflared write: `cp /etc/cloudflared/config.yml{,.pre-ssh-gateway-bak}`.
- Rollback path: `uninstall_ssh_gateway()` mirror function that `systemctl disable --now cloudflared@ssh-gateway`, deletes its config + credentials, leaves desktop-stream tunnel untouched.
**Detection:** Integration test on Mini PC scratchpad — install SSH gateway, verify `systemctl is-active cloudflared@desktop-stream` still returns `active`, verify `pc.{user}.livinity.io` still resolves end-to-end.

### B-09 — C1 Status Code Allowlist Too Permissive (502/504 Forwarded as 429)
**Phase:** C1 broker 429 forwarding (Phase 49)
**Severity:** BLOCKER
**Warning sign:** Anthropic's API has an outage (returns 502 Bad Gateway). Broker forwards as 429. UI shows "subscription cap reached" banner. User panics, thinks they're rate-limited, clicks "upgrade" — but actually upstream is just down. Trust damage.
**Why it happens:** Naive implementation of C1: `if (upstreamRes.status >= 400) forward as 429`. 502 and 504 are infrastructure errors, not rate-limit. They have different UX semantics (retry-with-backoff, not "you've hit your cap").
**Prevention:**
- Strict allowlist: ONLY forward status 429 verbatim. Status 502/503/504 → forward as their own status code.
- Forward Anthropic's `error.type` field — `rate_limit_error` → display cap banner; `api_error` / `overloaded_error` → display "upstream issue, try again" banner.
- Unit test for each upstream status code: `[400, 401, 403, 429, 500, 502, 503, 504, 529]` — assert broker returns matching status with correct UX category.
- The Express response capture middleware that writes `broker_usage` rows MUST not write a usage row for non-200 responses (no tokens consumed; otherwise the dashboard chart shows phantom usage during outages).
**Detection:** Negative-path tests — mock upstream returning 502, assert broker returns 502 (not 429), assert no `broker_usage` row inserted, assert UI banner says "upstream error" not "rate limited."

### B-10 — C1 Retry-After Header Format Normalization Loss
**Phase:** C1 broker 429 forwarding (Phase 49)
**Severity:** BLOCKER
**Warning sign:** Anthropic returns `Retry-After: Wed, 21 Oct 2026 07:28:00 GMT` (HTTP-date format). Broker normalizes to integer seconds (`60`). UI shows "retry in 60s" but actually upstream wants 6 hours. Or vice versa: Anthropic returns `Retry-After: 60`; broker normalizes to ISO timestamp; UI parses as Unix epoch and shows "retry in 50 years."
**Why it happens:** RFC 7231 allows BOTH `delta-seconds` (integer) and `HTTP-date` formats for `Retry-After`. Anthropic's docs are silent on which they emit; production may switch unannounced.
**Prevention:**
- Forward `Retry-After` header VERBATIM — do NOT parse, do NOT normalize, do NOT re-emit.
- UI parser handles both formats: `if (/^\d+$/.test(header)) seconds = parseInt(header); else seconds = Math.max(0, (new Date(header).getTime() - Date.now()) / 1000);`
- Test fixtures with BOTH formats: `Retry-After: 60` AND `Retry-After: Wed, 21 Oct 2026 07:28:00 GMT`. Assert both produce correct countdown.
**Detection:** Unit test — parse fixture pair, assert UI countdown component renders correctly for both. Integration test against mock 429 response with each header variant.

### B-11 — C2 Sacred File Re-Pin Accidentally Includes Source Change
**Phase:** C2 sacred SHA re-pin (Phase 49)
**Severity:** BLOCKER
**Warning sign:** Commit message says "audit-only re-pin BASELINE_SHA" but `git show <commit> -- nexus/packages/core/src/sdk-agent-runner.ts` shows non-empty diff. Trojan-horse a behavior change inside an audit-only commit — extremely hard to spot in review.
**Why it happens:** Developer rebases / cherry-picks during the re-pin; whitespace fixup or unrelated edit slips in. Or: re-pin captures CURRENT SHA (`4f868d31...`) but the working tree was inadvertently modified between SHA capture and commit.
**Prevention:**
- The C2 commit MUST satisfy: `git diff --shortstat HEAD~1 -- nexus/packages/core/src/sdk-agent-runner.ts` returns empty (zero insertions, zero deletions).
- Pre-commit hook (or CI gate) that fires when `sdk-agent-runner-integrity.test.ts:33` BASELINE_SHA changes: validates `git show HEAD -- nexus/packages/core/src/sdk-agent-runner.ts` is empty.
- Re-pin commit message template: explicit "Source content unchanged. SHA moved due to commits [list]. No new behavior."
**Detection:** Post-commit gate: `git show <c2-commit> --stat | grep sdk-agent-runner.ts` should ONLY show the test file, NOT the source file.

### B-12 — C3 httpOnlyPaths Routes Hang Under WebSocket Reconnect
**Phase:** C3 httpOnlyPaths additions (Phase 49)
**Severity:** BLOCKER
**Warning sign:** Admin opens Settings, clicks "Connect Claude" (`claudePerUserStartLogin` mutation) → mutation hangs forever → admin closes tab in frustration. Repro: kill livinityd while UI is open, restart livinityd, try the mutation immediately upon reconnect. The WebSocket reconnect window swallows the mutation; HTTP fallback wasn't enabled because the route wasn't in `httpOnlyPaths`.
**Why it happens:** tRPC has both WS and HTTP transports. Mutations on routes NOT in `httpOnlyPaths` route through WS by default. During WS reconnect (~5s window), mutations queue and may drop. C3 fix adds 3 routes to the allowlist, but if even one is missed, the user-visible bug remains.
**Prevention:**
- Adding to `httpOnlyPaths` in `common.ts` is trivial — but verify each route also handles WS-active scenario correctly. There's a brief window where the client tries BOTH transports.
- Manual test pattern: kill livinityd via `systemctl restart livos` while Settings tab is open, wait for reconnect indicator, immediately fire each route. Assert response within 2s.
- Automated: kill+restart livinityd in test env, assert `claudePerUserStartLogin` returns 200 within 2s of reconnect.
- Add ALL three routes (`claudePerUserStartLogin` sub, `usage.getMine` query, `usage.getAll` query) in a SINGLE commit so partial fix is impossible.
**Detection:** Integration test — restart-livinityd-mid-session test for each of the 3 routes individually + together.

### B-13 — C4 OpenAI SSE Chunk Order Violates Spec
**Phase:** C4 OpenAI SSE usage chunk (Phase 49)
**Severity:** BLOCKER
**Warning sign:** OpenAI Python SDK throws `pydantic.ValidationError: usage field on chunk with finish_reason expected before [DONE]`. Or worse: silent — SDK swallows the malformed chunk, `broker_usage` row gets inserted with token=0, dashboard shows "OpenAI streaming uses zero tokens" forever.
**Why it happens:** OpenAI streaming spec requires the FINAL chunk (with `usage` field) to arrive BEFORE `data: [DONE]\n\n`. Naive implementations emit `[DONE]` first, then try to append usage afterward — violates spec.
**Prevention:**
- Exact wire format (verified against OpenAI API docs):
  ```
  data: {"id":"chatcmpl-...","object":"chat.completion.chunk","created":...,"model":"...","choices":[{"index":0,"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":N,"completion_tokens":M,"total_tokens":K}}\n\n
  data: [DONE]\n\n
  ```
- The usage chunk MUST have `choices` array (empty delta + `finish_reason: "stop"`) — older OpenAI clients reject chunks without `choices`.
- The capture middleware extracts `usage` from this final chunk (NOT from the `[DONE]` marker).
- Integration test runs the verbatim openai Python SDK smoke test from Phase 42's UAT — assert `response.usage.total_tokens > 0` for streaming traffic.
**Detection:** Run `pnpm --filter livinity-broker test -- openai-sse` + the Python SDK smoke test; assert dashboard `usage.getMine` returns nonzero rows for OpenAI streaming.

## Moderate Pitfalls (WARNING class)

### W-01 — Fail2ban Async Unban: UI Optimistic Update Without Refresh Confirmation
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** UI shows "Unbanned 1.2.3.4" toast, but the banned-IPs table still includes 1.2.3.4 for ~1s afterward. Admin clicks unban a second time, unban-twice creates a confused log trail.
**Why it happens:** `fail2ban-client set sshd unbanip` returns immediately; the in-process ban list update is async (~500ms - 1s). The next `status sshd` call may still return the IP.
**Prevention:**
- UI shows optimistic "Unbanning…" badge with spinner, replaces with "Unbanned ✓" only after the next poll confirms IP is gone from `bannedIps` list.
- Disable the unban button for that IP for 5s after click (prevent double-fire).
- Backend wrapper: after `unbanip`, poll `status <jail>` up to 3 times at 500ms intervals; if IP still present after 3 attempts, return `{status: "pending", message: "Fail2ban acknowledged but list still shows IP. Verify next refresh."}` rather than fake-success.
**Detection:** Manual UAT step — click unban, observe spinner-then-confirmation transition.

### W-02 — Fail2ban-client Polling Cadence Tradeoff
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** Polling every 1s — `top` shows fail2ban-client subprocess churn at noticeable CPU. Polling every 30s — admin clicks unban, waits, sees no change for 30s, gives up thinking unban failed.
**Why it happens:** Each `fail2ban-client status sshd` is a fresh subprocess; on Mini PC's spinning-rust I/O profile, ~50ms each. At 1s cadence with 4 jails parallel = 200ms/s overhead = noticeable.
**Prevention:**
- Default cadence: 5s polling + manual "Refresh" button next to the IP list.
- After any mutation (ban/unban), trigger ONE immediate poll, THEN return to 5s cadence.
- Cache the parsed status result in livinityd memory for 2s — if a tRPC query arrives within that window, return cached.
- Show "last updated Xs ago" timestamp in UI to set expectations.
**Detection:** Performance test — measure CPU% during 5min of fail2ban panel open; assert < 1% steady-state.

### W-03 — Fail2ban Multi-Jail UX: Hardcoded "sshd"
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** v29.4 ships hardcoded `getJailStatus("sshd")` everywhere. Future user adds a jail (e.g., recidive, nginx-http-auth) via custom `jail.local` — UI doesn't show it. User concludes the panel is broken.
**Why it happens:** `install.sh:528` only configures sshd jail. But fail2ban supports many jails; some users may add their own.
**Prevention:**
- Backend: `listJails()` calls `fail2ban-client status` first, parses the comma-delimited "Jail list:" line, then iterates `getJailStatus(jail)` per jail.
- UI: Tabs (or accordion) for each discovered jail. Default to "sshd" tab if present.
- Empty state if no jails: "No jails configured. Run `fail2ban-client reload` after editing /etc/fail2ban/jail.local."
**Detection:** UAT — manually add a `recidive` jail to `/etc/fail2ban/jail.local`, reload, verify UI shows new tab.

### W-04 — Fail2ban Service-State Detection Ambiguity
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** Admin sees red banner "Fail2ban is broken" even though they intentionally `systemctl stop fail2ban`. Or: admin sees green status but `fail2ban-client status` actually fails because no jails are configured.
**Why it happens:** Three distinct states: (1) binary missing, (2) service stopped, (3) service running but no jails. UI must distinguish.
**Prevention:**
- Backend `getServiceState()` returns: `{binary: "found"|"missing", service: "active"|"inactive"|"failed"|"missing-unit", jails: number}`.
- UI banner copy:
  - `binary: missing` → red: "fail2ban not installed. Run /opt/livos/install.sh."
  - `service: inactive` + `binary: found` → yellow: "Fail2ban service stopped. [Start service] button. (You may have intentionally disabled it.)"
  - `service: active` + `jails: 0` → yellow: "Fail2ban running but no jails configured."
  - `service: active` + `jails > 0` → green: "Fail2ban active, X jail(s) configured."
**Detection:** Each of the 4 states has a distinct UAT verification step.

### W-05 — Fail2ban Restart While UI Open Crashes Status Query
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** Admin restarts fail2ban for an unrelated reason. UI's next 5s poll calls `fail2ban-client status sshd` → returns "Could not find server" briefly → UI throws unhandled error → user sees blank panel for 30s until manual refresh.
**Why it happens:** Fail2ban server restart leaves a ~5s window where the Unix socket isn't accepting connections.
**Prevention:**
- Backend: catch "Could not find server" / connection-refused / socket-not-found errors. Return `{transient: true, retry: true}`.
- UI: on `transient: true`, show "Fail2ban restarting…" badge + retry-with-backoff (1s, 2s, 4s, max 10s) — do NOT crash.
- Hard timeout: if 30s of `transient: true`, escalate to error banner with "Service may have failed permanently. Check `journalctl -u fail2ban`."
**Detection:** Integration test — `systemctl restart fail2ban` while panel is polling, assert UI stays in "restarting" state then recovers.

### W-06 — IP Source Mismatch: HTTP X-Forwarded-For vs SSH Source IP
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** Admin's "current IP" shown in UI is `10.0.0.1` (Caddy's view via X-Forwarded-For from CF) but fail2ban sees the SSH source as `203.0.113.1` (the user's actual public IP). Self-ban check fails to match — admin can ban their own SSH IP without warning.
**Why it happens:** Caddy injects X-Forwarded-For with the Cloudflare-edge-relayed client IP. SSH connections bypass Caddy entirely; fail2ban journal-tails sshd directly and sees the raw TCP source IP. These are DIFFERENT IPs even for the same admin.
**Prevention:**
- The "your current IP" detection used in self-ban check (B-02) MUST query SSH source IP, NOT HTTP source.
- One option: parse `who -u` or `loginctl list-sessions --output=json` to find current SSH sessions and their IPs.
- Cleaner: admin enters their SSH source IP manually OR UI shows BOTH "your HTTP source IP (X-Forwarded-For)" and "active SSH sessions (from `who`)" with explicit labels.
- Self-ban check matches against EITHER IP set.
**Detection:** Manual UAT — connect via SSH from one IP and HTTPS from a different IP (e.g., phone hotspot for HTTPS, laptop for SSH). Verify UI shows both, ban-attempt matches against either.

### W-07 — Cloudflared Version Pin vs Latest-Install
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** install.sh always installs latest cloudflared. Cloudflare ships a major version that bumps the tunnel-config schema. Re-running install.sh on an existing system breaks the running pc.* tunnel because the new cloudflared rejects the old config.
**Why it happens:** install.sh:502 is `command -v cloudflared || install_latest`. No version check.
**Prevention:**
- Pin a known-good version (e.g., `2026.3.0` in v29.4): `[ "$(cloudflared --version | awk '{print $3}')" = "2026.3.0" ] || install_specific_version`.
- OR: idempotent guard purely on `command -v` (current behavior) but do NOT auto-upgrade. User can manually upgrade later if they want.
- Add version to `installation.json` audit row alongside other binaries.
**Detection:** UAT — re-run install.sh on Mini PC, verify cloudflared version unchanged.

### W-08 — SSH-over-cloudflared ProxyCommand Not Surfaced to User
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** Admin enables SSH gateway. UI says "SSH gateway active at ssh.bruce.livinity.io." Admin runs `ssh ssh.bruce.livinity.io` from laptop → connection refused (Cloudflare doesn't route raw SSH; needs cloudflared client wrapper). Admin gives up, reverts to opening port 22 manually.
**Why it happens:** Cloudflare access SSH requires the CLIENT to use `cloudflared access ssh --hostname …` as ProxyCommand. Without it, the public DNS doesn't resolve to anything sshd-shaped.
**Prevention:**
- UI MUST surface the exact command the user runs from their laptop:
  ```
  ssh -o ProxyCommand="cloudflared access ssh --hostname ssh.{user}.livinity.io" bruce@ssh.{user}.livinity.io
  ```
- Plus a "Copy to clipboard" button.
- Plus a link to Cloudflare's `cloudflared` install page for laptops.
- Plus an `~/.ssh/config` snippet:
  ```
  Host ssh.{user}.livinity.io
    ProxyCommand cloudflared access ssh --hostname %h
    User bruce
  ```
**Detection:** UAT — admin follows the displayed instructions on a fresh laptop, connects successfully without external help.

### W-09 — Cloudflared Threat Model Leak Undocumented
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** Security-conscious user discovers post-deploy that Cloudflare's free tier terminates TLS at the edge — meaning Cloudflare CAN see plaintext SSH IF SSH was being run plaintext (which it isn't — SSH negotiates its own E2EE inside the tunnel, but the user doesn't necessarily know this).
**Why it happens:** Documentation gap. Default assumption is "Cloudflare can see my SSH" — which is FALSE for SSH because SSH is E2EE inside the tunnel, but the surface needs explicit explanation.
**Prevention:**
- Settings panel docs: "SSH establishes its own end-to-end encryption inside the cloudflared tunnel. Cloudflare sees only encrypted SSH bytes — your SSH session is NOT visible to Cloudflare even though the tunnel terminates TLS at their edge."
- Link to Cloudflare's own docs on Zero Trust SSH.
- Mention this is the same threat model as v18.0 noVNC (`pc.{user}.livinity.io`) which has been in production since v18.0 — no new threat surface.
**Detection:** Doc review — security-paragraph present in Settings panel before SSH gateway is enabled.

### W-10 — UFW Blocks cloudflared Access to localhost SSH
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** Admin enables SSH gateway → cloudflared starts → tunnel claims healthy → SSH connection through tunnel hangs. `journalctl -u cloudflared@ssh-gateway` shows "connection refused 127.0.0.1:22". UFW is blocking 127.0.0.1:22 outbound from the cloudflared process.
**Why it happens:** Some hardened Ubuntu setups block ALL outbound except whitelisted ports. cloudflared connects from itself to 127.0.0.1:22, which UFW may block.
**Prevention:**
- `install.sh:setup_ssh_gateway()` checks `ufw status` AND `iptables -L OUTPUT` for 127.0.0.1:22 rules.
- If blocked, surface a warning + offer to add UFW rule: `ufw allow out from 127.0.0.1 to 127.0.0.1 port 22`.
- Better: cloudflared's `ssh://localhost:22` is from-cloudflared-process to local sshd — should already work since both are on loopback.
- Test: `nc -zv 127.0.0.1 22` from cloudflared user context after install.
**Detection:** Pre-flight check in `setup_ssh_gateway()` returns warning if loopback blocked.

### W-11 — Cloudflare Account Auth Expires; Tunnel Fails Silently
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** ssh.{user}.livinity.io was working fine for weeks. CF account password rotated; cloudflared certs expire. Tunnel goes down silently. Admin can't SSH; can't tell why.
**Why it happens:** cloudflared cert lives in `~/.cloudflared/cert.pem` (or root's). When the underlying CF account auth state changes, the cert may stop being valid.
**Prevention:**
- Health-check tRPC route: `sshGateway.getHealth()` returns `{tunnelStatus, lastConnectAt, certExpiry, errors}`.
- UI surfaces "tunnel unhealthy / re-auth needed" banner with link to cloudflared re-auth instructions.
- systemd watchdog on `cloudflared@ssh-gateway`: `Restart=on-failure`, `StartLimitBurst=3` — after 3 failures, log loud error.
- Document cert expiry / re-auth procedure: `cloudflared tunnel login` → re-issues cert.
**Detection:** Negative-path test — manually delete `cert.pem`, verify health route surfaces error within 60s.

### W-12 — Tool Registry Diagnostic False Positives
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** WARNING
**Warning sign:** Diagnostic UI shows "missing built-ins: web_search, gmail_send, telegram_send" — but actually these are correctly missing because the user never set up API keys. Re-syncing won't help. Admin re-syncs, panic-clicks, possibly enables tools that NEED api keys to work.
**Why it happens:** "Missing because Redis lost it" vs "missing because precondition not met" are conflated. Phase 21 already has conditional tool registration (WhatsApp, channel, Gmail gated behind connection-state checks). The diagnostic must respect those gates.
**Prevention:**
- Diagnostic queries the SAME precondition logic that registration uses.
- Categorize results:
  - `missing_lost`: tool should be registered, isn't. RESYNC HELPS.
  - `missing_precondition`: tool conditional, precondition not met. RESYNC WON'T HELP. Show "Configure Telegram first" link.
  - `disabled_by_user`: user-set override. Show "[Re-enable]" button.
- Re-sync button is greyed out / no-op if zero tools in `missing_lost` category.
**Detection:** Test — disconnect Telegram, run diagnostic, assert `telegram_send` shows in `missing_precondition` not `missing_lost`.

### W-13 — Tool Registry Sync Re-Installs Marketplace Apps Surprisingly
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** WARNING
**Warning sign:** User manually uninstalled a marketplace MCP. Resync re-syncs from "all sources" including the marketplace install_history table. Tool reappears. User doesn't understand why.
**Why it happens:** "All sources" includes the marketplace registry. The user's resync intent is "restore built-ins"; the implementation goes wider.
**Prevention:**
- Resync UI offers TWO actions: "Restore built-ins only" (default, scope: hardcoded source) vs "Re-sync all sources" (scope: all 4 sources, requires extra confirmation).
- Backend: `flushAndResync({scope: "builtins" | "all"})`.
- Marketplace tools obey user's uninstall: `user_capability_overrides` includes `installed: false` flag for marketplace items. Resync respects.
**Detection:** Test — uninstall a marketplace MCP, run "Restore built-ins only", assert MCP stays uninstalled.

### W-14 — Tool Registry Sync History Audit Trail Lost on Flush
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** WARNING
**Warning sign:** Admin runs flushAndResync. Past sync timestamps (`capability:_meta:last_sync_at`) are wiped. No way to tell when a manual resync happened vs an automatic one.
**Why it happens:** `redis.del("capability:*")` is greedy; eats meta keys.
**Prevention:**
- Use specific key patterns: `redis.scan` for `capability:tool:*`, `capability:skill:*`, `capability:mcp:*`, `capability:hook:*`, `capability:agent:*` — DELETE only those.
- Preserve `capability:_meta:*`, `capability:_audit:*`, `capability:_history:*`.
- Each resync writes audit row: `redis.lpush("capability:_audit", JSON.stringify({ts, actor, scope, before, after}))` — capped at 100 entries.
**Detection:** Test — run resync, check `capability:_audit` list grew by 1, check meta keys preserved.

### W-15 — Tool Registry Integration Test Trashes Live Redis
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** WARNING (BLOCKER if test runs against prod accidentally)
**Warning sign:** Developer runs `pnpm test` locally, accidentally connected to Mini PC's Redis (REDIS_URL env var leaked from .env). Test calls `redis.flushdb()`. Mini PC user's actual capability registry wiped. Real downtime.
**Why it happens:** Default Redis for tests not isolated from default Redis for app.
**Prevention:**
- Test setup MUST use a fresh Redis DB index (`SELECT 15`) AND a fresh prefix (`test:capability:*`).
- Test setup asserts `redis.dbsize()` is 0 OR matches expected fixture size — refuse to run if the DB looks "lived in."
- CI uses ephemeral Redis container; local devs use `redis://localhost:6379/15`.
- Refuse to run integration tests if `REDIS_URL` matches Mini PC IP (`10.69.31.68`) — explicit guard.
**Detection:** Test setup module logs `redis.dbsize()` + `REDIS_URL` at the start of every test run; refuses if either looks production.

### W-16 — Identity Hallucination at High Temperature
**Phase:** A2 model-identity fix (Phase 46)
**Severity:** WARNING
**Warning sign:** A2 fix lands. User runs the diagnostic — broker `model: "claude-opus-4-7"` is correct. User then asks "hangi modelsin?" colloquially in Turkish. Model still answers "Claude 3.5 Sonnet" because the systemPrompt identity line was suppressed under conversational entropy.
**Why it happens:** LLMs hallucinate identity when prompted ambiguously. Even a perfect identity line doesn't survive arbitrary user prompting at temperature > 0.
**Prevention:**
- Test the DETERMINISTIC surface (literal API request showing `response.model` field) BEFORE assuming the systemPrompt line was suppressed.
- The systemPrompt edit's success criterion is "shows correct identity in NEUTRAL prompts" not "always responds correctly to all prompt variants."
- If user reports continued hallucination after A2 ships, route to "this is normal LLM behavior, not a bug" — UAT must distinguish.
- Document in Phase 46 plan: "A2 fixes the deterministic identity layer. It does NOT prevent all conversational hallucination — that's an LLM property, not a LivOS bug."
**Detection:** Phase 46 UAT has TWO test classes: (1) deterministic — `response.model` field correct; (2) probabilistic — neutral prompts get correct identity ≥ 90% of the time.

### W-17 — "44+ Commits Ahead of Origin" Lesson Re-Learned

**Severity:** WARNING
**Warning sign:** v29.4 closes with `git log origin/master..HEAD --oneline | wc -l` = 30+. User has been working locally without pushing. CI runs are batched at the end; if one phase introduced a regression in phase N+5, debugging is painful.

### W-18 — Backout Strategy: New Modules Lack `uninstall` Mirror
**Phase:** Fail2ban admin + SSH gateway (Phase 47, 48)
**Severity:** WARNING
**Warning sign:** Admin enables SSH gateway, decides they don't want it, asks "how do I undo this?" — discovers there's no undo button. Has to manually `systemctl disable cloudflared@ssh-gateway` + delete config files + `cloudflared tunnel delete`.
**Why it happens:** Forward-only feature additions. v29.2 factory reset is the only "undo all" path; users want fine-grained undo.
**Prevention:**
- **Fail2ban admin panel:** ship with Settings toggle "Show fail2ban admin in UI" (default on). Toggle hides the panel without uninstalling fail2ban — non-destructive backout.
- **SSH gateway B3b:** ship `uninstall_ssh_gateway()` mirror function in install.sh. Settings UI has "Disable SSH gateway" button that calls this. Mirrors install.sh's `setup_ssh_gateway()` exactly in reverse.
- Both backout paths leave the underlying binaries (fail2ban, cloudflared) installed — only feature configuration is removed.
**Detection:** Each Phase MUST include "backout test" UAT step.

### W-19 — Mini PC Mobile UX: Cellular IP vs WiFi IP for Self-Ban Check
**Phase:** Fail2ban admin (Phase 47)
**Severity:** WARNING
**Warning sign:** Admin opens Fail2ban panel from phone on cellular. UI's "your current IP" shows the CGNAT IP (e.g., 100.64.0.123). Admin bans 100.64.0.123 thinking it's an attacker — cellular network's other users now blocked too. Or admin switches WiFi mid-session, "your current IP" stale, self-ban check misses.
**Why it happens:** Mobile users have rapidly changing source IPs (cellular, WiFi, VPN). The "your current IP" detection lags by one request.
**Prevention:**
- UI surfaces TWO labels:
  - "Your HTTP request came from: 1.2.3.4" (with refresh button to re-detect)
  - "Active SSH sessions from: 5.6.7.8, 9.10.11.12 (from `who`)"
- "I'm on cellular / behind CGNAT" toggle: when checked, suppress the self-ban check (because admin can't reliably know if other CGNAT users would be affected).
- For mobile: extra confirmation modal before any ban action.
**Detection:** Manual UAT — open panel from phone on cellular, verify both IPs displayed, verify cellular toggle suppresses self-ban warning.

### W-20 — Test Environment vs Production: Mocking fail2ban-client / cloudflared Hides Bugs
**Phase:** Fail2ban admin + SSH gateway (Phase 47, 48)
**Severity:** WARNING
**Warning sign:** Unit tests mock `execFile("fail2ban-client", …)`. All tests pass. Deploy to Mini PC. fail2ban-client output format differs from mock (e.g., trailing whitespace, different jail list comma vs newline). UI breaks.
**Why it happens:** Mocking external binaries hides real-world output drift.
**Prevention:**
- NO unit-test mocks for `fail2ban-client` / `cloudflared` / `journalctl`. These are integration boundaries.
- Two strategies:
  1. **Containerized integration tests:** Docker container with real fail2ban + sshd; LivOS test suite shells in. Slow but real.
  2. **Skip integration tests; rely on UAT.** Phase 47 UAT runs on Mini PC scratchpad with real fail2ban. Faster CI, less coverage.
- Recommend strategy 2 for v29.4 (matches v29.3 broker UAT-deferred pattern), but add a "verbatim output capture" fixture in unit tests: paste real fail2ban-client output into a test fixture, parser asserts against it. Update fixture when fail2ban version changes.
**Detection:** Phase 47 plan explicitly forbids `vi.mock("child_process")` for fail2ban-client; uses fixture-based parser tests.

### W-21 — Sync History Wiped Erodes Audit Trail Across Resyncs
**Phase:** Nexus tool registry restore A1 (Phase 45)
**Severity:** WARNING
**Warning sign:** Admin runs flushAndResync 3 times in 1 day. Audit log shows only the last one. Compliance / debugging requires "when did this happen N times?" history.
**Why it happens:** Same root cause as W-14 (covered separately) — but specifically about cumulative history vs single-most-recent timestamp.
**Prevention:**
- Use Redis LIST `capability:_audit_history` (LPUSH on each event, LTRIM 0 99 — keep last 100).
- Mirror to PG `capability_audit` table for permanent history.
- Settings UI shows "Last 5 resyncs" with timestamps + actor + scope.
**Detection:** Run resync 3 times, verify list has 3 entries.

### W-22 — Detached spawn cgroup-escape needed for SSH gateway install from UI
**Phase:** SSH gateway B3b (Phase 48)
**Severity:** WARNING
**Warning sign:** UI's "Install SSH gateway" button calls `setup_ssh_gateway()` via livinityd → systemctl restart cloudflared during install → cloudflared@ssh-gateway is a NEW unit livinityd just configured → restart cascades through livinityd's own cgroup → livinityd dies.
**Why it happens:** Same systemd cgroup-escape pattern from Phase 30/31 + v29.2 factory reset. Long-running setup that triggers systemd actions on its own ancestry needs `systemd-run --scope --collect` to escape.
**Prevention:**
- UI's install action MUST use the cgroup-escape pattern: `systemd-run --scope --collect /opt/livos/data/scripts/setup-ssh-gateway.sh` (mirrors v29.2 factory reset's spawn).
- Reference the existing pattern in `feedback_subscription_only.md` and `reference_cgroup_escape.md`.
**Detection:** UAT — install SSH gateway via UI, verify livinityd PID unchanged before/after install.

## Minor Pitfalls

Add ONE-line README update with link to settings panel docs.

Use UUID suffix: `<ts>-<uuid8>-banevent.json`.

No code change; document in Phase 47 plan: "Fail2ban admin panel does NOT touch Caddy config. Do not add caddy reload to wrapper."

Pin parser to fail2ban 1.0.2 output format; add version-detection fallback that warns on version mismatch.

### M-05 — UI keyboard shortcut for "ban" too easy to fat-finger
Don't bind a keyboard shortcut for ban actions. UI button-only.

### M-06 — admin-only checks miss the role-mismatch corner case
ALL fail2ban + SSH gateway tRPC routes use `adminProcedure` (Phase 12 RBAC pattern), NOT `protectedProcedure`. Verify with grep.

### M-07 — fail2ban-client subprocess spawn cost; 5s timeout is tight
Use 10s timeout (sufficient slack for slow Mini PC I/O peaks).

### M-08 — i18n: status banners hardcoded English
Match existing LivOS i18n pattern (English baseline; user has expressed Turkish status updates preference — ensure key error messages are translatable).

### M-09 — `node_modules/@nexus/core` symlink resolution differs across pnpm versions
A2 diagnostic step 5 (`readlink -f`) may fail on pnpm 9.x due to a different symlink layout. Test the diagnostic on the actual deployed pnpm version.

### M-10 — fail2ban journal source needs sudo journalctl read on some systems
Mini PC: livinityd is root, no issue. Document constraint for future non-root deployment.

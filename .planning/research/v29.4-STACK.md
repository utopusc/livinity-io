# v29.4 Stack Research — Server Management Tooling + Bug Sweep

**Domain:** Fail2ban admin UI + SSH gateway + Nexus tool registry diagnostic + 4 carry-forward bug fixes
**Researched:** 2026-05-01
**Confidence:** HIGH for all NEW additions (each verified against existing LivOS patterns + current upstream docs)

## Executive Summary

**The headline finding: install.sh already installs fail2ban (line 517-540) and cloudflared (line 502-512) with idempotent `command -v` guards.** That changes the entire shape of v29.4:

- **Fail2ban admin panel: ZERO new dependencies.** The binary is present, the SSH jail is configured, systemd unit is enabled. We only need a thin Node-side `child_process.exec` wrapper. The existing `livinityd` already runs as root on the Mini PC (it manages systemd services, Docker, and writes to `/etc/`) — sudoers/polkit are non-issues.
- **SSH gateway B3b: also ZERO new dependencies.** cloudflared is already installed; if user picks B3b we just add an idempotent `cloudflared tunnel run` systemd-managed config — NO Tailscale, NO WireGuard.
- **Nexus tool registry restore (A1): ZERO new stack.** Existing Redis client + capability-registry init flow handles it.
- **C1-C4 carry-forwards: ZERO new stack.** All surgical edits to existing files.
- **Optional B4 SSH viewer: ONE small dep if shipped** — `maxmind` for offline geo-IP, OR skip the dep entirely and stream `journalctl -u ssh -o json -f` raw. Recommendation: skip the dep for v29.4.

**Net stack delta for v29.4: 0 new packages if B4 deferred. 1 new package (`maxmind` ^5.x) if B4 ships with geo enrichment.**

## Detailed Recommendations Per Question

### 1. Fail2ban Admin Panel

**1a — Parse `fail2ban-client status sshd` output:**

There is **no native JSON output** in fail2ban 1.0.2. The supported flavors are `short`, `basic`, `cymru`. JSON output has been an open feature request since 2017 and is not landed.

**Recommended: text-parse the default `basic` output.** It's stable across 0.9 → 1.1.x:

```
Status for the jail: sshd
|- Filter
|  |- Currently failed: 0
|  |- Total failed: 142
|  `- Journal matches: _SYSTEMD_UNIT=ssh.service + _COMM=sshd
`- Actions
   |- Currently banned: 2
   |- Total banned: 17
   `- Banned IP list: 1.2.3.4 5.6.7.8
```

A 30-line regex parser is more reliable here than any wrapper library. The `Banned IP list:` line is whitespace-separated IPs.

For the multi-jail overview, parse two commands:
- `fail2ban-client status` → comma-delimited jail list after `Jail list:`
- `fail2ban-client status <jail>` per-jail (parallelized via `Promise.all`)

**1b — Privilege model for unban from livinityd:**

`livinityd` already runs as root on the Mini PC. **No sudoers, polkit, or D-Bus bridge needed.** Just `execFile("fail2ban-client", ["set", "sshd", "unbanip", ip], {timeout: 5000})`.

If a future audit moves livinityd to a non-root user, the right sudoers entry would be:
```
livos ALL=(root) NOPASSWD: /usr/bin/fail2ban-client set sshd unbanip [0-9.]*, /usr/bin/fail2ban-client status, /usr/bin/fail2ban-client status sshd
```
But that's a v30+ concern, not v29.4.

**1c — Detect-and-suggest install:**

Trivial — same pattern as `install.sh:518`:
```ts
const hasFail2ban = await checkBinary("fail2ban-client");
if (!hasFail2ban) {
  return { error: "fail2ban_missing", suggestion: "Run /opt/livos/install.sh to reinstall" };
}
```
**Anti-fragile fallback:** since install.sh installs fail2ban, this branch should only fire on a manually-broken system. Surface a clear error in UI banner + one-click "Install now" button via the existing systemd-run cgroup-escape pattern (mirrors v29.2 factory-reset spawn).

**Backend wrapper location:** `livos/packages/livinityd/source/modules/fail2ban-admin/` (new module, mirrors v29.3 `livinity-broker/` shape):
- `index.ts` — `getJailStatus(jail)`, `listJails()`, `unbanIp(jail, ip)`, `banIp(jail, ip)`
- `client.ts` — execFile wrapper + 5s timeout + structured error types
- `parser.ts` — text-output parsers
- `events.ts` — JSON event row writer at `/opt/livos/data/security-events/<ts>-<action>.json`

### 2. SSH Gateway

**Recommendation: B3a (passive) for v29.4 default. If user picks B3b, use cloudflared (already installed).**

| Criterion | cloudflared (recommended) | Tailscale | WireGuard |
|-----------|---------------------------|-----------|-----------|
| Already on Mini PC | YES (install.sh:502) | NO | NO |
| Latest stable | 2026.3.0 (Mar 2026) | 1.96.5 (Apr 2026) | kernel module |
| NAT traversal | Outbound QUIC only | Outbound + DERP relay | Requires inbound UDP OR public relay |
| Idempotent re-install | YES (deb + command -v) | YES (apt + auth-key) | YES (modprobe + wg-quick @) |
| Ubuntu 24.04 quirks | None | systemd-resolved DNS | systemd-resolved DNS |
| Auth model fit | CF account + named tunnel ID | TS account + ACL tags | Static keys per-peer |
| Threat model fit (D-NO-SERVER4) | Routes via CF edge → Mini PC. Server5 untouched. Scoped to `ssh.{user}.livinity.io` | Mesh — every Tailnet peer can attempt to reach Mini PC unless ACLs locked | Point-to-point per peer |
| Billing/quota | Free unlimited tunnels | Free 100 devices, 3 users | Self-hosted |

**One-paragraph rationale (B3b → cloudflared):** Already installed, NAT-traversal works out of the box (outbound QUIC only — no port forwarding, no static IP), and it scopes naturally to `ssh.{username}.livinity.io` mirroring the existing `pc.{username}.livinity.io` pattern from v18.0. Tailscale offers slightly better mesh semantics but adds a second VPN daemon, second account model, second package update path. WireGuard requires inbound port forwarding (defeating the "user has no static IP" assumption) or a public WG relay Server5 doesn't host. cloudflared also fits the threat model — Server5 stays untouched, Cloudflare DNS-only is unchanged, tunnel credential lives entirely on the Mini PC.

**B3b install.sh patch landing zone:** new function `setup_ssh_gateway()` at `install.sh:~610` (right after `setup_desktop_streaming` so SSH gateway sits next to desktop streaming gateway):
```bash
setup_ssh_gateway() {
  [ -f /etc/cloudflared/ssh-tunnel.json ] && { ok "SSH gateway already configured"; return; }
  # cloudflared tunnel create ssh-{hostname}
  # cloudflared tunnel route dns ssh-{hostname} ssh.{user}.livinity.io
  # write /etc/cloudflared/config.yml with ssh:// origin
  # systemctl enable --now cloudflared@ssh-tunnel
}
```

**B3a (passive) for v29.4:** zero install.sh changes. Unban-via-UI is the only mechanism. Document SSH gateway as future v30+ optional feature.

### 3. SSH Session Viewer (Optional B4)

**Recommendation: WS-stream `journalctl --output=json --follow` raw, no geo enrichment for v29.4.**

**Why no geoip dep for v29.4:**
- MMDB binary ~7MB shipped, ~30MB unpacked
- GeoLite2 license requires deleting stale databases within 30 days of release — adds cron/lifecycle concern
- Raw IP + ASN-from-cymru flag (already from `fail2ban-client status sshd cymru`) gives 80% of the value for 0% install cost
- If user later wants country flags, add `maxmind` + `node-geolite2-redist` as a SINGLE follow-up phase in v30+

**Streaming pattern:**
```ts
const proc = spawn("journalctl", ["-u", "ssh", "--output=json", "--follow", "--since", "1 hour ago"]);
proc.stdout.pipe(splitStream("\n")).on("data", (line) => {
  const entry = JSON.parse(line);
  if (entry._SYSTEMD_UNIT === "ssh.service" && /sshd/.test(entry.MESSAGE)) {
    wsServer.broadcast({ ts: entry.__REALTIME_TIMESTAMP, msg: entry.MESSAGE, ip: extractIp(entry.MESSAGE) });
  }
});
```
NO new deps — `child_process.spawn` + Node line-splitting (or Phase 28 cross-container log streaming utility).

**WS endpoint:** `/ws/ssh-sessions` mirrors `/ws/docker/logs` from Phase 28 — same JWT auth, same admin gate.

**~50 LOC of glue.** No new Redis client, no new persistence layer, no new ORM.

## Executive Summary

**v29.4 adds 3 net-new modules and modifies 7 existing files.** It introduces zero new packages, zero new daemons (cloudflared was already installed by `install.sh:502` — see STACK), zero new database tables (audit log reuses `device_audit_log`), and one new tRPC namespace (`fail2ban`). Plus a possible second optional new namespace (`security` or repurposing `apps`) for the marketplace health probe.

**Headline integration insights:**

1. **Fail2ban admin module is a near-clone of `livinity-broker/`** — same file shape (`index.ts` / `client.ts` / `parser.ts` / `events.ts` / `routes.ts` + integration tests), same registration ritual (one line in `server/trpc/index.ts`, three lines in `httpOnlyPaths`), same runs-as-root assumption (livinityd already roots the Mini PC).
2. **`device_audit_log` table reuses cleanly for ban/unban events** — schema is `(user_id, device_id, tool_name, params_digest, success, error, timestamp)`. Repurpose `device_id := 'fail2ban-host'` and `tool_name := 'unban_ip' | 'ban_ip'`; `params_digest := sha256(JSON.stringify({jail, ip}))`. Append-only trigger already enforces immutability. **Saves a migration, saves a test surface, fits the audit philosophy already in the codebase.**
3. **Diagnostics card is a single shared UI surface** — A1 (registry restore) + A2 (model identity) + A3 (Bolt.diy reachability probe) all fold into one new `livos/packages/ui/src/routes/settings/_components/diagnostics-section.tsx` with three sub-cards. One scaffold, three diagnostic primitives. Saves ~2x scaffolding cost vs three independent panels.
4. **Server Management surface decision is a fork in the road** — Fail2ban UI can either land as a 13th sidebar entry inside `LIVINITY_docker` (cheap, ships in days) OR become a peer system app `LIVINITY_security` (more work, cleaner long-term Server Management home for v30+ backups, monitoring, alerts). Both options surface below with explicit trade-offs; lean is **13th sidebar entry** for v29.4 with a v30+ promotion path.
5. **A2 model identity is a diagnose-then-fork** — root cause is unknown (pnpm-store dist drift OR `systemPrompt` confabulation). Phase must start with an on-Mini-PC diagnostic step that branches into either an `update.sh` patch (architecturally trivial) OR a Phase-40-style D-40-01 surgical edit on the sacred file (architecturally heavy, requires C2 SHA re-pin to land first). Sequencing matters: **C2 must land before A2's heavy branch can.**

**Build-order recommendation (parallelizable):**
- **Track 1 (carry-forward sweep, 1 phase):** C1 + C2 + C3 + C4 — independent files, no UI, lands first.
- **Track 2 (Fail2ban + Diagnostics surface, 2 phases parallel to Track 1):** Phase A: backend module + tRPC + audit-log reuse + tests. Phase B: UI sidebar entry/section + diagnostics card.
- **Track 3 (A1+A2 diagnostic-then-fix, 1 phase, depends on C2 from Track 1):** A1 ships fully. A2 ships diagnostic; fix branch decided by Mini PC observation.
- **Track 4 (Bolt.diy probe + B3 SSH gateway, 1 small phase, fully independent):** A3/A4 health-probe tRPC + B3a documentation OR B3b cloudflared install.sh patch.

Total: **5-6 phases**, with Tracks 1/2/4 fully parallelizable and Track 3 sequenced after C2.

#### Cross-Cut: Audit Log Reuse vs New Table

**Recommendation: REUSE `device_audit_log`.** Schema (verified in `database/schema.sql:109-118`) is:
```sql
device_audit_log (
  id UUID, user_id UUID, device_id TEXT, tool_name TEXT,
  params_digest TEXT, success BOOLEAN, error TEXT, timestamp TIMESTAMPTZ
)
```

It fits ban events exactly:
- `device_id := 'fail2ban-host'` (sentinel string for "the LivOS host itself")
- `tool_name := 'unban_ip' | 'ban_ip' | 'list_jails'`
- `params_digest := sha256(JSON.stringify({jail, ip}))`
- `user_id := ctx.currentUser.id` (admin who clicked unban)
- The append-only trigger (`audit_log_immutable`) already protects ban events from tampering — same property fail2ban admin needs.

**Why NOT a new `security_audit_log` table:**
- Migration cost (schema.sql edit, indexes, backfill check, append-only trigger duplication)
- Two parallel audit query surfaces in UI (one for devices, one for security) — fragments the operator mental model
- Future events (registry credential rotation, factory reset trigger, B3b SSH gateway open/close) would each want their own table, leading to fragmentation
- Reusing `device_audit_log` extends the "audit row = single source of truth for sensitive admin actions" pattern from v22.0/v26.0

**Anti-pattern to avoid:** writing ban events to `/opt/livos/data/security-events/*.json` ONLY without DB row. JSON files are not query-friendly for the audit list view; v29.2 only used JSON-row pattern for update history because there's no DB-backed history table for installs. We have a DB table here — use it. **Keep the JSON event-row WRITE as a belt-and-suspenders backup for offline analysis** (mirrors Phase 33 OBS-01 update-history convention) but make `listEvents` read from PG.

### 2. SSH Gateway (B3a vs B3b)

#### B3a — Passive (recommended default)

**Architecture: zero new code outside Fail2ban admin scope.** B3a means "rely on B1 to unban yourself when SSH locks you out — no SSH-tunnel-as-service infrastructure."

**File changes: ZERO.** Documentation mentions only.

**Trust model:** unchanged. Operator SSHes via direct IP / existing channel; if banned, opens `https://livinity.io` (admin user already authenticated via JWT cookie), navigates to Settings > Security > Fail2ban, clicks "Unban my IP." No new attack surface, no new daemon, no Caddy interaction.

**Phase cost:** zero new phase. Documented as a usage pattern in B1's phase notes.

#### B3b — Active cloudflared (deferred or optional)

**File changes:**

```
install.sh                                            [+~80 lines — setup_ssh_gateway()]
livos/packages/livinityd/source/modules/system/[NEW]
  └── ssh-gateway-status.ts        [NEW] tRPC adminProcedure: read /etc/cloudflared/config.yml status, return {enabled, hostname, lastSeen}
livos/packages/livinityd/source/modules/server/trpc/common.ts  [no change — query only]
livos/packages/livinityd/source/modules/server/trpc/index.ts   [+1 import — system already registered, just adds a sub-route]
livos/packages/ui/src/routes/settings/_components/[NEW]
  └── ssh-gateway-card.tsx         [NEW] Settings > Advanced > SSH Gateway card; toggle disabled if cloudflared missing
```

**Landing zone in install.sh:** new function `setup_ssh_gateway()` placed **immediately after `setup_desktop_streaming()`** (v18.0 sibling). Both are "expose a host service through a tunnel" gateways; co-location helps future readers.

**systemd unit pattern question (resolved):**

cloudflared supports BOTH single-config-file-multi-tunnel and per-tunnel systemd units. **Recommendation: separate unit per tunnel.** Reasoning:
- The existing `pc.{user}.livinity.io` (v18.0) flow uses the relay tunnel via Server5 — it does NOT use cloudflared. So there's no pre-existing cloudflared tunnel to share with.
- Separate `cloudflared@ssh-tunnel.service` (templated unit) means unit failure is scoped — restart loop on the SSH tunnel doesn't kill any other future cloudflared use.
- If user ever runs install.sh without B3b consent, no unit is created → zero idle daemon overhead.

**Caddy interaction: cloudflared owns its own TLS to the Cloudflare edge.** No Caddy entry needed for `ssh.{username}.livinity.io`. The ssh:// origin is internal localhost:22 and cloudflared encrypts on the user's behalf via its named-tunnel credentials. Caddy's role is unchanged.

**Open question / D-decision needed:** Cloudflare credentials for the named tunnel. Either:
- (a) User runs `cloudflared tunnel login` interactively at install time (fits `install.sh`'s existing TUI pattern from v10.0 whiptail wizard)
- (b) Skip cloudflared auth at install time, gate B3b behind a UI flow that opens the auth URL

**Recommendation: (a)** — fits the install.sh pattern, matches the user's mental model ("install.sh asks me everything once").

**Trust boundaries:**

| Surface | Trust |
|---------|-------|
| `setup_ssh_gateway()` in install.sh | Runs as root (install.sh always does). Writes `/etc/cloudflared/config.yml` (mode 0600). |
| `cloudflared@ssh-tunnel.service` | Daemon, runs as `cloudflared` user (the package installs that user). |
| `system.sshGatewayStatus` tRPC | adminProcedure. Reads `systemctl is-active cloudflared@ssh-tunnel`. |
| `ssh.{username}.livinity.io` (external surface) | Cloudflare Access optional — recommend YES, scoped to user's email. Without Access this is a public SSH endpoint protected only by host SSH auth. |

**B3b is OPTIONAL for v29.4** per stack research (B3a is the default recommendation). If user picks B3b, it's one self-contained phase that's fully parallelizable with everything else.

### 4. Model Identity Stability (A2)

#### Two-Branch Architecture (root cause TBD)

This phase MUST start with a diagnostic step on the Mini PC. The architecture diverges based on the observation.

##### Branch 1 — pnpm-store dist drift (most likely per memory)

**File changes:** `update.sh` only.

```
update.sh (Mini PC)                                    [+~30 lines — existing 'multi-dir @nexus+core dist-copy loop' from v29.0 BUILD-02 needs hardening]
```

**Architecture impact:** trivial. update.sh already has a loop (Phase 31 BUILD-02) — this is a hardening pass, not a structural change.

**No sacred-file edit.** No C2 dependency. Architecturally low cost.

##### Branch 2 — `systemPrompt` confabulation (verified path, requires sacred-file edit)

**File changes:**
```
nexus/packages/core/src/sdk-agent-runner.ts            [SACRED — ONE surgical edit, mirrors Phase 40 D-40-01]
nexus/packages/core/src/providers/sdk-agent-runner-integrity.test.ts  [BASELINE_SHA re-pin]
```

The current sacred file (line 244-285 region grepped) builds `systemPrompt` as `_identityLine + this.config.systemPromptOverride ?? "You are Nexus..."` — and Phase 43.10/43.12 added a `_identityLine` injection but the model can still confabulate when it ignores the line.

The cleaner fix is to use the SDK's `{type: 'preset', preset: 'claude_code'}` system-prompt mode for built-in agent runs (where identity confabulation is the bug), keeping `systemPromptOverride` only for explicit override callers. This is a Phase-40-style D-40-01 surgical edit — change a single string-construction line into a conditional preset/string union.

**Architecture impact:** sacred-file edit. **Sequencing constraint: C2 (BASELINE_SHA re-pin to current `4f868d31...`) MUST land BEFORE this branch can edit-and-re-pin.** Otherwise the integrity test will fail twice (once because of v43.x drift, once because of the new edit).

#### Build-Order Decision Tree

```
Phase A2-diag (lands first, lightweight)
  ├── On Mini PC: confirm which root cause via 1 targeted test
  │      └── e.g. set systemPromptOverride='', ask "what model are you?"
  │             → If model says "Claude 3.5 Sonnet" with current dist: Branch 2 (confabulation)
  │             → If model says correct ID with current dist: Branch 1 (dist drift)
  │
  └── Decide branch

Phase A2-fix (Branch 1)              OR     Phase A2-fix (Branch 2)
  └── update.sh hardening                    ├── REQUIRES C2 already shipped
  └── No sacred-file edit                    ├── Sacred file surgical edit
  └── ~30 LOC                                ├── Re-pin BASELINE_SHA again
                                             └── ~10 LOC + audit comment
```

**Sequencing recommendation:** ship C2 in Track 1 first; A2 phase splits into A2-diag + A2-fix as a single phase with two outcome paths. Roadmapper should sequence A2 AFTER C2 lands so Branch 2 is available.

#### Trust Boundaries

A2 has no UI mutation surface beyond the diagnostic display in the shared diagnostics-section.tsx. The "fix" is ops-level (update.sh re-run OR a code patch + redeploy). **No new RBAC surface.**

## Anti-Patterns to Avoid (Architecture Specifically)

### Anti-Pattern 1: sudoers entry for fail2ban-client

**Mistake:** "Let's add a sudoers rule so livinityd can call fail2ban-client without root."
**Why wrong:** livinityd ALREADY runs as root on the Mini PC. Sudoers entries are net-new attack surface — a misconfiguration in the rule (e.g. the `[0-9.]*` IP regex from STACK.md) becomes a privilege-escalation bug. Don't add what's not needed.
**Do this instead:** direct `child_process.execFile("fail2ban-client", ...)` from livinityd. Document the "livinityd runs as root" assumption in the module README. If a future audit demands non-root livinityd, sudoers is a v30+ migration concern, not a v29.4 architecture choice.

### Anti-Pattern 2: New `security_audit_log` table

**Mistake:** "Ban events are different from device tool events — make a new table."
**Why wrong:** the `device_audit_log` schema fits ban events 1:1. Two tables doubles the migration surface, fragments the audit query UI, and forces every future "audit-able admin action" to pick a table — a non-decision that becomes a fragmentation source.
**Do this instead:** REUSE `device_audit_log` with sentinel `device_id := 'fail2ban-host'` and `tool_name := 'unban_ip'|'ban_ip'`. Append-only trigger already protects ban events for free. New events table is a v30+ "audit log unification" refactor, not a v29.4 architecture call.

### Anti-Pattern 3: Three independent diagnostic panels

**Mistake:** "A1, A2, A3 are different concerns — each gets its own settings panel."
**Why wrong:** they share 80% of the UI scaffold (card with title + status badge + detail + action button). Three panels = three sets of layout boilerplate, three sets of empty-state handling, three sets of "operator's mental map." When operator hits a bug, they want ONE place to look first.
**Do this instead:** one `DiagnosticsSection` with three `DiagnosticCard` sub-components. ~25% LOC savings, ~50% mental-model savings.

### Anti-Pattern 4: Adding 'fail2ban' as the sidebar entry name (not 'Security')

**Mistake:** "The module is called fail2ban-admin, so the sidebar entry should be 'Fail2ban'."
**Why wrong:** the operator doesn't think about fail2ban specifically — they think "I got locked out / I need to unban this IP / I want to see who's been banned." The sidebar entry should match the operator vocabulary, not the module's package name. Naming it 'Fail2ban' also boxes in the section — future audit-log/sessions/alerts sub-pages don't fit semantically.
**Do this instead:** name the sidebar entry **'Security'**. Module name stays `fail2ban-admin/`. Section can grow into a multi-tab security home in v30+ without rename pain.

### Anti-Pattern 5: B3b cloudflared sharing a unit with desktop streaming's v18.0 tunnel

**Mistake:** "v18.0 already runs a tunnel via Server5; let's reuse the same daemon for SSH."
**Why wrong:** v18.0 desktop streaming uses the LivOS relay tunnel via Server5 — NOT cloudflared. There's no daemon to share with. cloudflared is a brand-new daemon if B3b ships. Conflating them creates a phantom dependency and probably a broken systemd unit name collision.
**Do this instead:** if B3b ships, separate templated systemd unit `cloudflared@ssh-tunnel.service` distinct from anything v18.0 owns.

### Anti-Pattern 6: Health-probe as adminProcedure

**Mistake:** "App health diagnostics should be admin-only."
**Why wrong:** v29.3 Phase 44 already established the pattern that users self-probe their own resources (`usage.getMine`) and admins probe everyone's (`usage.getAll`). Making the marketplace app health probe admin-only inverts this — users can no longer self-diagnose Bolt.diy reachability without messaging admin.
**Do this instead:** `apps.healthProbe` is `privateProcedure`, query scoped to `user_app_instances WHERE user_id = ctx.currentUser.id`. Admin gets nothing extra (admins can already see all apps via existing routes; the probe is per-app self-service).

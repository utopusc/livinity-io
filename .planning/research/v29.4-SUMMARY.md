# v29.4 Research Summary — Server Management Tooling + Bug Sweep

**Project:** LivOS — Livinity Self-Hosted AI Server
**Domain:** Fail2ban admin panel + SSH gateway + Nexus tool registry diagnostic + v29.3 carry-forward bug sweep
**Researched:** 2026-05-01
**Confidence:** HIGH

## Executive Summary

v29.4 is a tight bug-sweep-plus-surgical-additions milestone that adds roughly 940 LOC on top of an already-running multi-user home server. The headline finding that reshapes the entire milestone is that fail2ban and cloudflared are already installed by `install.sh` (lines 517-540 and 502-512 respectively), which reduces the stack delta to exactly zero new packages. Every feature in scope — Fail2ban admin panel, SSH gateway passthrough, Nexus registry diagnostic, and the four v29.3 carry-forwards — is achievable with existing Node primitives (`child_process.execFile`, Redis, PostgreSQL, journalctl) and the tRPC/React Query patterns already established in v26.0–v29.3.

The recommended build approach is Cockpit-tier minimalism for the Fail2ban panel: list + unban + per-jail status cards + admin-attributed audit log. The single most important differentiator is the "Add to ignoreip whitelist" checkbox in the unban modal — without it, the "Claude SSH from cloud" use case breaks within 60 seconds of each unban because fail2ban re-bans the Cloudflare egress IP on the next failed auth attempt. All maximalist features (custom jail editor, editable findtime/maxretry/bantime, manual ban-IP control, geo/whois enrichment, B3b active cloudflared SSH gateway) are explicitly anti-features for the home-server tier and must not ship in v29.4.

The highest-risk sequencing constraint is sacred-file SHA coordination: C2 (audit-only re-pin of `BASELINE_SHA` from `623a65b9...` to `4f868d31...`) must land as its own commit before Phase 47's A2 branch can make the surgical `systemPrompt` edit. A2 itself must start with a 6-step on-Mini-PC diagnostic to determine whether the model identity bug is a pnpm-store dist-copy deployment bug (fix: update.sh hardening) or a source-level confabulation (fix: sacred-file surgical edit). Conflating these two root causes would result in the wrong fix being applied under the wrong commit message.

## Key Findings

### Recommended Stack

Zero new packages are required for v29.4. The existing runtime — livinityd (tsx/Node 22, runs as root on Mini PC), Redis (ioredis), PostgreSQL, React 18 + tRPC + React Query — covers everything. Fail2ban admin is a thin `child_process.execFile("fail2ban-client", ...)` wrapper with a 30-line text parser; the `status sshd` output format is stable across fail2ban 0.9–1.1.x. SSH gateway passthrough (B3a passive) requires zero new code beyond the unban + whitelist feature. SSH viewer (B4) is deferred; if it ships later, `journalctl -u ssh -o json -f` streams raw and needs no new dep either. GeoIP enrichment (maxmind + node-geolite2-redist) is explicitly deferred to v30+ due to license lifecycle overhead.

**Core technologies (existing, no additions):**
- `fail2ban-client` binary (already on Mini PC via install.sh:517): CLI-executed from livinityd for status, unban, and ignoreip writes
- `child_process.execFile` (Node built-in): privilege-safe argument-array invocation — no shell-string expansion, no sudoers needed (livinityd already runs as root)
- `device_audit_log` PG table (Phase 26 v26.0): reused for ban/unban events via sentinel `device_id := 'fail2ban-host'` — saves a migration and fragments no new audit surface
- `capability-registry.ts` (nexus/packages/core, Phase 22 v22.0): existing `syncFromAllSources()` reused for A1 registry restore; only `diagnoseRegistry()` and `flushAndResync()` are new (~80 LOC additions)
- `httpOnlyPaths` in `livos/packages/livinityd/source/modules/server/trpc/common.ts:8` (NOT `nexus/packages/api/src/common.ts` — path corrected by architecture research): all new tRPC mutations must be added here

### Architecture Approach

v29.4 adds three net-new modules and modifies seven existing files. The `fail2ban-admin/` module is shaped identically to `livinity-broker/` (index.ts / client.ts / parser.ts / events.ts / routes.ts) — anyone who has read the broker reads this in zero seconds. The Fail2ban UI lands as a 13th sidebar entry inside `LIVINITY_docker` named "Security" (not "Fail2ban") — the naming leaves room for v30+ security sub-features (alert center, audit-log query, sessions) without renaming pain. A new standalone `LIVINITY_security` system app is deferred to v30+ when backup admin and monitoring overlays justify the extra scaffolding. The shared `diagnostics-section.tsx` consolidates A1 (registry), A2 (model identity), and A3/A4 (Bolt.diy probe) into one Settings card grid, saving approximately 25% UI LOC versus three independent panels.

**Major components:**
1. `fail2ban-admin/` module (livinityd) — execFile wrapper + text parser + events writer + tRPC routes; registers as `fail2ban.*` namespace
2. `docker/sections/security/` (UI) — 13th sidebar entry in LIVINITY_docker; jail-list, banned-IP table, unban modal with whitelist checkbox, history tab
3. `settings/_components/diagnostics-section.tsx` (UI, NEW shared) — single scaffold hosting A1 registry card, A2 model identity card, A3/A4 marketplace health card
4. `nexus/packages/core/src/capability-registry.ts` (modified) — adds `diagnoseRegistry()` + `flushAndResync()` helpers consumed via tRPC `ai.diagnoseCapabilityRegistry` / `ai.flushAndResyncRegistry`
5. Carry-forward edits in: `livinity-broker/router.ts`, `livinity-broker/agent-runner-factory.ts`, `livinity-broker/openai-sse-adapter.ts`, `server/trpc/common.ts`, `sdk-agent-runner-integrity.test.ts`

### Critical Pitfalls

1. **Sacred file SHA coordination failure (B-04 + B-11)** — C2 commit must be audit-only (zero diff on `sdk-agent-runner.ts`, verifiable via `git diff --shortstat`), landing before A2's surgical edit. Any commit touching `sdk-agent-runner.ts` without a paired BASELINE_SHA update in the integrity test must be blocked by pre-commit hook. The two SHA movements (C2 audit-only re-pin, A2 behavior-change re-pin) must be separate commits with distinct audit comment blocks.

2. **A2 wrong-root-cause fix (B-05)** — A2 must NOT begin with a source edit. The phase plan must embed a 6-step on-Mini-PC diagnostic (check `response.model` field, inspect `ls /opt/livos/node_modules/.pnpm/@nexus+core*` for multiple dirs, `readlink -f` to resolve the actual import path) before branching into either update.sh hardening (deployment bug) or sacred-file edit (source bug). Skipping the diagnostic risks shipping a source edit that has no effect because the running binary is stale dist.

3. **Self-ban cycle in Fail2ban admin (B-01 + B-02)** — The unban call must use the action-targeted form to clear the match counter, not just remove the IP from the banned list. The ban route must extract the admin's source IP and compare against the ban target, returning a `{warning: "self_ban"}` payload that triggers a type-`LOCK ME OUT` confirmation gate (mirrors v29.2 factory reset). Input validation must reject CIDR /0 through /7 via Zod `z.string().ip()`.

4. **Tool registry flush+resync race window (B-06 + B-07)** — `redis.del("capability:*")` followed by `syncFromAllSources()` is non-atomic. The resync must build new manifests under a temporary prefix and then atomically swap. Critically, user-set `enabled: false` overrides from `user_capability_overrides` table must be re-applied after each resync — otherwise the re-sync silently re-enables tools the admin intentionally disabled.

5. **httpOnlyPaths invariant (B-12 + X-04)** — Every new tRPC mutation and long-running query added in v29.4 must be registered in `livos/packages/livinityd/source/modules/server/trpc/common.ts` `httpOnlyPaths` array. The path is at `common.ts:8`, NOT at `nexus/packages/api/src/common.ts` (common misconception, confirmed by architecture research). Missing this causes mutations to silently queue and drop during the ~5s WS reconnect window.

## Implications for Roadmap

Based on combined research, four phases cover the milestone in the B3a-default (passive SSH gateway) path. All four are partially parallelizable with a single hard sequencing constraint (C2 before A2 Branch 2).

### Phase 45 — Carry-Forward Sweep (C1 + C2 + C3 + C4)

**Rationale:** All four carry-forwards touch independent files and have no UI surface. They block nothing downstream except A2 Branch 2 (which needs C2's SHA re-pin). Landing these first ensures the integrity test is green before any sacred-file surgical work begins, and delivers immediate value (broker 429 forwarding, WS hang fix, OpenAI streaming usage rows) with the lowest risk.

**Delivers:** Broker correctly forwards 429 + Retry-After verbatim; three tRPC routes added to httpOnlyPaths; OpenAI SSE streaming finally writes broker_usage rows; integrity test BASELINE_SHA re-pinned with full audit comment trail.

**Addresses:** C1 (broker error forwarding), C2 (SHA audit debt), C3 (httpOnlyPaths UX hang), C4 (OpenAI usage dashboard gap)

**Avoids:** B-04 (SHA coordination), B-09 (502 forwarded as 429), B-10 (Retry-After normalization loss), B-11 (source change hidden in audit-only commit), B-12 (WS hang), B-13 (SSE chunk order)

**Research flag:** Standard patterns — no additional phase research needed. All four fixes are surgical edits to verified file paths.

### Phase 46 — Fail2ban Admin (B1 + B2)

**Rationale:** The Fail2ban admin panel is the largest new surface (~400 LOC backend + UI combined). It is fully independent of Phase 45 and can build in parallel or immediately after. The backend module must land before the UI (dead-code window is acceptable within a single phase). The "Security" sidebar entry naming is chosen now so it doesn't have to be renamed when v30+ features land.

**Delivers:** Banned-IP list with 5s poll; per-jail stat cards; single-row unban with confirmation modal; "Add to ignoreip whitelist" checkbox (the highest-leverage v29.4 feature); admin-attributed JSON event rows written to `device_audit_log` + `/opt/livos/data/security-events/`; Past Bans history tab; `getServiceState()` returning four distinct states (binary missing / service inactive / active no jails / active with jails); fail2ban-client missing banner with docs link.

**Addresses:** B1 (list + status), B2 (unban + whitelist), B3a passive (unban is the SSH recovery mechanism), LivOS differentiator (admin-attributed audit log)

**Avoids:** B-01 (self-ban cycle), B-02 (ban own IP), B-03 (CIDR /0 unban), W-01 (optimistic-update timing), W-03 (hardcoded sshd), W-04 (service-state ambiguity), W-06 (HTTP vs SSH source IP mismatch), W-19 (cellular IP CGNAT), X-03 (execFile not shell-string), X-04 (httpOnlyPaths for mutations)

**Research flag:** Standard patterns — execFile wrapper, tRPC adminProcedure, React Query 5s poll, alertDialog confirm modal are all established in the codebase. Parser tests use real fail2ban 1.0.2 fixture output (no mocking `child_process`).

### Phase 47 — AI Diagnostics (A1 + A2 + A3/A4)

**Rationale:** These three items share a single UI scaffold (`diagnostics-section.tsx`) which saves ~25% LOC vs three independent panels. A2 MUST start with the on-Mini-PC diagnostic before any source edit. C2 must have landed before this phase begins (Phase 45 prerequisite) so that A2 Branch 2 (sacred-file edit) is available without sequence risk. A1 is the highest-priority user-visible win in the entire milestone (broken AI chat tools restored).

**Delivers:** Shared `diagnostics-section.tsx` with three `DiagnosticCard` sub-components; A1 `diagnoseRegistry()` + `flushAndResync()` restoring missing built-in tools with atomic-swap race protection and user-override preservation; A2 model identity fix (either update.sh dist-copy hardening OR sacred-file `systemPrompt` conditional preset, decided by on-Mini-PC diagnostic); A3 Bolt.diy operational reachability fix; A4 `apps.healthProbe` tRPC route for per-user app reachability self-diagnosis.

**Addresses:** A1 (broken AI tools), A2 (model identity hallucination), A3/A4 (Bolt.diy marketplace probe)

**Avoids:** B-04 (SHA coordination — Phase 45 C2 already landed), B-05 (wrong root cause — diagnostic-first mandate), B-06 (registry race window — atomic temp-prefix swap), B-07 (user override revert — re-apply overrides post-resync), W-12 (false positives — categorize as missing_lost / missing_precondition / disabled_by_user), W-13 (marketplace resync re-installs uninstalled apps — scope parameter: "builtins" vs "all"), W-15 (integration test trashes live Redis — DB index 15 + guard against Mini PC IP)

**Research flag:** A2 needs a live Mini PC diagnostic step before the code plan is finalized. The phase plan must embed two explicit fix branches (Plan 02a: update.sh hardening; Plan 02b: sacred-file surgical edit) with a decision gate at Plan 01 output. This is the one phase where the roadmapper should flag "mini-research needed during execution."

### Phase 48 — SSH Gateway B3b (optional, fully parallel)

**Rationale:** B3a passive (unban + whitelist from Phase 46) already satisfies the "Claude SSH from cloud" use case. B3b active cloudflared tunnel is deferred unless the discuss-milestone elevates it. If shipped, it is fully independent of all other phases and can run in parallel.

**Delivers (if elevated):** `setup_ssh_gateway()` function in install.sh (idempotent, separate unit from any existing cloudflared desktop-streaming config); `cloudflared@ssh-gateway.service` separate templated systemd unit; `system.sshGatewayStatus` tRPC adminProcedure; Settings > Advanced SSH gateway card with ProxyCommand snippet + clipboard copy; `uninstall_ssh_gateway()` mirror function.

**Avoids (if elevated):** B-08 (tunnel collision with pc.* desktop streaming), W-07 (cloudflared version pin), W-08 (ProxyCommand not surfaced), W-09 (threat model undocumented), W-10 (UFW localhost block), W-11 (cert expiry silent failure), W-22 (cgroup-escape for UI-triggered install)

**Research flag:** Skip unless discuss-milestone confirms. If elevated, Cloudflare Access ACL configuration is the blocking concern — do not ship B3b without it (unprotected tunnel is strictly worse than B3a passive).

### Phase Ordering Rationale

- Phase 45 (carry-forwards) runs first to clear integrity test debt; C2 unblocks A2 Branch 2
- Phases 46 and 47 can run in parallel with each other (no shared files) but both depend on Phase 45 completing first for the httpOnlyPaths pattern to be current
- Phase 47 has a hard dependency on Phase 45's C2 commit IF A2 takes Branch 2 (sacred-file edit)
- Phase 48 is fully independent and can be dropped without affecting the other three
- The architecture recommendation is to assign these as phases 45-47 (or 45-48 if B3b elevates), which continues the existing phase numbering from v29.3's phases 39-44

### Gaps to Address

- **A2 root cause (deployment vs source):** Cannot be resolved from research alone. The on-Mini-PC diagnostic in Phase 47 Plan 01 is the only resolution path. Roadmapper must embed this as a mandatory first plan item, not an optional check.
- **B3b Cloudflare Access ACL complexity:** If B3b is elevated, the policy-authoring UX for Cloudflare Access is complex and unvalidated. Research recommends either shipping ACL config as part of the tunnel setup wizard or deferring B3b entirely — do not ship the tunnel without the ACL.
- **Registry flush atomic swap:** The temp-prefix atomic-rename approach (building `capability:_pending:*` then renaming) needs validation against the specific Redis version on the Mini PC before coding begins. An alternative (in-memory IntentRouter cache serving stale data during the 500ms sync window) may be simpler to implement.
- **`device_audit_log` sentinel collision:** `device_id := 'fail2ban-host'` is a string sentinel, not a UUID. Confirm the PG query for listing ban events uses `tool_name IN ('unban_ip', 'ban_ip')` rather than device_id alone, to be safe against future sentinel reuse.

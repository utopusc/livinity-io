# v29.4 Feature Landscape — Server Management Tooling + Bug Sweep

**Domain:** Self-hosted home-server admin UI surfaces — Fail2ban panel + SSH gateway + plugin-registry diagnostic + live SSH log viewer + bug-fix UX surfaces
**Researched:** 2026-05-01
**Confidence:** HIGH (each feature cross-checked against Cockpit / Webmin / Yunohost / CasaOS / Cosmos Cloud / Umbrel / VS Code Extensions Marketplace patterns)
**Pairs with:** `v29.4-STACK.md` (zero new deps, ~940 LOC delta total)

## Executive Summary

Across the seven competing self-hosted admin UIs studied, **Fail2ban admin surfaces cluster into two clean tiers**: a **minimal "list + unban" tier** (Cockpit-fail2ban, Yunohost) and a **rich "list + unban + jail-edit + audit" tier** (Webmin, swissmakers/fail2ban-ui). LivOS should ship the minimal tier in v29.4 plus a **single differentiator** — admin-attributed audit log (which `fail2ban.log` natively does NOT provide) — leveraging the Phase 33 OBS-01 JSON event-row pattern already in the stack.

For SSH gateway, the user's "Claude can SSH from cloud" use case is satisfied by **B3a passive (UI unban) for v29.4**. The B3b active cloudflared tunnel pattern is opt-in install.sh patch territory and should ship as a follow-up phase only if discuss-milestone confirms. **Yunohost and CasaOS both ship without any built-in SSH gateway** — the model home-server pattern is "fix your network, then SSH normally." Adding cloudflared as default would be a differentiator but complicates the threat model.

For Nexus tool registry restore (A1), the dominant pattern from VS Code Extensions, Chrome Extensions Manager, and Cockpit's "reset to defaults" affordance is **diagnostic-first, action-second**: surface what's missing, surface what's stale, then offer ONE big "re-sync" button. **Never offer a registry editor** — every sampled product that did (Webmin's match-actions editor) is universally cited as the source of more breakage than it fixes.

For the SSH session viewer (B4), Cockpit's Logs tab is the closest precedent — but it's a **journalctl wrapper, not an SSH-specific surface**. v29.4's distinct value would be a dedicated SSH lens (filter pre-applied, IP click-to-copy). **B4 is the only optional v29.4 feature** — defer if Phase budget tight.

For bug-fix UX surfaces (A2-A4 + C1-C4), the surface area is intentionally small: a "Test broker 429" admin button (C1) is the only NEW UI control needed; everything else verifies through existing surfaces (model identity in agent chat, Bolt.diy in marketplace browse, OpenAI usage rows in v29.3 dashboard).

**v29.4 priorities (by feature):**
1. **A1 Nexus registry diagnostic + restore** — table-stakes (the user's chat tools are broken)
2. **B1 Fail2ban list/status panel** — table-stakes (dependency for B2)
3. **B2 Unban from UI** — table-stakes (the "I locked myself out" recovery loop)
4. **C1-C4 carry-forward sweep** — must-fix (debt accepted at v29.3 close)
5. **A2-A4 bug verifications** — UAT-only, no new UI
6. **B4 SSH session viewer** — optional differentiator (defer if slack)
7. **B3b active SSH gateway** — defer to v30+ unless discuss-milestone elevates

## Feature 1: Fail2ban Admin Panel (B1 + B2)

**Pattern landscape:** Webmin offers the maximalist surface (jail editor + match-actions CRUD + filter editor), Cockpit-fail2ban offers IP-only minimum, swissmakers/fail2ban-ui sits between (rich list + whois + WS push). Yunohost ships **no fail2ban admin UI at all** — they document `fail2ban-client unbanip` as a CLI step. The dominant pattern for "home server" tier (Yunohost / CasaOS / Umbrel / Cosmos) is **either nothing OR list+unban only** — none ship a jail editor.

LivOS sits at the home-server tier. Mirror Cockpit/Yunohost minimalism with **one differentiator** (admin-attributed audit log) that distinguishes from upstream and aligns with v26.0's existing audit-log infrastructure.

**UI placement:** Sidebar entry inside the v28.0 `LIVINITY_docker` window app, NOT a new system app. Per the milestone-context directive, the entry sits as the 13th sidebar item titled "Security" (or "Bans"). This mirrors Cockpit's pattern where fail2ban is one panel among many, never a top-level app.

### 1a. List View (B1)

| Column | Category | Complexity | Mobile? | Rationale |
|--------|----------|------------|---------|-----------|
| **Banned IP** | Table stakes | Trivial | YES | Universal across all 7 surveyed products |
| **Jail name** | Table stakes | Trivial | YES | Required for routing the unban call (`fail2ban-client set <jail> unbanip <ip>`) |
| **Banned at** (timestamp) | Table stakes | Trivial | YES | "When did this happen" is the #1 admin question |
| **Ban duration** (e.g., "10m / 1w / permanent") | Table stakes | Trivial | YES | Distinguishes auto-expiring from sticky bans |
| **Total failures** (count of attempts) | Table stakes | Trivial | YES | Surfaced from `fail2ban-client status sshd` "Total failed:" line |
| **Last attempted user** (username from log) | Differentiator | Moderate | NO | Requires log parsing (`Failed password for invalid user X` regex) — most surveyed UIs don't show this; high signal for SSH operator |
| **ASN / Country (geo)** | Anti-feature for v29.4 | Moderate | NO | Requires GeoIP MMDB + license lifecycle (per v29.4-STACK.md). Defer to v30+ if user requests. |
| **Whois / ISP** | Anti-feature for v29.4 | Complex | NO | Requires outbound whois call per row OR cached DB. Skip. |
| **Reason / fingerprint hash** | Anti-feature for v29.4 | Complex | NO | Cockpit community has had a 3-year open feature request for this without a merge — the underlying log signal is too weak |

**Pagination:** Simple (no virtualization) — Mini PC SSH jail typically has 0-50 rows; if it exceeds, the user has a bigger problem. Trivial.

**Sort:** Default newest-first by Banned-at. Click headers to re-sort. Trivial.

**Refresh model:** **Polled every 5s, not WebSocket.** Rationale:
- swissmakers/fail2ban-ui uses WS — but that's overkill for a 0-50-row dataset that updates ~10x/day
- Phase 28 cross-container Logs WS aggregator IS a precedent, but logs is a high-frequency stream (lines/sec) where this is a state-snapshot poll
- Cockpit uses 5s polling. Yunohost reloads on demand. Neither uses WS.
- Avoids opening another WS slot in the cap (Phase 28 already enforces 25-socket cap)
- **Decision:** 5s `setInterval` + tRPC poll. ~30 LOC less than WS variant.

**Multi-jail view:** Tabs across jails (sshd, recidive, sshd-aggressive if configured), default tab = SSHD. Trivial — `fail2ban-client status` returns the comma-delimited jail list.

### 1b. Unban Action (B2)

| Sub-feature | Category | Complexity | Mobile? | Rationale |
|-------------|----------|------------|---------|-----------|
| **Single-row unban button** | Table stakes | Trivial | YES | Universal pattern across Cockpit / Webmin / swissmakers |
| **Confirmation modal** ("Unban 1.2.3.4 from sshd?") | Table stakes | Trivial | YES | Mirrors v29.2 factory-reset confirmation pattern (NOT the type-confirm gate — too heavy; just an `<AlertDialog>` Yes/No) |
| **Optimistic UI removal** (row disappears immediately, rolls back on error) | Differentiator | Moderate | YES | React Query `onMutate` + `onError` rollback pattern. The 5s poll re-syncs ground truth. |
| **Add to whitelist** (ignoreip) checkbox in the modal | Differentiator | Moderate | YES | Webmin's killer feature — "unban AND don't re-ban this IP". Implementation: append to `[DEFAULT] ignoreip = ` in `/etc/fail2ban/jail.local` + `fail2ban-client reload`. **Recommend: SHIP THIS in v29.4** because the "Claude SSH from cloud" use case will re-trigger ban every 60s without it. |
| **Bulk unban** (multi-select checkboxes + bulk action bar) | Differentiator | Moderate | NO (mobile) | Useful when a CIDR block is misclassified; under v29.4's home-server scope (1-3 actual ban events/week) the value is low. **Defer to v30+.** |
| **Manual ban an IP** (text-input + "Ban" button) | Anti-feature | — | — | Operationally dangerous from a web UI; no surveyed home-server UI offers this. Use SSH if you really need to. |

**One-line decision:** Ship single-row unban + confirmation modal + whitelist-checkbox (the "add to ignoreip" toggle). Skip bulk for v29.4.

**Why whitelist matters:** Without it, the "Claude can SSH from cloud" loop is broken — Claude SSHs from a Cloudflare egress IP, fails auth twice, gets banned, user unbans, Claude retries from the same IP, gets banned again. Whitelist (ignoreip) is the only sustainable answer short of B3b active gateway.

### 1d. Audit Log (B1d — the LivOS differentiator)

**Why this matters:** Vanilla `fail2ban.log` records ban+unban events with timestamps and IPs but **does NOT record WHO unbanned** when the unban was a manual `fail2ban-client set ... unbanip` call. This is universally cited as a fail2ban gap (per Cockpit community thread + Webmin forum + serverfault answers). LivOS is uniquely positioned to fix this because every unban call goes through livinityd which already has `ctx.currentUser` from Phase 26.0 RBAC.

| Sub-feature | Category | Complexity | Mobile? | Rationale |
|-------------|----------|------------|---------|-----------|
| **Append JSON event row to `/opt/livos/data/security-events/<ts>-<action>.json`** | Table stakes (LivOS-specific) | Trivial | — | Mirrors Phase 33 OBS-01 + v29.2 factory-reset event-row pattern verbatim. Schema: `{ts, action: "unban"|"ban"|"whitelist", jail, ip, admin_user_id, admin_username, source: "ui"|"api", success: bool}` |
| **Past Bans/Unbans tab** in the panel (read-only timeline) | Table stakes | Moderate | YES | List the JSON event rows + the parsed `fail2ban.log` rows (auto-bans / auto-unbans) merged on timestamp |
| **Filter by admin user / IP / action** | Differentiator | Moderate | NO | Useful at admin scale; for home use (1 admin) it's noise. Defer to v30+. |
| **Export CSV / JSON** | Differentiator | Trivial | NO | Pattern from v28.0 OBS-01 update-history export. Defer unless asked. |
| **Per-row "Re-ban this IP" button** | Anti-feature | — | — | Reverse of unban — tempting but operationally a footgun (might re-ban the user's own egress). Skip. |

**One-line decision:** Ship the JSON event row + a "History" tab in the panel showing merged log + admin-attributed events. Filtering and export defer.

**Tech debt note:** The merged-view requires parsing both fail2ban.log (for auto-events) and the LivOS JSON rows (for admin-attributed events). This is the "moderate complexity" cell — about ~80 LOC of parser glue.

## Feature 2: Mini PC SSH Gateway (B3 — passive vs active)

**Pattern landscape:** Among the 7 surveyed products, **Yunohost / CasaOS / Umbrel / Webmin / Cockpit do NOT ship any built-in SSH gateway** — they assume the user already has SSH reachability through their network. Only **Cosmos Cloud** advertises a built-in VPN/proxy layer (via SSO + reverse proxy) but it's HTTP-only, not SSH. **No competitor ships SSH-via-cloud-tunnel out of the box.** This means B3b would be a genuine differentiator, but the absence is also a signal: the home-server market doesn't expect this.

The "Claude can SSH from cloud" use case is **specific to LivOS's AI-orchestration thesis** and not represented in competitor product surfaces. This is opportunity territory but also un-validated UX territory.

### 2a. B3a (Passive — Unban-from-UI Only)

| Sub-feature | Category | Complexity | Mobile? | Notes |
|-------------|----------|------------|---------|-------|
| **Use existing B2 unban + whitelist flow** | Table stakes | Trivial (already in B2) | YES | Zero install.sh changes, zero new infra |
| **Documentation page**: "Adding Claude's egress IP to the SSH whitelist" with step-by-step | Table stakes | Trivial | — | Mirrors `/help/factory-reset-recovery` from v29.2 |
| **One-tap "Allow my current IP" on the panel** (auto-detects requesting IP, opens unban + whitelist modal pre-filled) | Differentiator | Trivial | YES | Tiny but high-leverage UX — the user is currently locked out, opens the panel from another network, one click adds their original IP back |

**One-line decision: B3a is the v29.4 default.** It's literally B1+B2+whitelist + docs. Zero new code beyond Feature 1. The "Claude SSH from cloud" use case is solvable by whitelisting Cloudflare's egress range once (a /16 entry in ignoreip), no tunnel needed.

**Threat model trade-off:** B3a leaves SSH on a public IP (which is what the Mini PC currently has). If the IP is reachable at all, fail2ban is the only protection — and now there's a documented user-controlled bypass (the whitelist). Acceptable for a single-admin home server; **NOT** acceptable if LivOS ever lands a multi-tenant SSH gateway.

### 2b. B3b (Active — cloudflared SSH tunnel)

| Sub-feature | Category | Complexity | Mobile? | Notes |
|-------------|----------|------------|---------|-------|
| **install.sh `setup_ssh_gateway()` function** (per v29.4-STACK.md §2 patch landing zone) | Table stakes (if shipped) | Moderate | — | ~80 LOC bash, idempotent (`[ -f /etc/cloudflared/ssh-tunnel.json ] && return`) |
| **`ssh.{username}.livinity.io` DNS routing via Cloudflare named tunnel** | Table stakes (if shipped) | Moderate | — | Mirrors v18.0 `pc.{username}` pattern |
| **Per-tunnel ACL** (Cloudflare Access policy: only authenticated users can hit the tunnel) | Table stakes (if shipped) | Complex | — | Without ACL, the tunnel is a public-internet-reachable SSH endpoint that bypasses fail2ban entirely — a footgun |
| **UI toggle in Settings > Advanced > SSH Gateway** ("Enable cloud SSH gateway") + status indicator | Table stakes (if shipped) | Moderate | YES | Mirrors v18.0 desktop streaming Settings panel |
| **Setup wizard** (asks for Cloudflare API token, validates, runs setup_ssh_gateway in background) | Differentiator | Complex | NO | Mirrors v7.0 multi-user wildcard cert flow which already requires CF API token |
| **Connection status / last-handshake / current peer count** display | Differentiator | Moderate | YES | Mirrors v14.0 device card "online/offline pulse" pattern |
| **Auto-revoke / disable button** | Table stakes (if shipped) | Trivial | YES | One-click `systemctl stop cloudflared@ssh-tunnel && cloudflared tunnel delete ssh-{hostname}` |

**One-line decision: DEFER B3b to v30+ unless discuss-milestone elevates.** Rationale:
- B3a satisfies the user's articulated use case (Claude SSH) at zero new-infra cost
- No competitor ships this — the UX is unvalidated
- Cloudflare Access ACL is the real cost (free tier supports up to 50 users but the policy authoring is complex; without it the tunnel is dangerous)
- The threat-model claim "active is safer than unban-from-UI" is **false** unless ACL is configured properly. With a misconfigured tunnel, B3b is strictly worse than B3a.

**If shipped (escalation path):** Phase order should be (1) install.sh patch + systemd unit, (2) UI toggle + status, (3) Cloudflare Access ACL helper, (4) revoke button. Phase 4 is where the "you can't ship without it" gate sits.

## Feature 3: Nexus Tool Registry Diagnostic + Restore (A1)

**Pattern landscape:** The closest analogues are VS Code Extensions Manager ("Restore extension" + "Repair" buttons), Chrome Extensions Manager ("Repair" button when corruption detected), and Cockpit's "Reset to defaults" affordance in module config. Slack/Notion app marketplaces have **no equivalent** — they don't surface this state because they don't have the "missing built-in" problem (apps are never expected, only optional).

**The dominant pattern is two-step:**
1. **Diagnose** (passive surface — list what's expected vs present vs extra)
2. **Restore** (single button — wipe + re-sync from source of truth)

**Critical design lesson** from VS Code's Issue #43298 thread (cited in research): users get confused when "reset" silently disables their custom settings. **Always show what will change BEFORE acting.**

### 3a. Diagnostic View

| Sub-feature | Category | Complexity | Mobile? | Notes |
|-------------|----------|------------|---------|-------|
| **"Expected built-ins" list** with present/missing dot per row | Table stakes | Trivial | YES | Fixed list (shell, docker_*, files, web_search, etc.) baked in code; render `redis.exists(capability:<id>)` per row |
| **Last sync timestamp** ("Last synced: 2h ago") | Table stakes | Trivial | YES | Read from `redis.get("capability:registry:lastSyncAt")` |
| **Sync source breakdown** (4 sources from Phase 22 — built-in / skill_* / mcp_* / agent_*) with row counts | Differentiator | Trivial | YES | High info value for the user; "the marketplace MCP added 12, built-ins 0" tells them what broke |
| **"Extra" capabilities** (in Redis but not in any source) | Differentiator | Moderate | NO | Useful for diagnosing ghost manifests post-uninstall; defer if budget tight |
| **Per-capability "registered/enabled/disabled" badge** | Anti-feature | — | — | Too granular — opens the door to per-capability toggling which is what creates the bug |
| **Sync error log** (last 5 sync attempts + error messages) | Differentiator | Moderate | NO | Mirrors Phase 33 OBS-01 update-history pattern |

**One-line decision:** Ship "expected built-ins" list + last sync ts + 4-source breakdown. Skip "extra" detection and per-capability toggles.

### 3b. Restore Action

| Sub-feature | Category | Complexity | Mobile? | Notes |
|-------------|----------|------------|---------|-------|
| **"Re-sync registry" button** | Table stakes | Trivial | YES | Calls `capabilities.flushAndResync` tRPC route which `redis.del("capability:*")` + `syncFromAllSources()` |
| **Confirmation modal** ("This will rebuild the capability registry from source. Active conversations may briefly miss tools.") | Table stakes | Trivial | YES | Lightweight `<AlertDialog>` — NOT a type-confirm gate (lower stakes than factory reset) |

| **Post-sync diff** ("Added 4, removed 1, updated 2") | Differentiator | Moderate | YES | Built from before/after capability set diff; high signal |

| **Per-capability "delete this entry" button** | Anti-feature | — | — | Direct registry edit = the bug class we're fixing. **Skip.** |
| **Manual capability add** (paste JSON manifest) | Anti-feature | — | — | Use the existing marketplace install path. Skip. |
| **Sync history audit log** (which sync added/removed which capability) | Differentiator | Complex | NO | High value for forensics; complex because requires before/after snapshots per sync. **Defer to v30+.** |

**One-line decision:** Ship the button + confirmation + spinner + post-sync diff. Skip everything else.

## Feature 4: SSH Session Viewer (Optional B4)

**Pattern landscape:** Cockpit has a "Logs" tab (journalctl frontend) but it's general-purpose, not SSH-scoped. lnav is a CLI tool — closest interactive precedent. swissmakers/fail2ban-ui shows ban events as logs; not the same as live SSH session attempts. **No surveyed product ships an SSH-specific live viewer.** This is genuine differentiator territory.

The use case: admin wants to see "is someone hammering my SSH right now?" in real time, click-to-copy the IP, take action (ban via Feature 1, or whitelist).

## Anti-Feature Catalogue (consolidated, with one-line skip rationale)

| Anti-feature | Why skip | Suggested instead |
|--------------|----------|-------------------|
| Editable findtime/maxretry/bantime | Yunohost/CasaOS skip it; opens support burden | Document SSH override for power users |
| Custom jail editor | Webmin's #1 support-ticket driver | SSH-only jail is enough |
| Filter regex editor | Same — high-skill audience, low payoff | Document the file path for power users |
| Manual ban-IP control from web UI | No surveyed home-server UI offers it | Use whitelist (ignoreip) instead |
| Per-row "Re-ban this IP" button | Footgun — might re-ban admin's own egress | None |

| Per-capability delete in registry | Same bug class we're fixing | Re-sync button only |
| Manual capability JSON paste | Use marketplace install path | Marketplace |

| Multi-host SSH viewer | Mini PC is only LivOS host | None |
